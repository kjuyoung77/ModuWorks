## 🧠 파트별 티칭 스크립트 (핵심만)

### P1. 왜 Git인가 · 버전관리란
- 누구나 겪은 `보고서_최종 → 진짜최종 → 진짜진짜최종_수정본` = **원시적 버전관리**. 문제: 뭐가 최신인지·누가 뭘 바꿨는지·되돌리기 어려움.
- 코드는 파일이 **수십~수백 개** → 로그인 하나 고치면 화면·서버·설정·테스트 파일이 같이 바뀜 → "어느 변경 때문에 망가졌나?" 추적 필요.
- **Git = 언제·누가·어떤 파일을·왜 바꿨는지 기록하고, 필요하면 이전 상태로 되돌리는 도구.** 단순 백업이 아니라 **시간 흐름을 구조적으로 기록**.
- 탄생사: 리눅스 커널 개발이 2005년 BitKeeper 무료권한 철회 → **Linus Torvalds가 직접 제작**. 목표=속도·단순설계·비선형(브랜치)개발·완전분산·대형프로젝트.

### P2. Git ≠ GitHub
- **Git** = 버전관리 도구. 내 컴퓨터 안(로컬)에서, 인터넷·계정 없어도 동작.
- **GitHub** = Git 저장소를 **온라인에 보관 + 협업**하게 해주는 호스팅 서비스. **여러 선택지 중 하나**(GitLab·Bitbucket·자체서버도 가능).
- 저장소(repository)=프로젝트+변경이력 공간. 로컬 저장소 vs 원격 저장소. **remote**=내 로컬이 아는 "다른 저장소 주소".
- GitHub만의 협업기능: **Issue**(할 일·버그 기록) / **PR**(변경 합쳐달라 요청) / **Code Review**(코드에 의견) / **Actions**(테스트·빌드·배포 자동화=CI/CD).
- 한 줄: "GitHub에 올린다"=단순 업로드가 아니라 **내 Git 이력을 원격과 동기화**.

### P3. 실습① 기본 흐름 (라이브)
```
git --version                       # 설치 확인
git config --global user.name "..."  # 계정 등록
git init                            # .git 생성 = "이 폴더 git으로 관리!"
echo "# first" >> readme.md
git status                          # Untracked file 확인
git add readme.md                   # Staging Area에 올림
git commit -m "new readme file"     # 스냅샷 확정
git log                             # 커밋 해시(SHA-1)·작성자·메시지 확인
```
- **3공간:** Working Directory(작업) → **add** → Staging Area(커밋 준비) → **commit** → HEAD(최신 커밋 포인터).

### P4. 실습② 원격 연결 (라이브)
```
git remote add origin https://github.com/USER/first-repository.git  # origin=별칭
git push origin main   # 로컬 커밋 → 원격 전송 (인증: 토큰 PAT 필요)
git pull origin main   # 원격 변경 → 로컬 가져오기+병합
git clone <url>        # 원격 저장소 통째로 복제
```
- push 인증은 비밀번호가 아니라 **개인 액세스 토큰(PAT)**: Settings→Developer Settings→Generate new token. **토큰 절대 외부 노출 금지.**

### P5. Branch & Merge
- **브랜치 = 특정 커밋을 가리키는 가벼운 이름표(포인터).** 프로젝트 통째 복사가 **아님**. "여기서부터 다른 가능성 실험" 포인터 하나 만드는 것.
- 활용 3사례: **새 기능**(위시리스트 브랜치) / **버그수정**(로그인-버그-수정 브랜치) / **실험**(새-디자인 브랜치) → 메인에 영향 없이 작업 후 **merge** or 삭제.
- **merge**=갈래 합치기. 다른 부분 고쳤으면 자동 병합, 같은 부분 다르게 고쳤으면 **충돌(conflict)**.
- **충돌 = 고장 아님.** "여기는 두 변화가 동시에 맞을 수 없으니 **사람이 판단하라**"는 Git의 정직한 신호.

### P6. AI 시대의 Git
- AI 도구가 Git을 요구하는 **5가지 이유:**
  1. **상태 파악** — status/diff/log로 뭐가 바뀌었는지 안정적으로 읽음
  2. **안전장치** — 커밋=게임 세이브포인트. 큰 작업 전 커밋하면 실패해도 되돌리기 쉬움
  3. **협업의 언어** — PR·review·revert·release note가 Git 기록 위에서 작동
  4. **로컬 우선** — 네트워크 없이도 로그·브랜치·커밋·비교 가능
  5. **책임 경계** — 어느 파일·어느 줄을 왜 바꿨는지 기록(좋은 커밋 메시지)
- **바이브코딩 루프:** 작업 전 커밋 → AI에 작은 변경 요청 → `git diff` 확인 → 테스트 → OK면 커밋 / 아니면 되돌리기 → 다른 브랜치로 다른 접근.
- **"Git 필요" vs "GitHub 권한 필요":** 전자=내 코드 이력 추적·되돌리기(로컬·diff). 후자=온라인 저장소·Issue·PR·Review·Actions까지 읽고쓰기.
- 마무리 프레임: AI는 확률적(생성), Git은 결정적(고정). **"AI는 가능성을 늘리고, Git은 선택을 역사로 남긴다."**

## 🎯 예상 문제 은행 (S팀 드릴용 + 던전 대비) — ⭐=출제 유력

**⭐ Q1. Git과 GitHub의 차이는?**
> Git=내 컴퓨터에서도 돌아가는 **버전관리 도구**(변경 이력 기록·되돌리기). GitHub=Git 저장소를 **온라인 보관+협업**하게 해주는 호스팅 서비스 중 하나. Git은 GitHub 없이도 사용 가능.

**⭐ Q2. 버전관리란 무엇이고 왜 필요한가?**
> 프로젝트가 시간에 따라 어떻게 바뀌었는지 기록하고 이전 상태와 비교·되돌리기. `보고서_진짜최종`의 체계화 버전. 코드는 파일이 수십~수백 개라 더 중요.

**⭐ Q3. add와 commit의 차이 / staging area란?**
> Working Directory(작업) →`add`→ Staging Area(커밋할 것 준비) →`commit`→ 스냅샷 확정. add=준비단계, commit=버전 확정. HEAD=최신 커밋 포인터.

**⭐ Q4. branch란? 프로젝트를 통째로 복사하는 것인가?**
> 아니다. 브랜치는 **특정 커밋을 가리키는 가벼운 이름표(포인터)**. 복사가 아니라 "여기서부터 다른 가능성 실험"하는 포인터.

**⭐ Q5. merge와 충돌(conflict)? 충돌은 Git이 고장난 것?**
> merge=서로 다른 갈래 합치기. 같은 부분을 다르게 고치면 충돌. 충돌은 고장이 아니라 **"사람이 판단하라"는 Git의 정직한 신호.**

**Q6. push와 pull의 차이? origin은?**
> push=로컬 커밋을 원격으로 보냄 / pull=원격 변경을 로컬로 가져와 병합. origin=원격 저장소 별칭(alias).

**Q7. 로컬 저장소 vs 원격 저장소 / remote?**
> 로컬=내 노트북 안 저장소. 원격=내 컴퓨터 밖(GitHub 등). remote=내 로컬이 아는 다른 저장소의 주소.

**Q8. git init을 하면 무슨 일이 일어나나?**
> 그 폴더를 git으로 관리 시작. 관리정보 담는 `.git` 숨김 디렉토리 생성(`ls -a`로 확인).

**⭐ Q9. AI 시대에 Git이 왜 더 중요한가? (하나 이상)**
> ①상태 파악 ②안전장치(커밋=세이브포인트) ③협업의 언어 ④로컬 우선 ⑤인간-AI 책임 경계. "AI는 생성, Git은 고정."

**Q10. AI 도구가 "Git 필요"라 할 때와 "GitHub 권한 필요"라 할 때 차이?**
> Git 필요=내 코드 이력 추적·되돌리기(로컬·diff). GitHub 권한=온라인 저장소·Issue·PR·Review·Actions 읽고쓰기.

**Q11. 좋은 커밋 메시지가 왜 중요? 좋은 예?**
> 판단의 단위. "버그 수정"보다 "로그인 실패 시 토큰 갱신 루프를 막기 위해 만료 조건 추가"가 좋음. 나중에 AI에 다시 물을 때 맥락이 됨.

**Q12. git log의 SHA-1 해시란?**
> 각 커밋을 고유 식별하는 40자리 16진수. 내용이 조금만 바뀌어도 해시가 바뀜 → 변조·손상 감지, 특정 시점 복귀 기준.

**Q13. Git은 누가 왜 만들었나?**
> 리눅스 커널이 BitKeeper 무료권한 철회(2005)되자 **Linus Torvalds**가 직접 제작. 속도·분산·대형 프로젝트 대응.

**Q14. 좋은 바이브코딩 흐름을 Git으로 설명하면?**
> 작업 전 커밋 → AI에 작은 변경 요청 → `git diff` 확인 → 테스트 → OK면 커밋 / 아니면 되돌리기 → 다른 브랜치 시도.

**Q15. Git 내부 객체 3가지? (심화)**
> blob(파일 내용)·tree(디렉토리 구조)·commit(tree+부모커밋+작성자+메시지). 이름이 아니라 **내용**으로 기억, 해시로 식별.
