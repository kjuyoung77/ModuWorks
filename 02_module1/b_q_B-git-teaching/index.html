📌 30초 요약

  • Git = 프로젝트가 언제·누가·무엇을·왜 바꿨는지 기록하고 되돌릴 수 있게 하는 버전관리 도구. 로컬에서 인터넷 없이도 동작.
  • GitHub = Git 저장소를 온라인에 보관 + 협업(Issue·PR·리뷰·자동화)하게 해주는 서비스. 여러 선택지 중 하나(GitLab·Bitbucket도 있음).
  • AI 시대에 Git이 더 중요해진 이유 = AI가 코드를 빠르게 대량으로 바꾸는 만큼, 무엇이 바뀌었고 문제 시 되돌릴 수 있는 안전장치가 필수이기 때문.

1. 핵심 개념

주요 명령어

명령어의미

git status현재 변경 상태 확인

git clone <url>원격 저장소를 통째로 복제
git diff이전 상태와 현재의 차이 확인

Branch(브랜치) & Merge(병합)

  • 브랜치 = 특정 커밋을 가리키는 가벼운 이름표(포인터). 프로젝트 통째 복사가 아님.
  • 활용: 새 기능 / 버그 수정 / 실험 — 메인에 영향 없이 작업 후 마음에 들면 merge, 아니면 삭제.
  • merge = 갈래 합치기. 같은 부분을 다르게 고쳤으면 충돌(conflict).
  • 충돌은 Git 고장이 아니라 "여기는 사람이 판단하라"는 정직한 신호.

AI 시대에 Git이 필요한 5가지

  1. 상태 파악 — status/diff/log로 무엇이 바뀌었는지 안정적으로 읽음
  2. 안전장치 — 커밋 = 게임 세이브포인트. 큰 작업 전 커밋 = 실패 비용↓
  3. 협업의 언어 — PR·리뷰·revert가 Git 기록 위에서 작동
  4. 로컬 우선 — 네트워크 없이도 로그·브랜치·커밋·비교 가능
  5. 책임 경계 — 어느 파일·어느 줄을 왜 바꿨는지 기록(좋은 커밋 메시지)

지속가능한 바이브코딩 루프: 작업 전 커밋 → AI에 작은 변경 요청 → git diff 확인 → 테스트 → OK면 커밋 / 아니면 되돌리기 → 다른 브랜치로 다른 접근.

2. 출제 유력 문제 (⭐ = 유력)

⭐ Q1. Git과 GitHub의 차이는?
답: Git=내 컴퓨터에서도 돌아가는 버전관리 도구 / GitHub=Git 저장소를 온라인에 보관·협업하게 해주는 서비스 중 하나.
Git은 GitHub 없이도, 인터넷·계정 없이도 쓸 수 있음. "GitHub=Git"이 아님이 핵심.
⭐ Q2. 버전관리란 무엇이고 왜 필요한가?
답: 프로젝트가 시간에 따라 어떻게 바뀌었는지 기록하고 비교·되돌리는 것.
보고서_진짜최종.docx가 원시적 버전관리. 코드는 파일이 수십~수백 개라 추적이 필수.
⭐ Q3. add와 commit의 차이 / Staging Area란?
답: add=변경 파일을 스테이징에 올리는 준비 단계 / commit=그 순간을 스냅샷으로 확정.
바로 커밋되는 게 아니라 준비→확정 2단계인 것이 포인트. HEAD는 최신 커밋 포인터.
⭐ Q4. branch란? 프로젝트를 통째로 복사하는 것인가?
답: 아니다. 브랜치는 특정 커밋을 가리키는 가벼운 이름표(포인터).
main·feature/login 같은 이름표가 각각 다른 커밋을 가리킴.
⭐ Q5. merge와 충돌? 충돌은 Git이 고장난 것인가?
답: merge=서로 다른 갈래를 합치기 / 충돌=같은 부분을 다르게 고쳤을 때 발생. 고장이 아님.
충돌은 "사람이 판단하라"는 Git의 정직한 신호.
⭐ Q6. AI 시대에 Git이 왜 더 중요한가?
답: ①상태 파악 ②안전장치 ③협업의 언어 ④로컬 우선 ⑤인간-AI 책임 경계.
AI가 코드를 빠르고 대량으로 바꾸므로 되돌릴 장치가 더 필요. "AI는 생성, Git은 고정."
Q7. push와 pull의 차이? origin은?
답: push=로컬→원격 전송 / pull=원격→로컬 병합. origin=원격 저장소의 별칭.
Q8. git init을 하면?
답: 그 폴더를 git으로 관리 시작하고 .git 숨김 폴더를 생성.
Q9. 로컬/원격 저장소, remote의 뜻은?
답: 로컬=내 컴퓨터 안 / 원격=밖(GitHub 등) / remote=로컬이 아는 다른 저장소의 주소.
Q10. "Git 필요" vs "GitHub 권한 필요" 차이는?
답: Git 필요=로컬 변경 이력 추적·되돌리기 / GitHub 권한=온라인 저장소·Issue·PR·리뷰·Actions.
Q11. 좋은 커밋 메시지가 왜 중요한가?
답: "버그 수정"보다 "로그인 실패 시 토큰 갱신 루프를 막기 위해 만료 조건 추가"가 좋음.
나중에 AI에게 다시 물을 때 강력한 맥락이 됨.
Q12. git log의 SHA-1 해시란?
답: 각 커밋을 고유 식별하는 40자리 16진수. 내용이 바뀌면 해시도 바뀜 → 변조 감지.
Q13. Git은 누가 왜 만들었나?
답: 리눅스 커널이 BitKeeper 사용권을 잃자(2005) Linus Torvalds가 제작.
속도·비선형(브랜치)·완전 분산·대형 프로젝트 처리를 목표로 태어남 = AI 시대와 닮음.
Q14. 좋은 바이브코딩 흐름을 Git으로 설명하면?
답: 작업 전 커밋 → AI에 작은 변경 요청 → git diff → 테스트 → 커밋/되돌리기 → 다른 브랜치.
Q15. (심화) Git 내부 객체 3가지는?
답: blob(파일 내용)·tree(디렉토리 구조)·commit(tree+부모+작성자+메시지).
Git은 파일을 이름이 아니라 내용으로 기억하고 해시로 식별.
