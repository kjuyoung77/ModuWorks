1-1강 · AI에게 일을 맡길 작업대 차리기(8강)

환경설정~터미널 Claude Code. 목표는 도구 문법 암기가 아니라 "정확히 지시 → 내 컴퓨터에서 결과 확인 → 다시 수정 요청"하는 루프를 한 번 여는 것. 메인 도구는 Claude Code.

1·1 · AI에게 일을 맡길 작업대를 차린다는 것 00:30

환경 설정 = 도구 설치가 아니라, AI에게 일을 시키고 결과를 바로 검토·수정할 작업대를 만드는 것.

왜 채팅 AI가 아니라 Claude Code?

일반 챗봇은 답과 코드를 보여줄 뿐. 웹서비스를 만들려면 답이 아니라 실제 파일이 필요하고 그게 내 컴퓨터에서 도는지 확인해야 한다. Claude Code는 자연어로 지시하면 직접 파일을 만들고·고치고·명령을 실행한다. 이 '말하면 손이 움직이는' 협업 = 에이전틱 코딩 도구.

1·2 · 내 작업대 상태 확인하기 01:00

설치했다 ≠ 된다. "버전 확인"이 진짜 수업. Claude Code·Node.js·npm·Git 준비.

🅰️ 빠른 길 — Claude Code에게 맡기기

Claude Desktop 안 Claude Code에게 자연어로: "내 OS는 ○○야. Node·npm·Git 확인하고 없으면 공식 방법으로 설치, 각 버전을 출력해줘."

🤔 왜 "버전을 보여줘"? AI가 "설치했습니다"라 답해도 내 컴퓨터에서 진짜 되는지는 모른다. 버전 출력 = AI의 주장을 내가 검증하는 가장 단순한 방법. 이게 모든 바이브 코딩의 기본.

🅱️ 손으로 직접 설치 (기본기)

Node.js·npm은 nvm으로 관리. 시스템에 직접 깔면 전역 패키지 설치 때 EACCES 권한오류·sudo 문제가 잦다. nvm은 홈(~/.nvm)에 깔아 sudo 없이 전역 설치 가능.

# macOS / Linux / WSL — nvm 설치 후 LTS
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.5/install.sh | bash
nvm install --lts
nvm use --lts
node --version
npm --version

Windows는 별개인 nvm-windows의 nvm-setup.exe 설치 후 nvm install <LTS버전> / nvm use <LTS버전> (--lts 별칭 없음, 버전번호 명시). nvm use 실패 시 관리자 권한 터미널로.

Claude Code 설치:

# macOS/Linux/WSL
curl -fsSL https://claude.ai/install.sh | bash
# Windows PowerShell
irm https://claude.ai/install.ps1 | iex
# 또는 Node 22+ 있으면
npm install -g @anthropic-ai/claude-code
Claude Code는 Python 패키지가 아니라서 uv tool install로 안 깐다. (uv는 Python 프로젝트·도구 관리용.)

✅ 확인 브릿지 — 어떤 길로 깔았든 내가 확인

pwd
node --version
npm --version
uv --version
git --version
claude --version
오류는 실패가 아니라 단서. command not found가 나오면 숨기지 말고 "환경 / 실행한 명령 / 실제 출력"을 분리해서 AI에게 그대로 준다. "안 돼요"라고 하면 AI도 추측하지만, 정확한 출력을 주면 다음 행동을 좁혀준다.

🔧 AI에게 첫 파일 생성을 맡기는 프롬프트

백지에서 HTML을 외워 쓰지 않는다. 폴더에서 Claude Code 실행 후:

index.html에 브라우저에서 바로 열 수 있는 첫 화면을 만들어줘.
- 한국어 문서로.
- 큰 제목엔 [내 이름/역할].
- 본문엔 [내 도메인]에서 AI로 해보고 싶은 일 한 문장.
코드를 대신 써달라는 요청이지만, 책임까지 넘기는 건 아니다. 나온 결과를 반드시 검토 — 제목이 내 역할을 드러내나 / 본문이 너무 추상적이지 않나 / 브라우저에서 글자가 보이나. "돌아간다"에서 멈추지 않고 "내 목표에 맞나"까지 본다.
📝 산출물 체크: 폴더 있다 / index.html 있다 / 브라우저에서 제목·본문 보인다 / 내 도메인 문장으로 바꿨다.

1·4 · 바이브 코딩 루프 한 번 돌리기 00:30 이 노드의 중심

설치·파일 생성은 준비. 바이브 코딩은 여기서 시작.

① 만들 것을 자연어로 말한다 → ② AI가 코드 생성 → ③ 내가 읽고 판단 → ④ 근거 들어 다시 지시 → 다시 ①

🔧 두 번째 요청: 도메인 섹션 추가

첫 요청과 결정적으로 다른 점: "무엇을 추가할지"와 "무엇은 건드리지 말지"를 함께 말한다.

방금 만든 index.html을 수정해줘.
추가: 내 도메인의 반복 업무 3개를 목록으로 + 각 한 줄 설명.
지킬 것: 기존 제목·첫 문장 유지, 들여쓰기 읽기 좋게.
🤔 AI가 실수할 지점을 미리 예측: 기존 문장을 바꾸거나, 안 시킨 CSS를 넣거나, 업무 3개가 도메인과 안 맞거나.

재요청 예시:

좋아. 다만 목록 3개가 내 실제 업무와 달라. 아래로 바꿔줘.
1. [내 업무1] 2. [내 업무2] 3. [내 업무3]
이 한 번의 왕복(지시→생성→판단→수정지시)이 돌았다면, 개발 환경은 더 이상 "설치 목록"이 아니라 작업대가 된 것.

문서의 % pwd, $ npm --version에서 %·$는 예시 프롬프트. 실제로 칠 건 pwd, npm --version뿐. 초보자 최다 실수.

명령이 지나가는 길

터미널 에뮬레이터  (% pwd 입력·화면표시)
     ↓  PTY (가상 터미널 통신 채널: master↔slave)
  셸 (zsh·bash)   (문장 파싱 → 명령 위치 탐색)
     ↓  시스템 콜
  커널            (권한 확인 + 자원 중개)
  CPU·메모리·파일·네트워크

pwd 한 줄이 거치는 7단계: ① 터미널이 키 입력 받음 → ② PTY가 셸에 넘김 → ③ 셸이 문장 파싱(공백·따옴표·|·>) → ④ 셸이 명령 위치 탐색($PATH, 단 pwd는 셸 내장(builtin)일 수 있음) → ⑤ 필요시 fork+exec로 새 프로세스 → ⑥ 시스템 콜로 커널에 요청 → ⑦ stdout이 PTY 거쳐 화면 복귀.

오류를 층으로 나눠 읽기

증상의심할 층
창이 안 열림·글자 깨짐·멈춤터미널 앱·인코딩·PTY
command not found셸의 명령 탐색 또는 설치/$PATH
Permission denied·EACCES커널이 관리하는 파일 권한
Claude Code는 어느 층? 터미널도 셸도 커널도 아닌, 셸 위에서 도는 프로그램. claude 입력 → 셸이 $PATH에서 찾아 실행 → 그 셸의 환경·권한 안에서 파일 읽고 명령 실행 요청. AI가 강력해도 현재 권한과 작업 폴더(pwd)를 초월하지 못한다. 그래서 오류 한 줄 + pwd + 실행한 명령을 같이 주는 습관이 중요.

각 층 직접 관찰 (읽기 전용)

echo $SHELL     # 로그인 셸 경로 (지금 실행 셸과 꼭 같진 않음)
echo $0         # 현재 실행 중인 셸 이름
tty             # 연결된 터미널 장치 = PTY의 흔적
which pwd       # $PATH에서 찾은 위치 (pwd는 builtin → type pwd 도)
echo $PATH      # 명령 찾는 디렉터리 목록 (: 로 구분)
🪟 PowerShell(내 환경): 위는 zsh/bash용이라 그대로 안 됨. 대신 $PSVersionTable(지금 셸 확인) · Get-Command pwd(어디서 찾나) · $env:Path(탐색 경로). 관찰 포인트는 동일: 지금 내 셸이 뭐고, 명령을 어디서 찾는가.

2·1 · AI는 잘 만드는데, 내 말은 왜 안 통할까 00:10

웹페이지는 한 덩어리처럼 보이지만, 성격이 다른 세 겹이 겹쳐 있다.

HTML — 구조

화면에 무엇이 있는지. 제목·문단·버튼에 "이건 제목/버튼"이라 의미를 붙임.

CSS — 표현

그것이 어떻게 보이는지. 색·간격·크기·배치.

JavaScript — 동작

사용자가 무언가 했을 때 무엇이 바뀌는지. 클릭 반응 등.

<!-- HTML: 태그로 의미를 붙임 -->
<h1>내 첫 웹페이지</h1>
<button>시작하기</button>

/* CSS: 어떻게 보일지 */
h1 { color: blue; text-align: center; }

// JS: 누르면 무엇이 바뀔지
button.addEventListener('click', () => { alert('안녕하세요!'); });
🎯 오늘의 핵심 한 문장: 웹페이지는 구조(HTML)·표현(CSS)·동작(JS) 세 겹이고, 셋을 나눠 AI에게 지시하면 원하는 화면을 빠르게 만들고 고칠 수 있다.
겹담당AI에게 시킬 때 말결과 판단 질문

CSS어떻게 보이는지"모바일에서도 읽히게 간격·색·글자 크기 조정해줘"눈이 어디로 가나, 깨지는 구간 없나
JS했을 때 뭐가 바뀌는지"버튼 누르면 화면 내용이 바뀌게 해줘"클릭·입력·상태 변화가 기대대로인가
🤔 예측: ① 문구가 내 업무 말과 안 맞음 → HTML / ② 모바일에서 글자 겹침 → CSS / ③ 버튼 눌러도 반응 없음 → JS. 요청을 세 겹으로 나누면 AI가 고칠 지점이 좁아진다.

2·2 · 세 겹을 나누어 이해하기 — 이름에 담긴 사연 00:15

세 겹은 다른 시기에, 다른 문제를 풀려고 태어났다. 이름 안에 그 사연이 있다.

HTML — 잃어버린 문서에서 시작

1980년대 말 CERN은 정보가 엉킨 거대 연구소였다. 연구자가 떠나면 자료 위치도 사라졌다. 연구원 팀 버너스리가 1989년 제안: 문서 속 단어에서 다른 문서로 바로 건너뛰게 연결하자 = 하이퍼텍스트(hyper=순서를 뛰어넘어). 상사 코멘트는 "막연하지만 흥미롭다". 1990년 말 NeXT 컴퓨터 한 대로 최초의 웹 브라우저+서버를 만듦(본체에 "전원 끄지 마시오" 손글씨 스티커).

🔗 최초의 웹사이트 info.cern.ch — 배경·색·버튼 없이 링크 달린 글자뿐. "CSS도 JS도 없던 웹". 1993년 CERN이 웹 기술을 로열티 없이 공개 → 특정 회사 제품이 아닌 모두의 기반이 됨.

HTML = HyperText Markup Language. '마크업'은 편집자가 원고 여백에 "여기는 제목"이라 표시하던 교정 관행. 검토 기준은 꾸밈이 아니라 정보 — 필요한 정보가 다 있나, 의미가 맞게 붙었나. (예: 병원 안내에 '제증명 발급'이 빠졌는지는 현장을 아는 눈에만 보인다 → 최종 검토자는 나.)

※ '문서 연결' 발상 자체는 1945년 버니바 부시(Memex), 1965년 테드 넬슨('하이퍼텍스트' 명명)이 먼저. 버너스리의 공헌은 인터넷 위에서 실제로 작동하게 만든 것.

CSS — "제목 색을 전부 바꿔주세요"의 악몽

초기 웹은 꾸밈 태그를 본문 곳곳에 직접 끼워야 했다. 문서 수백 개에서 "제목 색 전부 바꿔줘"가 오면 하나하나 열어 고쳐야 함(내용과 꾸밈이 붙어서). 1994년 호콘 비움 리가 인쇄(기사≠조판 규칙)에서 힌트 → 꾸밈 규칙을 문서 밖 스타일시트로 분리(1996 표준).

Cascading = 폭포처럼 층층이. 브라우저 기본·사이트 공통·페이지 개별 규칙이 겹칠 때 우선순위가 폭포처럼 정해짐 → Cascading Style Sheets.

🔗 CSS Zen Garden(2003, 데이브 셰이) — HTML 하나 고정, CSS만 바꿔 일본 정원도 잡지 표지도 됨(내용은 한 글자도 안 바뀜). "내용 그대로, 표현만 바꿔줘"가 이 원리. 검토 기준 = 정보가 아니라 읽히는 방식.

박스모델 — CSS를 말로 고치는 최소 지도

버튼·카드·문단은 모두 상자. 안에서 밖으로 content → padding → border → margin. 안쪽 여백 = padding, 바깥쪽 여백 = margin. 나란한 요소 사이 간격 = gap(flex/grid). 폭 정할 때 box-sizing: border-box를 쓰면 width 안에 padding·border 포함 → 예상보다 넓어지는 일 줄임.

이렇게 말하면AI가 볼 CSS
"카드 바깥에 여백 필요해"margin
"버튼 글자가 테두리에 너무 붙어"padding
"버튼들이 서로 너무 붙어"gap, margin
"카드에 테두리·둥근 모서리"border
"내용을 가운데로 모아줘"flex, justify-content, margin:auto
💬 요청 예: "카드 내용은 바꾸지 말고, 카드 바깥·버튼 사이 간격을 모바일에서도 읽기 좋게 조정해줘. 어떤 CSS를 바꿨는지도 알려줘."

JavaScript — 열흘 만에 태어난 언어

1995년 웹페이지는 눌러도 반응 없는 종이였다. 넷스케이프가 즉석 반응 언어를 원했고 브렌던 아이크가 첫 버전을 열흘 남짓에 만듦. 이름: Mocha→LiveScript→JavaScript(마지막은 인기 언어 Java에 올라탄 마케팅). "자바와 자바스크립트는 햄과 햄스터의 관계" — 이름만 닮은 남남. 겉모습만 Java, 속은 다른 언어.

첫 임무는 폼 검증. 느린 모뎀 시절, 제출 전 브라우저가 그 자리에서 "이메일 형식 아닙니다"라 말해주는 것. JS는 시간에 따라 변하는 것(클릭·입력·필터·토글)을 맡음. 검토 기준: "무엇을 누르면 무엇이 바뀌어야 하나"가 기대대로인가.

2005 구글 지도 — 지도를 끄는 동안 뒤에서 조각을 받아 이어붙임. 새로고침 없이 서버와 데이터 주고받는 기법 = AJAX. 웹이 '문서'→'애플리케이션'으로 넘어간 분기점. "JS는 한 번에 한 가지만 하는데 어떻게 동시에?" → 다음 CS 강에서 답.

CSS내용·꾸밈이 붙어 수정이 번짐읽히나, 깨지는 폭 없나
JS눌러도 반응 없는 정적 문서행동 뒤 변화가 기대대로인가
CS 교양 — 관심사의 분리: 데이크스트라(1974)가 말한 separation of concerns. 한 번에 한 측면만 집중. "페이지 고쳐줘"(범위 미지정) vs "카드 정보는 그대로, 모바일 간격만"(겹 지정) — 겹을 지정하면 손댈 범위가 좁아지고 괜찮은 부분이 덜 흔들린다.

프론트엔드 vs 백엔드 (식당 비유)

프론트엔드백엔드

두 요청을 섞어 던지면 화면은 만들었는데 저장이 빠지거나 그 반대가 나온다. 오늘은 프론트엔드 세 겹만. 백엔드·API는 뒤 강의.

2·3 · [오늘의 CS] 동기·비동기, 프로세스·스레드·코어 00:12

웹이 멈추지 않는 이유. 컴퓨터가 기다림을 다루는 방식.

동기 = 전화 📞

상대가 받을 때까지 붙잡힘. 그동안 다른 일 못 함. 한 작업 끝까지 다음 줄로 안 감.

비동기 = 문자 💬

보내놓고 하던 일 계속, 답 오면 확인. 오래 걸리는 일을 걸어두고 다음 진행.

카페 진동벨: 벨 없으면 카운터 앞에 서 있어야(동기, 뒷줄 길어짐), 벨 받으면 자리서 일하다 울리면 감(비동기). 커피 나오는 시간은 같고, 기다리는 동안 다른 일을 하느냐가 다름.

JavaScript는 일꾼이 한 명 (싱글 스레드)

웹페이지의 JS는 기본적으로 한 번에 한 가지만 = 싱글 스레드. 동기로만 일하면 서버 응답 3초 기다리는 동안 화면 전체가 멈춤(스크롤·버튼·영상 다). 그래서 오래 걸리는 일(fetch, setTimeout)은 비동기로 걸어두고, 브라우저가 뒤에서 처리 후 결과를 대기줄에 올림 → 일꾼이 하던 일 끝내고 대기줄 확인. 이 구조 = 이벤트 루프.

출생의 비밀: 새로고침 없이 데이터 주고받는 부품 XMLHttpRequest는 1990년대 말 MS 아웃룩 웹메일 팀이 만듦. 넷스케이프의 열흘 언어 + 경쟁사의 웹메일 부품이 만나 구글 지도에서 꽃핌. AI 코드에서 async·await·setTimeout·.then()을 보면 "여기는 기다리지 않고 걸어두는 부분"이라 읽으면 충분.

프로세스와 스레드 — 작업실과 일꾼

프로세스 = 실행 중인 프로그램 하나 = 독립된 작업실(자기 메모리, 남의 메모리 못 봄). 스레드 = 그 작업실 안 일꾼(여러 명 가능, 작업실 재료=메모리 공유).

2008 크롬: 예전 브라우저는 프로그램 전체가 프로세스 하나 → 탭 하나 오류가 전체를 넘어뜨림. 크롬은 탭마다 독립 프로세스(OS식 격리) → 탭 하나 죽어도 나머지 멀쩡. (작업 관리자에 크롬 항목 수십 개 = 탭·확장마다 작업실 따로.)

멀티스레딩멀티프로세싱
그림한 작업실에 일꾼 여러 명작업실 자체를 여러 개
메모리공유각자 독립
강점가볍고 재료 바로 주고받음하나 죽어도 나머지 안전
약점같은 재료 동시에 건드리면 꼬임무겁고 전달 비용

일꾼 둘이 같은 장부에 동시에 쓰면 꼬임 → JS가 아예 일꾼을 한 명으로 정한 것도 이 문제를 피한 선택.

코어 — 일손은 결국 몇 개인가

코어 = CPU 안 물리적 일손(8개면 진짜 동시에 8가지). 2000년대 초까지 속도(클럭) 경쟁 → 발열·전력 한계 → "적당히 빠른 걸 여러 개" = 멀티코어 시대. 허브 서터 "공짜 점심은 끝났다"(2005): 이제 가만히 있으면 안 빨라지고, 여러 일손을 쓰게 설계해야 빨라짐.

일손 8개가 수백 프로세스를 어떻게? OS가 아주 빠르게 번갈아 실행(사람 눈엔 동시). 이 시분할의 뿌리는 1960년대(존 매카시 제안, 코르바토 CTSS). 부산물: 여러 사람이 한 컴퓨터를 나눠 쓰며 파일 구분이 필요해져 비밀번호가 탄생.

🔎 내 컴퓨터 확인 (Day 4 터미널) — mac: sysctl -n hw.ncpu / Windows PowerShell: echo $env:NUMBER_OF_PROCESSORS. 코어 수 vs 프로세스 수 비교 — 일손보다 일감이 압도적인데도 안 멈추는 게 OS가 하는 일.
🤔 판단: ① 점수 +1 화면 표시 → 즉시(동기) / ② 서버에서 랭킹 받아 표시 → 비동기(기다림이 있으니).

2·4 · AI로 게임 만들기: 테트리스에서 내 게임까지 00:30 실습

두 판. 1차=테트리스(한 줄 요청), 2차=내가 고른 게임(규칙부터 설계). 요청 방법·도구가 왜 달라지는지가 관전 포인트.

🔧 1차 — "테트리스 만들어줘"

웹 채팅형 AI(claude.ai·ChatGPT)에 테트리스 만들어줘 한 줄. 미리보기가 떠서 바로 플레이. 한 줄이 통하는 이유: 테트리스는 1984년부터 규칙(블록·회전·줄삭제)이 AI 안에 있음 — 이름 자체가 규칙 명세서. 요청은 길이가 아니라 담긴 정보량이 가른다.

관찰까지가 연습: 코드 보기로 전환해 세 겹 찾기 — HTML(게임판·점수), CSS(색·배치, 보통 <style>), JS(키입력·낙하·줄삭제, <script>). 같은 요청도 사람·실행마다 결과가 다름(AI의 비결정성). 부가기능은 한 번에 하나씩 재요청("다음 블록 미리보기", "10줄마다 속도↑", "P키 일시정지", "게임오버 후 다시하기").
이 게임은 대화 창 안에 있어 내 컴퓨터엔 파일이 없다. 대화를 잃으면 게임도 사라짐 — 이 차이가 2차의 출발점.

🎮 2차 — 내가 고른 게임 (작업대 + 개발 에이전트)

이름만으론 안 통하는 게임이라 규칙을 내가 적는다. 도구도 채팅→Claude Code(작업대)로 바꿔 파일이 내 컴퓨터에 남게. 게임은 만든 사람이 첫 플레이어라 세 겹이 다 움직이고, 규칙을 내가 정해 AI가 규칙을 바꿔치기했는지 판정할 기준이 내 안에 있음.

게임의 최소 재료뜻가위바위보 예
보이는 것눈앞 요소점수판·가위바위보 버튼·결과 문구
상태"지금 상황"이김/짐/무승부, 선택 버튼 강조
규칙했을 때 상태가 어떻게 바뀌나내 선택 vs 컴퓨터 무작위 판정, 3승이면 종료

각 칸이 한두 줄이면 오늘 크기에 적당. 문단이 되면(레벨·저장·랭킹) 오늘보다 큼 → 규칙을 잘라 첫 판만.

🔧 2차 요청문 템플릿

새 폴더 my-game을 만들고, 그 안에 아래 게임을 만들어줘.

만들 게임: [가위바위보 / 숫자 맞히기 / 내가 정한 게임]
규칙: [진행 방식 2~3문장 직접]
끝나는 조건: [5판 승부 / 기회 10번 / 제한시간 30초]

파일은 index.html, styles.css, script.js 세 개로 나눠줘.
- 화면: [화면 요소]를 담아줘.
- 모바일에서도 조작 편한 크기로, [상태]가 색·문구로 보이게.
- 내가 적은 규칙 그대로 동작, 끝나는 조건이면 결과·다시하기 버튼.

규칙에 애매한 부분 있으면 만들기 전에 물어봐.
완료 후 브라우저 확인 방법과 수정 체크리스트를 알려줘.
핵심은 "세 파일로 나눠줘"가 아니라 화면·상태·규칙을 구체적으로 적은 것. 테트리스와 달리 이 규칙은 AI 안에 없으므로, 규칙까지 AI에 맡기면 이상해도 판정 기준이 없다.
브라우저 확인: 이기는/지는 경우 다 만들어 점수·문구·색 확인 · 버튼 연타 시 이중 점수/꼬임 없나 · 모바일 폭에서 누르기 편한가 · 끝 조건·다시하기(점수 초기화) · Tab 키만으로 플레이 가능한가 · Console 오류 없나.

2·5 · 바이브 코딩 루프: 관찰하고 다시 요청하기 00:15

AI와 일하는 리듬은 한 방향이 아니라 루프.

계획 맞추기 → 만들게 하기 → 결과 관찰 → 다시 요청 → (반복)

만들기 전에 PRD부터

PRD(제품 요구사항 문서) = "무엇을·누구를 위해·어떤 규칙으로"를 먼저 글로 못 박는 한 장. 바이브 코딩에선 더 중요 — AI는 내 마지막 한 마디에 맞춰 움직여 목표가 고정 안 되면 대화가 길수록 방향이 흘러감. 그래서 스펙 주도 개발(먼저 명세, 그다음 AI)이 권장됨. 앞의 화면·상태·규칙 표가 곧 PRD 뼈대(+누가·예외 상황).

바로 코드를 쓰지 말고, 위 PRD대로 어떻게 만들지 계획을 먼저 알려줘.
- 어떤 순서로 만들지
- 내가 미리 정해줘야 할 것이 있는지
계획이 내 생각과 맞으면 "진행"이라고 할게. 그때 만들어줘.
💡 손으로 코드 빨리 쓰는 힘은 이미 AI가 가졌다. 남는 실력은 무엇을 만들지 또렷이 정의하는 힘 — 좋은 명세를 쓰는 사람이 좋은 결과를 얻는다.

🔍 결과 관찰하고 다시 요청

재요청은 "어떤 CSS 속성 써라"(=AI의 일)가 아니라 "어디서 무슨 문제를 봤고, 어떻게 되길 원한다"를 평소 말로.

[어디서] · [무슨 문제] · [어떻게 되면 좋겠다]
  • "결과 문구가 알기 어려워. '이겼습니다/졌습니다'처럼 분명하게 바꿔줘."
  • "모바일에서 버튼이 좁아 누르기 불편해. 커지거나 줄바꿈되게 해줘."
  • "무승부인데 점수가 올라가. 이겼을 때만 오르게 해줘."

고쳐 돌려주면 그 부분이 실제로 바뀌었는지 직접 플레이/코드로 확인. "전체 다시 만들어줘"보다 문제 부분만 짚기(범위 좁으면 괜찮은 부분이 덜 흔들림).

2. GitHub Pages를 켜서 웹에서 열 수 있는 주소를 만들어줘.
3. 로그인·권한 확인처럼 내가 직접 눌러야 하는 단계는 멈추고 알려줘.
4. 배포가 끝나면 접속 주소를 알려주고, 잘 열리는지 확인 방법도 알려줘.
⚠️ 배포엔 계정 로그인·권한 승인처럼 사람이 직접 해야 하는 단계가 섞임 — AI가 멈추면 그 부분만 웹 화면에서 처리 후 이어가기. 배포 직후 주소가 뜨기까지 1~2분 걸릴 수 있음.
✅ 제출 전 체크: 링크를 새 창(시크릿)에서 열면 게임이 뜬다 · 내 컴퓨터 아닌 그 웹 주소에서 조작이 동작한다 · 휴대폰으로 열어도 플레이된다 · 링크를 동료에게 보내도 열린다.
📮 제출: 배포된 게임 링크(예: https://내아이디.github.io/my-game/). 제출 방법은 과정 안내(LMS 과제 제출란 또는 지정 채널)를 따른다.

3·1 · 프레임워크는 왜 이렇게 많을까 00:05

건축 현장에서 철골과 비계가 건물의 뼈대를 잡아주듯, 프레임워크는 웹서비스의 반복되는 구조를 미리 세워 준다.

지난 시간엔 웹페이지를 HTML·CSS·JS 세 겹으로 나눠 봤다. 손으로 만든 작은 페이지는 이 세 겹만 알아도 충분하다. 그런데 "프론트엔드 프레임워크 추천"을 검색하는 순간 상황이 달라진다 — React, Vue, Svelte, Angular, Solid, Qwik, 거기에 Next.js·Nuxt·SvelteKit 같은 메타프레임워크까지 붙는다. 이름을 하나씩 누르다 보면 질문이 바뀐다: "웹서비스 하나 만들려면 이걸 다 알아야 하나요?"

오늘의 답 = 아니요. 다만 막연한 위로면 안 된다. 왜 도구가 이렇게 많아졌는지, 어떤 문제를 덜어주려고 생겼는지 알아야 AI가 추천한 답도 판단할 수 있다. 그래서 오늘은 문법이 아니라 "프레임워크"라는 말의 뜻을 먼저 잡고, 내 프로젝트 조건을 AI에게 설명해 선택 근거를 비교하는 연습을 한다.
💡 오늘의 산출물은 코드가 아니다. "내 조건에서 어떤 프레임워크가 맞는지"를 AI에게 제대로 물을 수 있는 선택 프롬프트와 평가표. 이게 있어야 Claude Code에게 "React로 만들어줘" 같은 막연한 지시 대신, "내 조건에서는 왜 이 선택이 맞는지 설명하고 대안을 비교해줘"라고 요청할 수 있다.

🔬 먼저 짚기 — '프레임워크'가 대체 뭔가

프레임워크(framework)는 직역하면 '뼈대' 또는 '틀'. 집을 지을 때 매번 기둥·배관·전선을 0부터 설계하지 않고, 검증된 골조를 기준 삼아 그 위에 방 배치와 생활 방식을 올리는 것과 같다. 웹 프레임워크도 마찬가지 — 같은 버튼·카드를 여러 곳에서 다시 쓰기, 데이터가 바뀌면 화면을 함께 바꾸기, 주소에 따라 다른 화면 보여주기, 배포 가능한 형태로 파일 묶기처럼 반복되는 일을 미리 정리해 둔 개발 골조다.

이 골조는 부품 상자와 다르다. 부품 상자인 라이브러리는 필요할 때 꺼내 쓰는 느낌이고, 프레임워크는 "이 안에서는 화면을 이런 방식으로 나누고 연결하자"는 약속까지 함께 준다. 개발자들은 이 차이를 제어의 역전(Inversion of Control)이라 부른다 — 라이브러리는 내가 필요할 때 불러 쓰지만, 프레임워크는 전체 흐름을 쥐고 있다가 정해진 순간에 내가 쓴 코드를 불러 준다. 그래서 혼자 빠르게 만들 때도 도움이 되지만, 여러 사람이 같은 프로젝트를 오래 고칠 때 더 큰 힘을 낸다. 누가 들어와도 대략 어느 폴더를 봐야 하는지, 화면 이동과 데이터 갱신이 어디서 관리되는지 짐작할 수 있기 때문이다.

도메인으로 바꿔보면: 의사라면 전자차트(EMR), 변호사라면 사건관리 SaaS, 마케터라면 광고 운영 콘솔. 매번 업무 시스템의 기본 틀을 새로 만들지 않고 검증된 구조 위에 조직의 절차·데이터를 얹는다 — 프레임워크는 웹 화면을 만들 때 쓰는 그 기본 틀에 가깝다. 이름이 수십 개인 이유도 여기서 출발한다: 팀마다 중요하게 여긴 반복 업무가 달랐고, 웹서비스가 커지며 필요한 골조도 조금씩 달라졌다.

3·2 · 버튼 하나 고치려다 서른 곳을 손댄 날 00:10

레고 블록처럼, 화면을 재사용 가능한 조각(컴포넌트)으로 나누고 규격에 맞춰 조립하는 것이 프레임워크의 핵심 발상 — 프레임워크가 풀려는 세 가지 문제.

프레임워크는 개발자를 괴롭히려고 생긴 이름표가 아니다. 화면이 커질 때 반복해서 터지는 문제를 줄이려고 생겼다. 지난 시간에 본 세 겹은 여전히 유효하다 — HTML은 구조, CSS는 모양, JS는 움직임. 다만 순수 HTML/CSS/JS만으로 화면이 커지면 세 겹을 관리하는 부담이 함께 커진다. 처음엔 버튼 하나 고치면 끝났지만, 화면이 열 개가 되고 입력값과 결과 목록이 서로 연결되기 시작하면 같은 문제가 여러 곳에서 되풀이된다.

간단한 역사 — 반복을 덜어내는 20년

이 반복을 줄이려는 시도는 2000년대부터 이어져 왔다. 이름이 바뀌어도 풀려는 문제는 같았다.

연도등장풀려던 문제
2006jQuery"적게 쓰고 더 많이 하자" — 브라우저마다 다른 JS 호환성 문제를 한 번에 덮음. 큰 도움이었지만 화면이 수십 개로 늘면 코드가 여전히 뒤엉킴
2010AngularJS · Backbone.jsGoogle·오픈소스 — "화면을 구조적으로 나누자"를 본격화. 데이터가 바뀌면 화면도 규칙적으로 바뀌는 발상 등장
2013ReactFacebook(Meta) — 화면을 컴포넌트라는 재사용 조각으로 나눔. "어떻게 바꿔라"가 아니라 "무엇이어야 한다"를 선언하는 선언형 UI가 업계 기본이 됨
2014VueEvan You — React의 컴포넌트 사고를 받아들이되 더 점진적으로 배우는 구조. "쓰고 싶은 만큼만 쓴다"
2016~현재Svelte · Solid · Qwik각자 다른 병목 겨냥 — Svelte는 컴파일러로 브라우저 부담↓, Solid는 반응형 성능 극대화, Qwik은 서버 상태를 브라우저가 이어받는 resumability
여기까지는 모두 프론트엔드(화면)를 다루는 프레임워크. 프레임워크는 화면만의 이야기가 아니다 — 눈에 안 보이는 서버(백엔드) 쪽에도 같은 이유로 수많은 도구가 있다: 파이썬 Django·FastAPI, 자바 Spring, Node.js Express, 루비 Rails. 언어도 여러 개, 프레임워크도 여러 개인 까닭은 결국 하나 — "규모가 커질 때, 반복되는 일을 어떻게 줄일 것인가?" 프론트든 백엔드든 답의 방향은 늘 같았다.

🧭 오늘의 범위 — 백엔드는 나중에 따로 다룬다. 오늘은 눈에 보이는 화면, 프론트엔드로 좁혀서 이야기한다.

커질 때 생기는 문제순수 웹 코드에서 자주 겪는 모습프레임워크가 덜어주려는 일
반복비슷한 카드·폼·버튼 코드를 여러 곳에 복사화면 조각을 한 번 만들어 여러 곳에서 다시 쓰게 함
변화선택값·로딩중·실패·완료 같은 현재 상황을 여러 코드가 따로 바꿈데이터가 바뀌면 화면도 일정한 규칙으로 함께 바뀌게 함
연결화면이 늘수록 파일이 커지고, 어느 주소에서 어떤 화면을 보여줄지 헷갈림폴더 구조, 주소별 화면 연결, 빌드·테스트 관행을 맞춤

개발자들은 이 세 문제를 설명할 때 자주 컴포넌트·상태·라우팅이란 말을 쓴다. 낯설어 보여도 방금 표의 문제를 다른 이름으로 부르는 것에 가깝다.

🔬 표를 개발자 말로 번역하면

컴포넌트(component)는 화면을 이루는 재사용 가능한 조각. 같은 모양의 전문가 카드가 화면에 30개 있다면, 카드를 30번 복사해두면 이름 옆에 자격 배지 하나 추가할 때 30곳을 다 고쳐야 한다. 컴포넌트로 만들면 카드 설계를 한 곳에 두고 데이터만 바꿔 여러 번 불러올 수 있다. 도메인 서비스에서는 환자 목록의 한 행, 사건 카드, 광고 성과 요약 박스가 모두 컴포넌트가 될 수 있다.

상태(state)는 화면이 지금 어떤 형편인지 알려주는 현재값 — "검색어가 무엇인가", "결과를 불러오는 중인가", "로그인되어 있는가", "저장에 실패했는가" 같은 값. 사용자가 필터를 바꾸면 결과 수·카드 목록·빈 화면 안내 문구가 함께 바뀌어야 한다. 이 연결을 손으로 흩어놓으면 한 곳은 바뀌고 다른 곳은 예전 값으로 남는 버그가 생긴다. 프레임워크는 상태와 화면 사이의 연결 규칙을 한곳에 모아두게 돕는다.

라우팅(routing)은 주소(URL)에 따라 어떤 화면을 보여줄지 정하는 길 안내. /search는 검색 화면, /profile은 내 정보 화면, /reports/2026-06은 특정 보고서 화면처럼 주소와 화면을 연결한다. 페이지가 한두 개일 땐 크게 안 느껴지지만, 서비스가 커지면 주소 체계 자체가 사용자의 길 찾기이자 팀의 작업 지도 역할을 한다.

오늘은 "프레임워크가 반복되는 화면 조각, 바뀌는 현재값, 주소별 화면 연결을 함께 관리하도록 돕는다"는 정도만 가져가면 충분하다. 이 셋은 앞으로 더 자세히 다시 만난다.

🔬 눈으로 보기 — 같은 카드를 순수 JS와 React로

"컴포넌트가 왜 좋은지"는 말로만 들으면 와닿지 않는다. 같은 화면 — 전문가 프로필 카드 — 을 순수 JavaScript와 React 컴포넌트로 각각 만들어 비교한다. 도메인에 맞게 환자 카드·사건 카드·광고 성과 박스로 바꿔 생각해도 된다.

// ExpertCard.jsx: 카드 설계를 "한 번" 만듭니다
function ExpertCard({ name, badge, bio }) {
  return (
    <div className="expert-card">
      <h3>{name}</h3>
      <p className="badge">{badge}</p>
      <p className="bio">{bio}</p>
    </div>
  );
}

// ExpertList.jsx: 데이터만 바꿔서 여러 번 부릅니다
const experts = [
  { name: '김민준', badge: '형사전문 15년', bio: '서울중앙지검 특수부 출신.' },
  { name: '이서연', badge: '민사전문 12년', bio: '대형 로펌 파트너.' },
  // ... 30명
];

function ExpertList() {
  return (
    <div>
      {experts.map(expert => (
        <ExpertCard key={expert.name} {...expert} />
      ))}
    </div>

}
배지에 아이콘을 추가하고 싶다면? ExpertCard 컴포넌트 한 곳만 고치면 30개 카드가 모두 바뀐다 — 이것이 모듈화와 재사용의 힘이다.
💡 오늘 외울 필요 없음 — 이 코드를 당장 이해하지 않아도 괜찮다. 핵심은: 순수 JS에서는 같은 구조를 30번 복사해야 하지만, 컴포넌트에서는 한 번 만들고 데이터만 바꿔 쓴다. 이 차이가 "프레임워크가 왜 생겼는가"에 대한 가장 직접적인 답.

🔬 상태가 터지는 순간 — 필터를 바꾸면 무엇이 따라 바뀌어야 할까

컴포넌트가 "재사용"을 해결한다면, 상태(state)가 해결하는 건 "바뀔 때 함께 바뀌기". 시나리오: 전문가 검색 화면에서 사용자가 "형사전문" 필터를 골랐다면, 화면에서 바뀌어야 하는 곳은 한 곳이 아니다.

바뀌어야 하는 곳순수 JS에서의 위험상태가 있으면
필터 버튼 표시(선택됨)깜빡하면 "선택된 것처럼 안 보임"상태가 바뀌면 자동 갱신
결과 목록(형사전문만)목록만 바꾸고 카운트를 안 바꿈목록·카운트가 함께 갱신
결과 수("3명 찾음")목록은 바꿨는데 텍스트를 안 바꿈같은 상태를 공유하므로 일관됨
빈 결과 안내이 문구를 띄우는 조건을 빠뜨림상태가 비어 있으면 자동으로 표시
페이지 제목("형사전문 변호사 — 검색 결과")이것까지 바꾸는 걸 잊음상태에서 파생되는 값으로 자동 계산

순수 JavaScript에서는 이 다섯 곳을 개발자가 하나하나 챙겨야 한다. 한 곳이라도 빠지면 "필터는 바뀌었는데 목록은 예전 것"이나 "목록은 바뀌었는데 카운트는 안 바뀜" 같은 버그가 생긴다.

프레임워크에서는 "필터 값"이라는 하나의 상태를 중심에 두고, 그 상태에서 나머지가 파생된다.

// React 예시: selectedFilter 상태가 바뀌면,
// 아래 모든 것이 자동으로 일관되게 갱신됩니다
function ExpertSearch({ experts }) {
  const [selectedFilter, setSelectedFilter] = useState('all');

  // 상태에서 파생되는 값들 — 개발자가 따로 안 바꿈
  const filtered = selectedFilter === 'all'
    ? experts
    : experts.filter(e => e.specialty === selectedFilter);

  return (
    <div>
      {/* 1. 필터 버튼 */}
      <FilterButtons
        selected={selectedFilter}
        onSelect={setSelectedFilter}
      />

      {/* 2. 결과 수 — 자동 계산 */}
      <p>{filtered.length}명 찾음</p>

      {/* 3. 빈 결과 안내 — 자동 표시 */}
      {filtered.length === 0 && (
        <p>조건에 맞는 전문가가 없습니다.</p>
      )}

      {/* 4. 결과 목록 — 자동 갱신 */}
      {filtered.map(expert => (
        <ExpertCard key={expert.name} {...expert} />
      ))}
    </div>
  );
}
개발자는 selectedFilter를 바꾸는 한 줄만 적는다. "결과 수 텍스트도 바꿔라, 빈 결과 문구도 봐라, 목록도 다시 그려라"는 프레임워크가 알아서 한다. 한 곳을 바꾸면 연결된 곳이 모두 일관되게 바뀌는 것, 이것이 상태 관리가 주는 가장 큰 이점.
💡 도메인으로 바꿔 생각하기 — 환자 목록에서 "당뇨" 필터를 고르면 목록·결과 수·차트·안내 문구가 모두 바뀌어야 한다. 광고 성과에서 "지난 7일"을 고르면 표·그래프·요약 숫자·비교 문구가 다 바뀌어야 한다. 이런 "한 값이 여러 곳을 당겨 바꾸는" 상황이 많을수록 상태 관리의 이점이 커진다.

🧠 CS 교양 — 추상화와 모듈화

프레임워크가 왜 반복을 덜어주는지 한 겹 더 들어가면, 컴퓨터과학에서 오래 다듬어 온 두 생각과 만난다 — 추상화(abstraction)와 모듈화(modularity). 이름은 거창하지만 일상에서 이미 쓰는 방식이다.

추상화는 복잡한 속을 감추고 꼭 필요한 조작만 밖으로 드러내는 것. 자동차를 몰 때 엔진의 연소 과정을 몰라도 페달·핸들이라는 단순한 손잡이만 알면 된다. 웹에서도 마찬가지 — 브라우저는 화면 속 제목·버튼·목록을 DOM(Document Object Model)이라는 메모리 구조로 관리하는데, 순수 JavaScript로는 "이 버튼을 찾아 색을 바꾸고, 이 목록의 세 번째 항목을 지워라"처럼 DOM을 직접 만지는 일이 많았다. 프레임워크는 이 손 많이 가는 DOM 조작을 손잡이 뒤로 감춘다 — 개발자는 "지금 상태가 이러면 화면은 이렇게 보여야 한다"만 적고, 실제로 어떤 노드를 지우고 바꿀지는 프레임워크가 알아서 한다. 그래서 "어떻게 바꿔라" 대신 "무엇이어야 한다"를 적는 선언형(declarative) UI도 추상화의 한 모습이다.

모듈화는 큰 덩어리를 따로따로 이해하고 갈아 끼울 수 있는 조각으로 나누는 것 — 레고 블록. 블록 하나하나는 단순하지만 규격이 맞아 조립·교체할 수 있다. 앞에서 본 컴포넌트가 바로 이 모듈화다. 전문가 카드 하나를 독립된 조각으로 만들어두면 그 조각만 따로 고치거나 다른 화면에 그대로 옮길 수 있다. 조각 사이의 약속(어떤 데이터를 받아 무엇을 보여줄지)만 지키면, 한 조각을 고쳐도 다른 조각이 함께 무너지지 않는다.

💡 바이브 코딩에서 왜 중요한가 — AI에게 화면을 만들라고 할 때, 좋은 지시는 결국 이 두 언어로 된다. "전문가 카드를 컴포넌트로 만들어 목록에서 재사용해"(모듈화), "검색 결과가 바뀌면 목록이 따라 바뀌게 해"(상태=추상화)처럼. 문법은 AI가 채우지만, 무엇을 조각으로 나누고 무엇을 상태로 둘지는 여러분이 판단한다 — 이 판단이 프레임워크 문법보다 오래 쓸모 있다.
📖 역사 한 걸음 — 추상화·모듈화는 새 발상이 아니다. 1972년 미국 태생 컴퓨터과학자 데이비드 파나스(David Parnas)는 "큰 프로그램을 설계할 때, 서로 관련된 것끼리 모으는 게 아니라 바뀌어야 할 이유가 같은 것끼리 모아야 한다"고 주장했다. 이 원칙은 50년이 지난 지금도 React의 컴포넌트 분리, Vue의 파일 구조, Svelte의 컴파일 단위 설계 어디에나 살아 있다. 도구는 바뀌어도 복잡함을 조각으로 나누는 생각은 변하지 않았다.

이 두 가지는 프레임워크만의 이야기가 아니다. 함수로 반복 계산을 감추고, 파일·폴더로 코드를 나누고, 라이브러리로 남이 만든 기능을 가져다 쓰는 것 모두 추상화·모듈화다. React 공식 문서가 UI를 컴포넌트로 나눠 묘사하는 것, Vue 공식 문서가 상태 변화에 맞춰 화면을 갱신하는 반응성(reactivity)을 앞세우는 것도 결국 이 두 생각의 표현.

🧠 CS 교양 — 가상 DOM: 설계도를 먼저 그리고 차이만 칠하기

React가 DOM 조작을 손잡이 뒤로 감추는 방식이 가상 DOM(Virtual DOM)이다.

직접 칠하기 (실제 DOM 조작)

"저 벽은 연한 파랑, 이 벽은 흰색"이라며 하나하나 붓을 들고 감. 벽이 100개면 손이 100번 감.

설계도에 먼저 표시 (가상 DOM)

리모델링 뒤 모습을 설계도에 먼저 그리고, 지금 집과 나란히 놓고 다른 점만 찾음. 바뀐 벽이 3개면 붓은 3번만 들고 감.

React가 하는 일: ① 가상 DOM이라는 메모리 위의 설계도를 먼저 그린다 ② 현재 화면(실제 DOM)과 설계도를 비교한다 ③ 바뀐 부분만 찾아 실제 DOM에 적용한다. 이 과정을 재조정(reconciliation)이라 부른다 — 이름은 어려워도 하는 일은 "설계도와 현실의 차이를 찾아 바뀐 것만 고치기".

가상 DOM 이전엔 화면을 바꿀 때 개발자가 직접 어떤 요소를 찾아 바꿀지 정해야 했다. 상태가 바뀌면 어디를 바꿔야 하는지, 실수로 빠진 곳은 없는지, 너무 많이 바꿔서 느려지지 않는지 — 이 모든 걸 개발자가 챙겼다. 가상 DOM은 이 부담을 추상화로 감췄다. 개발자는 "무엇이어야 한다"만 적고 "어떻게 바꿀지"는 React가 알아서 한다 — 앞서 본 선언형 UI가 실제로 어떻게 작동하는지 보여주는 장치다.

오해 하나 — 가상 DOM이 항상 가장 빠른 건 아니다. Svelte처럼 컴파일 시점에 "무엇을 바꿔야 할지"를 미리 알아내는 방식은 실행 시점 비교가 필요 없어 더 빠를 수 있다. 가상 DOM의 강점은 "가장 빠르다"가 아니라 "개발자가 직접 DOM을 만지지 않아도 된다"는 편의성. 추상화는 속도보다 일관성과 개발 효율을 위한 선택이다.

  • Angular — Google이 유지하는 웹 프레임워크. 라우팅·폼·업데이트 도구 묶음이 강함. 큰 조직에서 장기 운영하는 서비스에 자주 쓰임
  • Svelte — 컴파일러를 활용해 브라우저에서 해야 할 일을 줄이는 방향
  • Solid — 반응형 UI의 성능을 극한까지 밈
  • Qwik — 서버 렌더링 상태를 브라우저에서 이어받는 resumability를 강조

이름마다 문제를 푸는 관점이 다르다. "어떤 문제를 푸는가"를 보면 왜 하나만 정답이 아닌지 보인다.

📐 문법은 AI가 짠다 — 그래서 '무엇을 고르냐'가 더 중요해졌다

먼저 안심해도 된다. React든 Vue든 Svelte든 코드 문법은 외울 필요가 없다. "상품 목록 컴포넌트 만들어줘"라고 하면 AI가 그 프레임워크 문법으로 정확히 써준다. 프레임워크마다 생김새는 달라도, 화면을 조각(컴포넌트)으로 나누고 데이터(상태)로 갱신한다는 뼈대는 다 같다.

그런데 "AI가 다 짜주니 아무거나 골라도 되겠네?"는 오해다. 오히려 선택이 더 중요해졌다 — AI가 프레임워크마다 코드를 짜는 정확도가 실제로 다르기 때문.

근거 ① — AI는 자료 많은 프레임워크를 더 정확히 짠다. 프런트엔드 코드 생성을 평가한 학술 벤치마크 DesignBench에서, 최신 AI 모델들은 React·Vue로는 컴파일 성공률이 0.83을 넘겼지만 Angular는 0.69~0.76으로 눈에 띄게 낮았다. AI 코딩 벤치마크들이 기준 프레임워크로 React를 자주 고르는 이유도 "학습 데이터에 가장 많이 담겨 있어서"다. (출처: DesignBench, arXiv:2506.06251, 2025) 즉 공개 자료가 많은 프레임워크일수록 AI의 첫 코드가 정확하다.

근거 ② — 규칙이 일관된 프레임워크일수록 AI가 일관되게 짠다. "이건 이렇게 하는 게 정석"이라는 규칙이 강한 프레임워크는 AI가 매번 비슷한 구조로 코드를 낸다. 반대로 자유도가 아주 높은 도구는 부를 때마다 구조가 달라져, 검토·수정할 게 많아진다.

3·4 · AI와 함께 PRD 만들기 00:22

나침반은 방향을 알려주지만 걸을지 말지는 결정하지 않는다. AI도 마찬가지 — 후보와 근거를 정리해주지만, 최종 선택은 조건을 아는 사람이 한다.

🗣️ AI와 대화하며 정하기 — 인터뷰 → 추천 → PRD

좋은 개발은 "○○ 만들어줘" 한 방으로 시작하지 않는다. 무엇을 만들지 AI와 주고받으며 정하는 편이 훨씬 좋은 결과를 낸다. 아래 세 단계로 대화한다.

1단계 — AI에게 나를 인터뷰하게 한다

한 번에 다 설명하려 하지 말고, AI가 되묻게 만든다.

내가 만들고 싶은 건 [수제 디저트 온라인 가게]인데,
[선물 사는 사람이 편하게 고르도록] 하고 싶어.
구체적인 PRD를 짜기 위해 정해야 할 것들을, 한 번에 하나씩 나에게 물어봐 줘.

그러면 AI가 "주 사용자는 누구인가요? 상품은 어떻게 분류되나요? 결제까지 필요한가요?"처럼 되묻는다. 답하다 보면 흐릿하던 아이디어가 정리된다. 모르는 항목은 "아직 모르겠어, 네가 제안해 줘"라고 답해도 된다.

2단계 — 프레임워크를 추천받는다

대화로 윤곽이 잡히면, 이제 만들 도구를 고른다.

이제 이 서비스를 만들기에 어떤 프레임워크가 좋을지,
3개 이상 장단점과 함께 추천해 줘.
여기서 나온 답은 곧이곧대로 받아들이지 말고, '왜 이걸 추천했는지', '내 조건에서도 정말 맞는지'를 한 번 더 되물어본다.

3단계 — PRD로 정리받는다

마지막으로 지금까지의 대화를 문서로 남긴다.

지금까지 정한 내용을 PRD로 작성해 줘.
추후 백엔드(로그인·저장·결제) 기능을 추가할 것도 고려해서 정리해 줘.

이 PRD가 오늘 과제의 뼈대이자, 나중에 백엔드 같은 기능을 붙여 나갈 때 길잡이가 된다.

📚 더 나은 공부법 — 지금 단계에서는 프로그래밍 문법을 파고드는 것보다, AI와 함께 만든 이 PRD와 개발 계획을 꼼꼼히 읽어보는 것이 훨씬 좋은 학습이다. 코드가 무엇을, 왜 하려는지 위에서 내려다보는 눈이 생기고, 그 눈이 있어야 AI가 만든 결과도 판단할 수 있기 때문이다.

3·5 · 백엔드 없이 프론트엔드 만들기 00:08

노점은 본점 주방이 완성되길 기다리지 않는다. 백엔드가 없어도 가짜 데이터로 화면을 먼저 돌려볼 수 있다 — 실무 개발자가 매일 쓰는 기술이자 AI 코딩에서 가장 빠른 프로토타이핑 열쇠.

프론트엔드 vs 백엔드 — 식당의 홀과 주방

식당웹서비스하는 일
홀(hall)프론트엔드손님이 직접 보고 만지는 공간 — 메뉴판 보고 주문하고 음식 받음
주방(kitchen)백엔드안 보이는 곳에서 재료 저장·요리·완성
서빙 창구API홀과 주방의 연결점 — 주문 전달·완성품 내보냄
주문서 양식JSON주고받는 공통 형식. "번호·메뉴·수량"처럼 정해진 규격

프론트엔드가 데이터를 요청할 땐 HTTP 요청을 쓴다. 브라우저가 서버에 "이 데이터 주세요" 하면 서버가 JSON으로 답한다.

// 프론트엔드가 백엔드에 "전문가 목록을 주세요"
const response = await fetch('https://내서비스.com/api/experts');
const experts = await response.json();

문제는 이 코드가 돌려면 응답하는 백엔드가 이미 있어야 한다는 것. 없으면 fetch는 실패하고 화면은 빈다.

💡 그래서 목업 데이터(mock data). "백엔드 없으면 프론트엔드 못 만드나?" → 아니다. 진짜 API가 보낼 것과 같은 구조의 가짜 데이터를 코드에 직접 적어두면, 백엔드 없이도 화면이 진짜처럼 동작한다.

🎭 목업 데이터가 유용한 이유 (실무에서 매일)

  • 백엔드를 안 기다림 — 프론트·백엔드 팀이 동시에 일하는 병렬 개발이 기본
  • 디자인·인터랙션 먼저 검증 — 데이터가 어떻게 들어올지 알면 배치·상호작용을 미리 테스트
  • AI 코딩에 특히 강력 — "이 데이터로 화면 만들어줘" 하면 백엔드 연결 없이 완성 화면을 즉시 봄
  • 나중에 진짜 API로 쉽게 교체 — 목업 구조를 진짜 응답과 맞춰두면 한 줄만 바꿈
🔧 가짜 데이터는 AI에게 부탁 — 손으로 안 적어도 됨. 알려줄 건 둘: ① 어떤 화면인지(쇼핑몰 상품 목록) ② 무엇을 보여줄지(이름·가격·품절 여부). + "실제 인물·기관 이름은 빼고 가상의 값으로". 요령: 극단 사례(목록이 텅 빔·이름이 유난히 긺)도 섞어달라 → 그런 상황에서 화면이 깨지는지 미리 확인.

🔄 목업 → 진짜 API, 교체가 쉬운 구조

데이터를 컴포넌트 안에 직접 박으면 교체할 때 컴포넌트 열 개를 다 뒤져야 한다. 별도 파일로 빼두면 백엔드가 생겼을 때 import 한 줄만 API 호출로 바꾸면 끝 — 화면 코드는 손 안 댐. 이것이 지난 시간 배운 관심사의 분리(separation of concerns)의 실전.

import mockExperts from './mockData';   // ← 이 한 줄만 바꾸면 끝
function ExpertList(){
  const experts = mockExperts;          // → useFetch('/api/experts')
  // ... 화면 코드는 그대로
}

3·6 · [CS 교양] API가 하는 일 이해하기 00:10

창구 너머로 실제 무엇이 오가는지, 화면을 만들 때 알아야 할 만큼만.

REST API — URL로 데이터를 요청

가장 널리 쓰이는 방식. 주소창에 URL 치는 것과 본질이 같되, 화면이 아니라 데이터를 달라고 한다.

요청:  GET https://내서비스.com/api/experts?specialty=형사
의미:  "형사 전문 변호사 목록을 줘"
응답(JSON): { "experts": [ {"id":1,"name":"김민준",...} ], "total": 2 }
동사URL 예식당 비유의미
GET/api/experts"메뉴판 보여줘"목록을 가져옴
POST/api/experts"이거 주문할게"새 데이터를 만듦
PUT/api/experts/1"1번 이렇게 바꿔줘"기존 데이터 수정
DELETE/api/experts/1"1번 취소할게"데이터 삭제
💡 바이브 코딩 팁: AI에게 "이 화면에 POST 요청 추가해줘"라 하면 이 약속을 안다. 프론트엔드 프로토타입에선 GET(가져오기)·POST(등록)가 가장 자주 쓰인다.

GraphQL — 원하는 필드만 골라 받기

REST는 하나의 URL이 정해진 덩어리를 돌려준다(이름·전화·사진 다). GraphQL은 "내가 필요한 필드만 골라 달라"고 요청 → 통신이 가벼워짐. 대신 백엔드 구조가 더 복잡해짐.

관점RESTGraphQL
학습 난이도낮음 (URL+동사면 충분)중간 (쿼리 문법)
데이터 효율정해진 덩어리(필요없는 것도)요청한 필드만 정확히
AI 코딩 친화도자료 많고 AI가 익숙느는 중이나 REST만큼은 아님
🧭 첫 프로토타입은 REST가 기본. 자료 많고 AI가 잘 다루며 목업 구조 맞추기 쉬움. 서비스가 커져 "요청이 너무 많다" 느낄 때 GraphQL을 후보에. 지금은 GET·POST만 알아도 충분.

역사: RPC(1980s, 원격 함수 호출) → SOAP(1990s, XML 엄격 규격, 은행·의료·정부) → REST(2000s, URL+HTTP동사, 웹 사실상 표준) → GraphQL(2015, Meta, 필드 선택). 형식은 바뀌어도 "요청하고 응답한다"는 본질은 같다.

📖 JSON이 XML을 이긴 이유 — XML은 <name>김민준</name>처럼 태그로 감싸 문서엔 좋지만 프로그램엔 무거웠다. JSON("name":"김민준")은 JavaScript 객체 문법과 같아 브라우저가 거의 가공 없이 바로 씀 → 가벼움·단순함이 이겼다.

단계내용

② 프레임워크 판단"뭐가 좋아요?"가 아니라 조건을 준다(화면·검색노출·자료 많음·나중에 백엔드). AI는 보통 Next.js를 첫 후보로 답함. 여기서 멈추지 말고 되물음 — 후보 하나만 미나? 근거 없이 "제일 좋아요"만 하나? 내 조건을 반영했나?

판단을 3문장으로: "Next.js를 골랐다. ① 선물 검색 유입이 중요한데 검색 노출에 유리하고, ② 비전공자가 참고할 한국어 자료·예제가 많으며, ③ 나중에 백엔드를 같은 틀에서 붙이기 쉽다." 이 3문장이 "유행 따라 골랐다"와 "조건 보고 골랐다"를 가른다.

4·1 · 화면 뒤에 서버가 필요한 이유 00:14 수강중

"링크 보냈는데 안 열려요" — 내가 만든 화면이 아직 세상에 없는 이유. 남이 못 열고, 껐다 켜면 사라진다 — 이 두 증상이 출발점.

장면 하나 — 밤새 만든 화면, 친구는 못 본다

어젯밤 근사한 화면을 만들어 친구에게 http://localhost:3000 을 자랑한다. 친구: "안 열리는데? 이 사이트에 연결할 수 없음." 게다가 노트북을 껐다 켜니 입력해 둔 내용이 싹 사라져 있다.

세 가지 증상, 하나의 원인

증상진짜 원인
새로고침하면 입력이 사라진다데이터를 저장해 둘 곳이 없다
친구에게 주소를 줘도 안 열린다localhost는 '내 컴퓨터'라는 뜻이다
노트북을 닫으면 화면도 꺼진다서비스가 내 컴퓨터에서만 돌아간다
공통 원인 하나: 지금까지 만든 것은 내 컴퓨터 안에만 있고 세상과 연결된 '뒤편'이 없다. 그 뒤편을 맡는 것이 백엔드(서버).

localhost가 뭐길래

localhost는 "지금 이 컴퓨터"를 가리키는 특별한 주소. 친구가 자기 컴퓨터에서 localhost:3000을 열면 친구 자기 컴퓨터의 3000번을 찾는다 — 거기엔 아무것도 없으니 "연결할 수 없음". 내 방 주소를 알려줬는데 친구가 자기 집에서 그 방을 찾는 셈.

먼저 예측해보기 — 서버가 필요한 기능은?

서버(백엔드)가 없으면 안 되는 것에 O, 화면(프론트)만으로 되는 것에 X.

클라이언트와 서버 — 손님과 24시간 주방

클라이언트 (내 브라우저·앱)

주문하러 온 손님. 왔다가 간다. 여러 명일 수 있다.

서버

24시간 켜져 있는 주방 겸 창고. 늘 그 자리에서 주문을 받고(요청)·음식을 내주고(응답)·재료를 보관(저장).

구분식당 비유하는 일
프론트엔드(화면)손님 눈에 보이는 홀메뉴판·자리·분위기
백엔드(서버)눈에 안 보이는 주방데이터 저장·"이 손님 회원 맞나" 확인·계산 처리

지금까지 만든 화면은 "홀 인테리어"만 있는 셈 — 예쁘지만 주방이 없으면 진짜 주문은 못 받는다. 오늘 주방에서 직접 요리(코드)하는 건 AI가 한다. 나는 "주방이 어떻게 돌아가고 무엇을 조심해야 하는지"를 알아 AI에게 제대로 시키고 결과를 판단하는 것이 목표.

🎯 오늘의 핵심 한 문장: 백엔드는 화면 뒤에서 데이터를 저장하고, 사람을 확인하고, 세상과 연결하는 부분이다. 코드는 AI가 쓰지만, 무엇이 필요하고 무엇이 위험한지 판단하는 것은 내 몫.

짧은 역사 — 인터넷은 어쩌다 이렇게 연결됐나

시기사건남긴 것
1969ARPANET — 미국 몇몇 대학 컴퓨터를 처음 연결한 실험"멀리 있는 컴퓨터끼리 메시지를 주고받자" = 인터넷의 씨앗
1970~80s공통 규칙 TCP/IP — 데이터를 잘게 나눠(패킷) 주소를 붙여 보내자오늘의 IP 주소
1989~91웹(WWW) — CERN 팀 버너스리가 "문서를 링크로 잇자" 제안·공개HTTP(주고받는 규칙) · URL(웹 주소)
2000s~클라우드 — 아마존·구글이 "우리 서버를 빌려 쓰세요"비전공자도 몇 분 만에 서비스를 올린다
"멀리 있는 컴퓨터를 잇자"(1969) → "공통 규칙"(TCP/IP) → "문서를 링크로"(웹·HTTP) → "서버를 빌려 쓰자"(클라우드)

손님이 몰리면 — 서버는 어떻게 버티나 (개념만)

수평 확장 · 로드 밸런서

똑같은 서버를 여러 대 두고 손님을 나눠 보낸다. 안내 직원 = 로드 밸런서("지금은 2번 주방으로").

캐시 · CDN

인기 이미지·파일을 손님과 가까운 여러 지점에 미리 복사 → 지구 반대편에서도 빠르게. 이것이 CDN.

비전공자가 지금 직접 다룰 일은 없다 — 클라우드가 대부분 자동. 다만 "인기를 끌면 서버 비용이 왜 오르나", "큰 서비스는 왜 안 느려지나"를 이해하는 감각으로 알아두면 좋다.
📦 오늘의 산출물: 코드가 아니라 "내 서비스를 세상에 내보내는 데 필요한 것들의 지도". 그 지도를 쥐면 AI에게 "이렇게 만들어 줘"라고 자신 있게 말할 수 있다.

4·2 · 네트워크 기본기 — 주소가 화면이 되기까지 00:18

백엔드를 이해하려면 "인터넷이 어떻게 연결되는지" **최소한만** 알면 된다. mysite.com 입력 → 화면이 뜨기까지 딱 네 조각: ①도메인·DNS ②서버·호스팅 ③포트·방화벽 ④HTTP·HTTPS.

① 이름을 주소로 — 도메인과 DNS

인터넷의 모든 컴퓨터에는 번호(IP 주소, 예 142.250.72.14)가 있다. 외우기 불편하고 수시로 바뀌기도 한다.

도메인 (mysite.com)

사람이 외우는 가게 이름. "김밥천국"처럼.

DNS

이름 → 번호로 바꿔 주는 전화번호부. "mysite.com 어디?" → "142.250.x.x".

도메인은 사는 것(연 1만원 안팎) → "이 도메인은 저 서버(IP)를 가리켜라" 연결(DNS 설정).

흔한 함정 — 왜 바로 안 바뀌지? DNS는 답을 한동안 기억(캐시)해 둬서, 바꿔도 즉시 반영이 안 된다. 몇 분~몇 시간이면 전 세계로 퍼짐(전파). "방금 바꿨는데 왜 안 바뀌지?"의 가장 흔한 정답.

도메인은 '별칭' 그 이상 — 이름 한 겹을 두는 힘

진짜 힘은 이름과 실제 서버 사이에 '한 겹'을 두는 것(추상화). 앞(도메인)은 그대로, 뒤(서버·IP)는 얼마든지 바꾼다.

따라오는 것뜻
서버 옮겨도 주소 그대로서버가 죽어 새 IP를 받아도 'DNS 한 줄'만 고침. 상호는 그대로.
하나의 이름 → 여러 서버부하 분산·지역별 안내(CDN). 하나 죽어도 잘 안 멈춤.
서브도메인 구조화api.·blog.·shop.로 갈라도 브랜드는 하나.
신원·신뢰의 증표이메일 @뒤·HTTPS 인증서가 '도메인'에 발급 → 정체성.
내 이름으로 보유·이동호스팅 바꿔도 들고 감 → 브랜드 자산.
자주 하는 오해 — 자물쇠(HTTPS)는 '믿을 만한 회사' 증명이 아니다. '이 도메인 주인과 암호 걸어 통신 중'이란 뜻일 뿐 → 피싱 사이트도 자물쇠를 붙일 수 있다. 자물쇠 유무보다 도메인 철자를 확인.

② 그 주소엔 무엇이 — 서버와 호스팅

서버 = 내 파일·데이터를 올려 두고 24시간 켜진 컴퓨터. "노트북 닫으니 꺼진" 이유가 이것. → 호스팅(늘 켜진 컴퓨터를 빌려 씀)을 쓴다. 서버를 직접 살 필요 없음.

예전지금(클라우드)
회사가 서버 직접 구입 — 전원·냉방·보안·고장 다 챙김아마존·구글 서버 창고에서 필요한 만큼만 빌림. 고장·보안·전원은 그쪽이. 클릭 몇 번이면 24시간 서버 생김.

내부망과 외부망 — "왜 내 노트북에선 되는데 밖에선 안 보이지?"

내부망(LAN)

집·회사 안 공유기로 묶인 '우리끼리'. 사설 IP(192.168.0.5).

외부망(WAN=인터넷)

그 바깥 전 세계. 밖에서 우리 집은 공유기 하나(공인 IP 하나)로만 보임.

문지기 = 공유기(NAT) — 안→밖은 자유, 밖→안은 기본 차단(그래서 평소 안전). 내 localhost:3000이 나한테만 보이는 이유. '세상에 연다' = 내부망의 것을 외부망에서 닿게 만드는 일(클라우드 서버에 올리거나 터널로 연결).

③ 어느 문으로 — 포트와 방화벽

서버 한 대는 여러 일(웹·메일·DB)을 동시에 한다 → "이 요청은 어느 창구로"를 정하는 게 포트.

  • IP='건물 주소', 포트='호실 번호'(0~65535). 웹은 보통 80(HTTP)·443(HTTPS). localhost:3000의 3000이 포트.
  • 방화벽 = 문 지키는 경비. 필요 없는 문은 닫아 둠.
"내 컴에선 되는데 밖에선 안 열려요"의 흔한 원인 = 열려야 할 문(포트)이 방화벽에 막힘. 배포 플랫폼은 대개 알아서 열어 줌 — 개념만 기억.

④ 무슨 말로 — HTTP와 HTTPS

브라우저↔서버 공통 언어 = HTTP("이 페이지 줘"→"여기 있어").

[요청]  GET /products      ← "products 목록 줘(GET)"
        Host: mysite.com
[응답]  200 OK             ← "성공! 여기 있어"  (+ 화면/데이터)

HTTPS = 이 대화를 암호로 감싼 안전한 버전(주소창 자물쇠). HTTP=엽서(엿봄 가능), HTTPS=밀봉 봉투. 로그인·결제 있으면 필수. 요즘 배포 플랫폼이 자물쇠(SSL)를 자동으로 붙여 줌.

참고 — API: 프로그램끼리 정해진 창구로 요청·응답을 주고받는 약속. "AI야, 지도 API 붙여줘"의 그 API가 HTTP 요청이 지나는 창구.

작동 원리 — 자물쇠(공개키)

열쇠가 두 개: 공개키=누구나 넣는 우편함 투입구(공개 OK) / 개인키=여는 유일한 열쇠(서버만). 누구나 잠글 순 있어도 열 수 있는 건 서버뿐. 접속 초반 공개키로 '임시 열쇠(세션 키)'를 주고받는 첫 인사 = TLS 핸드셰이크. 진짜 사이트 보증은 인증기관(CA)의 서명 인증서로.

작동 원리 — 패킷(TCP/IP)

데이터는 통째로가 아니라 작은 조각(패킷)으로 쪼개져 각자 여행 후 재조립. ①우편(머리표=주소·순번) ②피자(한 조각만 다시) ③내비(막히면 우회 → 일부 고장나도 안 멈춤). 이 공통 규칙이 TCP/IP.

요청 한 번의 여정 (엔터 치면)

브라우저가 DNS에 묻는다 — "mysite.com IP 어디?" (배송지 검색)
IP를 받아 서버에 연결 — 443번 문, HTTPS 자물쇠 (차량 이동)
"이 페이지 줘" HTTP 요청 → 서버가 화면·데이터 응답 (물건 받아 옴)
브라우저가 응답을 화면으로 그림 (포장 풀어 정리)

목업 데이터는 이 여정 없이 화면 안에 미리 넣은 것, 진짜 서비스는 매번 이 왕복을 돈다. 백엔드는 그 왕복의 반대편에서 대답하는 쪽.

4xx요청(나)의 잘못401 "누구세요?"(로그인) · 403 "넌 안 돼"(권한) · 404 "그런 페이지 없어" · 429 "천천히 좀"
5xx서버의 잘못500 "내가 터졌어"
실전 팁 — 4xx면 내 요청을 고치고, 5xx면 서버(만든 사람) 문제. AI에게 "500 에러 나" 하고 화면의 에러를 그대로 붙여 주면 원인을 훨씬 잘 찾음. (재미: 418 I'm a teapot은 1998 만우절 농담)
직접 해보기 — 아무 사이트에서 우클릭→검사→Network 탭→새로고침 → 맨 위 줄 Status에 200 같은 숫자. 안 뜰 때 빨간 줄(4xx·5xx)을 찾으면 단서. (출처: MDN)

4·3 · 백엔드가 실제로 하는 일 — 요청 받아 처리하고 돌려주기 00:16

백엔드가 하는 일 한 문장 — 요청을 받아서, 처리하고, 응답을 돌려준다. 정적 화면이 '창고 완성품을 그대로 꺼내 줌'이면, 백엔드는 주문마다 그때그때 만드는 동적 처리.

주방 비유백엔드
홀(프론트)에서 주문서를 받고요청 접수
레시피대로 조리하고비즈니스 로직(규칙 검사·계산)
창고에서 재료를 꺼내고데이터베이스 읽기/쓰기
완성한 요리를 내보냄응답

손님은 주방 안을 못 보고 '주문'과 '나온 요리'만 주고받음 — 백엔드가 자신을 감추는 방식.

개념 — 창구(엔드포인트)와 API

백엔드는 통째로 열지 않고 정해진 창구(엔드포인트)만 연다. 창구 하나 = "주소(URL) + 동작" = 기능 하나.

GET  /products   → "상품 목록 주세요"
POST /login      → "이 정보로 로그인 처리해 주세요"

이 창구들의 묶음이 API. 관례(REST) 핵심 감 — 주소는 '무엇'(명사), 동작은 메서드로. /getProducts(동사)❌ → /products(명사)⭕. 비전공자에게 필요한 건 완벽한 설계가 아니라 "내 서비스엔 어떤 창구가 필요하지?" 판단.

개념 — CRUD (거의 모든 서비스가 하는 네 가지)

C·R·U·D동작API 짝
Create만들기POST
Read읽기GET
Update고치기PUT/PATCH
Delete지우기DELETE

인스타 '게시물 올리기·보기·수정·삭제', 쇼핑몰 '상품 등록·조회·수정·삭제' — 겉만 다르고 속은 이 넷의 반복. "어떤 창구 필요?"="이 데이터에 CRUD 중 뭘 허용?"

개념 — 공용어 JSON

{ "name": "홍길동", "age": 30, "isMember": true }

'이름표: 값' 쌍의 가벼운 텍스트. 사람이 읽기 쉽고 언어를 안 가려서 어떤 프론트·백엔드를 붙여도 알아들음 → 사실상 표준. 문법 외울 필요 없이 "이렇게 생긴 걸로 주고받는구나"면 충분.

백엔드는 무슨 언어로 짜나

언어대표 프레임워크

그 밖Ruby on Rails · Go · Java/Spring

무난한 기본값 — ①JS 풀스택(언어 하나로 통일·자료 많음) ②Python(읽기 쉽고 AI 붙이기 좋음). 어차피 코드는 AI가 짬 → 나는 개념(요청·창구·CRUD·검증)만 잡고 방향 판단.

🔒 보안 — 백엔드의 진짜 가치: "프론트는 못 믿는다"
백엔드의 가치는 데이터를 나르는 게 아니라 규칙을 지키는 것(재고·최종가격·권한 = 비즈니스 로직). 가격·수량·권한 검증은 반드시 백엔드에서 다시. 프론트는 사용자 컴퓨터에서 도니 얼마든지 조작 가능 — 화면에 "10,000원"이어도 악의적 사용자는 100원으로 바꿔 보낼 수 있다. 최종 가격은 서버가 자기 데이터로 다시 계산. (OWASP)

4·4 · 프로토타입과 프로덕트 사이 — AI가 약한 곳 00:16

바이브 코딩의 흔한 실패 — 첫날 "하루 만에 화면이!" → 다음 주 "로그인 풀려·저장 사라져·클라우드 요금 폭탄". AI는 "돌아가는 코드"는 잘 만들지만 "안전하고 튼튼한 코드"는 놓친다. 명시적으로 요구하지 않으면 보안을 알아서 안 챙김.

보안회사 Veracode 실험 — AI 생성 코드의 약 45%에 보안 결함. 모델이 더 똑똑해져도 이 비율은 거의 안 줆(문법은 좋아져도 안전은 그대로). (2025 GenAI Code Security Report)

① 보안 — AI 코드의 최대 약점

가장 흔한 구멍 = "입력을 그대로 믿는" 실수. 검색창 이름 칸에 나쁜 사람이 명령어처럼 생긴 문자열을 넣고, 서버가 그대로 실행하면 → DB가 통째로 털리거나 지워짐(SQL 인젝션).

규칙 하나 — "사용자가 보낸 값은 일단 의심하고 검사한다." 비전공자가 할 일은 프롬프트로 요구하는 것:
"사용자 입력을 검증하고, 비밀번호는 해시로 저장하고, 모든 통신은 HTTPS로. 흔한 취약점(입력값 주입, XSS)을 막아줘."

보안 이야기 — 역사에 남은 유출 사고 (대부분 막을 수 있던 것들)

사고내용·교훈
에퀴팩스(2017)알려진 취약점 방치 → 1억 4,300만 명 주민번호·생일·주소 유출
소니 PSN(2011)약 7,700만 명 정보 유출, 서비스 장기 중단
야후 보이스(2012)비밀번호를 평문 저장했다 통째로 유출 → "비번은 해시로"의 이유
마이스페이스(2008)약 3.6억 계정 → 교훈: 같은 비번 재사용 금지

공통점 — 입력 함부로 안 믿기·비번은 해시로·알려진 취약점 방치 않기. 큰 회사도 기본을 놓쳐서 뚫린다.

② 환경변수 — 비밀은 코드에 적지 않는다

급한 마음에 API 키를 코드에 적어 GitHub에 올림 → 몇 분 만에 봇이 훔쳐 클라우드에서 비싼 작업 → 며칠 뒤 수백만 원 청구서.

GitGuardian 집계 — 2025년 공개 GitHub 유출 비밀키·비번 약 2,865만 개(전년비 +34%, 역대 최대). AI 서비스 키 유출 1년 새 +81%. (State of Secrets Sprawl 2026)
나쁜 예:  apiKey = "sk-a1b2c3d4..."   ← 그대로 올라가면 순식간 유출
좋은 예:  apiKey = 환경변수에서 불러오기  ← 실제 값은 코드 밖 .env 에
          (.env 는 .gitignore 로 깃에 안 올라가게)
AI에게 — "API 키·DB 접속정보는 코드에 넣지 말고 환경변수로 빼줘. .env는 깃에 안 올라가게." 한 줄: 코드=공개해도 되는 것 / 환경변수=절대 공개 안 될 것.

③ 인증 vs 인가 — 표 검사원과 VIP 바운서

인증(Authentication)

입구 표 검사원 — "표 주인 맞아요?"=로그인 확인. 실패=401.

인가(Authorization)

VIP 바운서 — "이 방 들어갈 권한 있어요?" 없으면 403.

로그인 성공 → 서버가 토큰(클럽 손목 밴드)을 줌 → 이후 요청마다 밴드만 보이면 통과. 비밀번호는 절대 그대로 저장 금지 — 해시(한쪽으로만 계산, 달걀→프라이는 되지만 프라이→달걀 불가)로 저장. AI가 만들어도 "권한 확인(인가) 되어 있나?"·"비번 해시로 저장되나?"는 꼭 되물을 것.

작동 원리 — 해시와 솔트

  • 해시 = 비번을 '되돌릴 수 없는 지문'으로. 같은 비번=같은 지문. 서버는 지문만 저장·비교.
  • 솔트(salt) = 사람마다 다른 무작위 값을 섞음 → 같은 123456도 사람마다 지문이 달라짐.
  • bcrypt = 일부러 느리게 계산 → 도둑이 수억 개 대입하기 어려움(느린 게 장점).
AI에게 — "비밀번호는 솔트를 섞어 bcrypt로 해시해서 저장해줘."

작동 원리 — 쿠키와 세션 (웹은 원래 건망증이 심하다)

초창기 HTTP는 요청마다 처음 보는 손님 취급(무상태) → 로그인 유지하려 나온 게 쿠키·세션.

쿠키

내 지갑 속 작은 카드. 사이트가 번호를 적어 주면 방문 때마다 보여 줌.

세션

사이트 금고 속 내 서류철. 카드 번호를 보고 "아, 이 사람!" 하며 정보(장바구니·권한)를 꺼냄.

흐름 — ①로그인 시 서버가 세션 만들고 고유번호(세션 ID) 발급 → ②쿠키로 브라우저에 줌 → ③요청마다 쿠키 함께 보냄 → ④서버가 번호로 나를 알아봄. 진짜 정보는 브라우저(쿠키)에 두면 안 됨(조작 가능) — 브라우저엔 의미 없는 번호만, 진짜는 서버 금고에.

정리 — AI는 아주 빠른 신입 개발자. 속도는 놀랍지만 보안·비밀 관리·권한 확인은 감독이 필요. 세 가지 — 입력 검증 · 비밀은 환경변수 · 권한 확인(+비번 해시) — 을 프롬프트로 요구하고 결과를 한 번 더 되묻는 것. 그것이 비전공자의 가장 강력한 품질 관리.

4·5 · 세상에 내보내기 — 배포 00:16

배포(deploy) = 내 컴퓨터에만 있던 서비스를 늘 켜진 서버에 올려 누구나 주소로 들어오게 하는 것. 예전엔 며칠 걸리던 전문가의 일, 지금은 비전공자도 클릭 몇 번+AI로 몇 분.

어디에 올리나 — 셋으로 나눠 (식당 비유)

무엇을어디에(예)식당 비유왜 여기
화면(프론트)Vercel · Netlify전 세계 여러 지점의 홀손님 가까이서 빠르게 보여줘야

각자 잘하는 게 달라 나눠 맡김. 무료로 시작하면 서버는 Render(무료 티어 유지)가 무난(Railway는 시작 크레딧 소진 후 유료).

GitHub에 올리면 자동 배포

내 코드 → GitHub 저장소 → 배포 플랫폼(Vercel 등)과 "연결" → 코드 바뀔 때마다 자동 재배포 → 주소(예: 내프로젝트.vercel.app) 생성

한 번 연결해두면 다음부턴 고치고 저장만 해도 세상에 반영. 어젯밤 안 열리던 그 링크가 이제 어디서든 열림.

초보가 가장 많이 막히는 세 곳: ① 환경변수 빼먹기 — .env(비밀값)는 GitHub에 안 올리므로 배포 플랫폼 설정에 따로 넣어야("로컬은 됐는데 배포는 안 돼"의 1위) ② CORS 막힘 — 화면(Vercel)과 서버(Render)가 다른 주소면 서버가 "이 화면 요청 허용"을 열어줘야(AI에게 "내 프론트 주소 CORS 허용해줘") ③ 무료의 함정 — 트래픽 늘면 유료 전환.
💸 요금 폭탄 방지: 요금 알림 켜기 · 무료 한도 알기 · 비밀 키 지키기(유출 키로 남이 돌린 요금도 내 몫 → 그래서 환경변수). 알고 시작하면 거의 예방 가능.

4·6 · 공짜로 세상에 올리기 — 무료 서버 고르기 00:14

비전공자도 무료로 웹앱을 띄울 곳이 여럿. 단 공짜의 세계는 계속 줄어드는 중 — Heroku(2022 종료)·Railway(2023)·Fly.io(2024)·Glitch(2025)가 상시 무료를 거둠(무료 계정의 채굴 악용 탓). 가입 직전 공식 요금 페이지 재확인이 안전.

4·7 · 내 컴퓨터를 서버로 — 집에서 돌려보기 00:12

"내 컴퓨터 켜두면 되잖아?" — 되긴 되지만 기본적으로는 밖에서 안 닿는다. 집 기기는 공유기가 준 사설 IP(192.168.…)를 쓰고, 밖에선 집이 공유기 하나로만 보임(NAT). 게다가 집 회선 주소는 대개 유동 IP.

전통 방법 — 포트 포워딩 + DDNS (원리엔 좋지만 주의 큼)

공유기에서 특정 문(포트)을 내 서버로 열고, DDNS로 바뀌는 주소를 따라가게. 단 ① 집 네트워크가 인터넷에 직접 노출(전 세계 공격 프로그램이 문 두드림) ② 요즘 많은 회선은 주소를 여러 집이 나눠 쓰는 CGNAT라 포트 포워딩 자체가 안 되는 경우 많음.

요즘 편한 무료 방법 — '터널'

터널은 발상을 뒤집음 — 밖에서 들어오게 여는 대신 내 컴퓨터가 바깥 서비스로 먼저 연결해 그 통로로 손님을 받음. 그래서 포트 개방 불필요·CGNAT 통과.

4·8 · 바이브 코딩으로 프론트-백엔드 연결하기 00:14

조각이 다 모임(화면 + 네트워크·백엔드·보안·배포). 마지막으로 AI로 풀스택(화면+서버)을 이을 때 — 초보가 가장 많이 헤매는 지점.

① 목업 화면 먼저 → ② 백엔드 API 붙이기 → ③ 가짜 데이터를 실제 호출로 교체

앱 전체를 한 프롬프트에 통째로 시키는 건 초보에게 거의 항상 실패(AI의 집중력·기억 한계). "데이터 모양 → API → 화면"처럼 작게 쪼개 시키면 실수를 일찍 잡음.

연결에서 실제로 겪는 문제 → 처방:
① 계약 불일치(최대 원흉) — 프론트가 기대하는 데이터 모양 ≠ 백엔드가 주는 모양(AI 쓰면 오히려 심해짐, 양쪽에 '얼추 맞는' 걸 따로 만들어 어긋남). 증상: 화면에 undefined. 처방: 주고받을 데이터를 처음부터 문서로(API 명세) 적고 그걸 기준으로 양쪽을 짬
② 로딩·에러 상태 누락 — 모든 호출을 로딩/성공/에러 세 상태로 나눠 각각 화면 준비
③ CORS·주소·환경변수 — 배포 때 그 문제들이 곧 연결 문제. 백엔드가 내 프론트 주소 허용 + 주소는 환경변수로
연결을 매끄럽게 하는 프롬프트 전략: API 명세를 문서로(근본 해결) — "엔드포인트 목록과 각 요청/응답 예시(JSON)를 명세 문서로 만들고, 앞으로 이 문서 기준으로 프론트·백엔드 짜자" · 한 기능씩 끝까지(수직 슬라이스) — "회원가입 하나"를 화면-API-저장까지 완성하고 바로 테스트 · 양쪽을 같은 대화·저장소에 · 매 단계 커밋(AI는 확신에 차서 멀쩡한 걸 망가뜨리기도).
🔒 내보내기 전 보안 점검 프롬프트: "이 코드 전체를 하드코딩된 비밀값·서버 입력 검증 누락·CORS 설정·인증 취약점 관점으로 점검해줘." AI는 보안이 나빠서가 아니라 프롬프트에 보안이 없어서 "일단 돌아가는" 코드를 만듦(NEXT_PUBLIC_·VITE_ 접두사 값은 "브라우저 공개"란 뜻 → 비밀값 금지).

5·1 · 데이터를 저장할 곳이 필요한 이유 00:12

지난 시간엔 백엔드(눈에 안 보이는 주방)를 봤다. 주방이 주문을 처리하는 건 알았는데 — 만든 데이터는 어디에 두나?

메모리(잠깐 기억) vs 데이터베이스(영구 창고)

지금까지 다룬 데이터(목업·화면에 잠깐 뜬 값)는 컴퓨터의 잠깐 기억(메모리)에만 있었다. 그래서 껐다 켜거나 새로고침하면 사라진다. 서비스가 되려면 데이터가 남아 있어야 한다. 그 "영구 창고"가 바로 데이터베이스(DB)다. 우리 데이터는 결국 물리적 저장장치 어딘가에 실제로 적힌다 — '재료를 두는 곳'이 여기.

데이터베이스가 뭐냐면 — 도서관, 그리고 마법 장난감 상자

마구 넣어도 "파란색 자동차만 다 꺼내 줘" 하면 정리해서 딱 그것만 꺼내 준다.

핵심: 데이터베이스는 그냥 쌓아 두는 곳이 아니라, 원하는 것만 빠르게 찾아 주는 곳이다. (출처: Stackby, "What is a database: A beginner's guide")

5·2 · 왜 종류가 많을까 — 데이터베이스의 역사 00:12

쇼핑몰 주문 목록을 엑셀에 적던 두 사람이 같은 행을 동시에 고쳤다. 한 사람의 수정이 다른 사람 내용을 덮어쓰며 하루 주문이 꼬였다. 데이터를 함께 쓰고 바뀐 내용을 잃지 않으려는 문제가 데이터베이스를 낳았다.

"DB도 종류가 왜 이렇게 많아?" — 프레임워크와 똑같다. 매 시대 병목이 달랐고, 그때마다 새 도구가 나왔다.

반복을 줄이려는 60년 — 연표

시기무슨 일
1960년대그냥 파일에 담다. 데이터를 파일에 쭉 적어 둠 → 양이 늘자 원하는 걸 찾기가 너무 번거로워짐.
1970년"표로 나누자" — 관계형의 탄생. IBM 연구원 에드거 코드(Edgar Codd)가 "데이터를 표(테이블)로 나누고 표끼리 관계로 잇자"는 논문. 같은 정보를 여기저기 중복 안 해도 되게 한 혁신. (출처: Codd, "A Relational Model of Data for Large Shared Data Banks", 1970)
1974년질의 언어 SEQUEL 등장(뒤에 SQL로 개명). 50여 년 지난 지금도 거의 그대로 쓰임 = 잘 만들어졌다는 뜻.
1979년첫 상용 관계형 DB Oracle. 회사들이 쓰기 시작.
2000년대인터넷 폭발, 그리고 NoSQL. 데이터 폭증·형태 제각각 → 엄격한 표만으론 버거워짐 → 더 유연하고 대규모에 강한 NoSQL 등장. (출처: Dataversity, "A Brief History of Non-Relational Databases")
오늘날클라우드 데이터베이스 — 켜 두고 관리하는 일까지 클라우드가 대신. 비전공자도 무료로 진짜 DB를 씀.

이름이 많은 건 어렵게 만들려는 게 아니라, 각 시대의 문제를 풀며 쌓여 온 것. (출처: Cockroach Labs, "A brief history of databases")

이야기 하나 — 코드의 '표 혁명'과 IBM의 망설임

1970년 당시 데이터는 나뭇가지처럼 얽힌 구조로 저장됐다. 원하는 걸 꺼내려면 그 가지를 타고 들어가는 전용 프로그램을 매번 짜야 했다. 코드는 IBM 연구소에서 이걸 보다 "그냥 엑셀 같은 표로 두고, 표끼리 관계로 이으면 안 되나?"라는 논문을 냈다(Communications of the ACM, 1970). 데이터의 '생김새'와 '저장 방식'을 분리하자는, 지금 보면 당연하지만 당시엔 파격적인 생각.

재미있는 반전: 정작 그 IBM이 처음엔 미지근했다. 이미 IMS라는 기존 DB 제품이 잘 팔리고 있어, 새 아이디어가 효자 상품을 갉아먹을까 봐 망설인 것. 결국 뒤늦게 'System R' 연구를 시작하지만, 이 틈을 노린 회사가 따로 있었다. 새 기술이 좋아도, 기존에 잘 팔리는 걸 가진 쪽은 주저한다 — 기술사에서 반복되는 장면. (출처: Wikipedia, "Edgar F. Codd"; Codd, 1970)

이야기 둘 — 'SQL'은 원래 '시퀄'이었다

IBM의 System R 팀(도널드 체임벌린, 레이먼드 보이스)이 처음 만든 이름은 SEQUEL — "Structured English Query Language(구조화된 영어식 질의 언어)". 영어 문장처럼 읽히게 만들자는 뜻이었다. 그런데 'SEQUEL'이 이미 어느 영국 항공기 회사의 상표로 등록돼 있었다. 변호사 편지를 받은 팀은 모음을 빼고 SQL로 줄였다.

오늘날 "에스큐엘"이라 읽기도, "시퀄"이라 읽기도 하는 이유가 여기 있다 — 만든 사람들조차 옛 이름 그대로 "시퀄"이라 부른다. 어느 발음을 들어도 당황하지 말 것. 둘 다 같은 걸 가리키고, 둘 다 맞다. (출처: Wikipedia, "SQL"; 체임벌린 회고)

이야기 셋 — 관계형을 만든 IBM보다 빨랐던 신생 회사

IBM이 망설이는 사이, 코드의 논문을 읽고 "이거 돈이 되겠다"고 확신한 사람 — 래리 엘리슨(Larry Ellison). 1977년 동료들과 작은 회사를 차렸고(SDL → 관계형소프트웨어 → 결국 오라클), 1979년 첫 상용 관계형 DB를 시장에 내놓았다. 관계형을 처음 만든 IBM(SQL/DS는 1981년)보다 2년 앞선 것. 거대 기업이 자기 논문을 붙들고 고민하는 동안, 신생 스타트업이 그 아이디어로 먼저 시장을 차지했다.

재미난 뒷이야기: 첫 제품 버전을 '1'이 아니라 '2'라고 붙였다. 갓 나온 소프트웨어를 아무도 안 믿을까 봐, 처음부터 "이미 한 번 개선했다"는 인상을 주려던 것. (출처: Wikipedia, "Oracle Corporation" / "Oracle Database")

이야기 넷 — NoSQL의 진짜 뜻은 'No'가 아니라 'Not Only'

2000년대, 구글과 아마존은 전에 없던 고민에 빠진다 — 전 세계가 동시에 접속하는 규모를 어떻게 감당하지? 기존의 반듯한 표 방식만으론 버거웠다. 그래서 각자 새 방식을 논문으로 공개한다(구글 Bigtable 2006, 아마존 Dynamo 2007). 표의 엄격함을 조금 포기하는 대신 엄청난 규모·속도를 얻는 방식.

이 흐름에 이름이 붙은 건 2009년 샌프란시스코의 한 개발자 모임 — 짧고 검색하기 좋은 이름으로 "NoSQL"이 정해졌다. 그런데 이 이름이 "SQL을 쓰지 말자"는 오해를 낳아, 나중에 "Not Only SQL(SQL만 쓰는 게 아니다)"로 재해석된다. 실제로도 NoSQL은 관계형을 대체한 게 아니라 용도에 따라 나눠 쓰는 또 하나의 선택지가 되었다. (출처: Wikipedia, "NoSQL"; Martin Fowler, "NosqlDefinition")

그래서 오늘날 큰 서비스들은 "SQL이냐 NoSQL이냐"로 싸우지 않는다. 둘 다 필요한 곳에 쓴다.

5·3 · SQL 구경과 데이터의 모양 00:14

카페 사장님이 "오늘 케이크 주문만 보고 싶어요"라는데 주문이 수백 건. 눈으로 못 찾는다. 필요한 조건의 데이터만 골라 달라고 DB에 요청하는 말 — 그게 SQL이다.

SQL은 거의 영어다 — 알아보는 정도면 충분

겁먹을 것 없다. 한 줄만 보자.

SELECT 이름, 가격 FROM 상품 WHERE 분류 = '케이크';

풀어 읽으면 — "상품 표에서(FROM 상품) 분류가 '케이크'인 것들의(WHERE) 이름과 가격을 줘(SELECT)." 정말 영어 문장에 가깝다.

단어하는 일
SELECT꺼내 보기 (조회)
INSERT새로 넣기
UPDATE고치기
DELETE지우기

이 네 가지를 묶어 CRUD(만들기·읽기·고치기·지우기). 대부분의 서비스는 결국 이 네 동작의 반복이다. 외울 필요 없다 — 실제 SQL은 AI가 써 준다. 여러분은 "아, 이 표에서 뭘 꺼내는(넣는) 거구나"를 알아보는 정도면 충분.

표로 생각하기 — 행, 열, 그리고 관계

관계형 데이터베이스는 결국 표다. 엑셀을 떠올리면 된다.

용어뜻예
표 하나한 종류상품 표
행(row)한 건딸기 케이크 하나
열(column)항목이름·가격·분류

그리고 표끼리 관계로 잇는다. 주문 표는 "어떤 상품을 샀는지"를 상품 정보를 통째로 베끼지 않고 각 상품의 고유 번호(ID)로 가리킨다. 이 "표로 나누고 관계로 잇기"가 바로 관계형(relational)이라는 이름의 뜻.

해 보기 — 쇼핑몰 데이터를 '표 몇 개'로 쪼개면

한 덩어리로 뭉치면 이렇게 된다 — "홍길동이 3/2에 초코케이크 2개를 12,000원에 샀고, 홍길동 주소는 …". 주문이 100건이면 홍길동 주소가 100번 적힌다. 그래서 성격이 다른 것끼리 표를 나눈다.

표담는 것예시 한 줄
회원사람 정보1번 · 홍길동 · 서울…
상품파는 것7번 · 초코케이크 · 12,000원
주문언제 누가 무엇을55번 · 1번 회원 · 7번 상품 · 2개 · 3/2
핵심: 주문 표가 이름·주소를 다시 안 적고 "1번 회원"이라고만 가리키는 것. 주소가 바뀌면 회원 표 한 곳만 고치면 끝. 표를 잇는 이 '번호'를 키(key)라 부른다 — 표와 표를 잇는 고리. 직접 설계할 필요 없다. AI에게 "쇼핑몰에 필요한 표들과 관계를 그려줘" 하면 만들어 준다. 내 일은 그림을 읽고 "내 서비스엔 어떤 표가 더 필요하지?"를 판단하는 것.

표는 어떻게 '이어지나' — 1:N과 N:M

① 일대다(1:N) — 한쪽이 여럿을 가진다

회원 1명은 주문 여러 건, 주문 1건은 한 회원 것. '주문' 표에 "누구의 주문인지"(회원 번호) 칸 하나만 넣으면 끝. 블로그 '글 1 ↔ 댓글 여럿', 유튜브 '채널 1 ↔ 영상 여럿'도 같은 모양.

② 다대다(N:M) — 양쪽 다 여럿

학생 1명이 여러 수업, 수업 1개도 여러 학생. 양쪽 어디에도 다 못 적는다(한 칸에 욱여넣으면 검색이 깨짐). 그래서 연결 표(수강신청 표)를 새로 만들어 학생 번호·수업 번호를 한 줄씩 짝지음 → N:M이 '연결 표를 사이에 둔 두 개의 1:N'으로 풀린다.

판단 감각 하나면 됨: "A 하나가 B 여럿을, 동시에 B 하나도 A 여럿을 갖나?" → 그렇다면 N:M(중간 표 필요), 한쪽만 여럿이면 1:N. 이 판단만 해 주면 표 만드는 SQL은 AI가 쓴다.

작동 원리 — DB는 왜 그렇게 빠를까 (책 뒤 '찾아보기')

두꺼운 전공책 맨 뒤 '찾아보기'("삼투압 → 245쪽") 덕에 처음부터 한 장씩 안 넘기고 바로 간다. DB의 색인(index)이 정확히 이것 — 자주 찾는 항목(이메일·상품명)에 색인을 만들어 두면 수백만 건에서도 순식간. 색인이 없으면 처음부터 전부 뒤진다(그래서 느려짐).

색인은 공짜가 아니다: '값 → 행 위치'를 미리 정리한 구조라 빠르지만, 넣고 고칠 때마다 색인도 갱신해야 하고 공간도 든다. 아무 데나 다 걸면 오히려 느려진다. 실전 요령 — AI에게 "색인 넣어줘"만 하지 말고, 먼저 "사용자는 자기 주문만 최근순 20개 본다"처럼 실제 화면의 질문을 문장으로 주면 맞는 색인을 제안한다.

작동 원리 — 계좌 이체가 '반쪽만' 되면? (ACID) 심화

A가 B에게 만 원 이체 = ① A에서 빼기 ② B에 넣기. ①하고 ②직전 정전이 나면 만 원이 증발한다. DB는 이를 막으려 여러 동작을 하나의 묶음(트랜잭션)으로 처리 — 원칙은 "전부 되거나, 전부 안 되거나." 이걸 포함한 네 약속이 ACID(1983년 명명).

글자뜻
A 원자성반쪽 실행 없음. 전부 아니면 전부 취소
C 일관성규칙에 어긋난 상태로 안 끝남
I 고립성동시에 여럿이 만져도 안 꼬임
D 지속성됐다고 하면 정전에도 남음
비전공자 감: 돈·주문·예약처럼 "틀리면 큰일"인 데이터는 이 성질이 강한 SQL DB가 안전. 마지막 티켓 한 장을 둘이 동시에 누를 수 있는 곳은 AI에게 "동시에 두 명이 마지막 하나를 요청하는 상황도 테스트하고 안전하게 처리해줘"라고 요구. 단, ACID는 백업이 아니다 — 실수로 지운 건 안 돌려주니 백업은 따로. (출처: Wikipedia "ACID"; Härder & Reuter, 1983)

설계 원칙 — 같은 걸 두 번 적지 않기 ('정규화') 심화

단골 주소가 바뀌었는데 주문 100건에 주소를 매번 적어 뒀다면 100군데를 다 고쳐야 하고, 하나라도 놓치면 데이터가 안 맞는다. 그래서 "같은 정보는 한 곳에만 적고 나머지는 가리키게" — 손님 정보는 회원 표에 한 번, 주문 표엔 '몇 번 회원'만. 이 정리정돈이 정규화(normalization), 코드가 꿈꿨던 "중복 없이"가 이것. 과제에서 데이터 모양을 그릴 때 "이 정보, 여기저기 똑같이 적히고 있진 않나?" 한 번만 의심하면 된다. (출처: Wikipedia "Database normalization")

마이그레이션(migration): 서비스가 자라면 표 구조 자체를 바꿔야 할 때가 온다(칸 추가·표 분리). 이미 데이터가 든 DB 구조를 안전하게 바꾸는 작업 = "DB 구조 변경을 Git처럼 이력으로 관리"(되돌리기도 됨). 단, 칸을 삭제하는 변경은 그 안의 데이터가 영구히 사라질 수 있다 → 구조 바꾸기 전엔 반드시 백업. (출처: Prisma, "What are database migrations")

그래서 나는 뭘 쓰나 — 한 줄 기준: 표로 잘 정리되고 관계가 분명하면(사용자·주문·상품) → SQL(관계형) · 형태가 자주 바뀌고 아주 대규모면 → NoSQL. 비전공자 첫 서비스 기본값은 SQL, 그중 PostgreSQL이 무난 — 자료가 가장 많고 필요하면 유연한 형식도 지원해 나중에 곤란할 일이 적다. (출처: TheLinuxCode·DEV, "SQL vs NoSQL 2026")

5·4 · 회원가입과 로그인 — 사용자를 다루기 00:14

서비스에 사람이 로그인하기 시작하면 다룰 게 훅 는다 — 회원을 어떻게 저장하고, 로그인 상태를 어떻게 유지하며, 빠르고 안전하게 하느냐. DB 강의에서 이걸 짚는 건 결국 전부 데이터를 다루는 일이라서다. 원리(인증·인가·쿠키/세션)는 2-2강에서 봤으니, 여기선 데이터·저장 관점으로 잇는다.

회원가입·로그인을 '데이터'로 보면

로그인·회원가입은 화면 기능 같지만 뿌리는 데이터다. 먼저 회원(users) 표 하나가 필요하다.

회원(users) 표예시 한 줄

회원가입 = 회원 표에 행 하나 넣기(Create). 로그인 = 그 이메일로 행을 찾아(Read) 비밀번호 대조 → 맞으면 통과, 틀리면 거절. 결국 앞서 본 CRUD의 조합이다.

로그인 상태는 어떻게 유지되나 — 세션과 토큰(JWT)

한 번 로그인하면 페이지를 넘겨도 왜 계속 로그인 상태일까? HTTP는 '무상태'라 매 요청이 남남 → 매번 '나 아까 그 사람'을 다시 증명해야 한다. 그 증명을 실어 나르는 게 쿠키(사이트가 브라우저에 건네는 작은 메모, 요청마다 자동 동봉 = '지갑 속 카드'). 이 쿠키에 무엇을 담느냐에서 두 갈래가 갈린다.

갈래 1 — 세션 (서버가 기억)

서버가 로그인 상태를 자기 '세션 저장소'에 적고, 브라우저엔 그걸 가리키는 번호표(세션 ID)만 쿠키로 줌. 서버가 상태를 쥐어 로그아웃·강제 차단을 즉시(저장소에서 지우면 번호표는 휴지 조각). 단점 — 서버가 여러 대면 저장소를 함께 나눠 써야 함. (MDN)

갈래 2 — 토큰 (JWT, 서버가 기억 안 함)

로그인하면 사용자 정보·만료 시각을 담아 위조 못 하게 서명(도장)한 증명서(JWT)를 통째로 줌. 이후 요청마다 토큰을 내밀고 서버는 저장소를 안 뒤지고 서명만 확인(그래서 '무상태'). 서버 확장·여러 서비스 연동에 강함. (Auth0)

JWT 꼭 알 것: 토큰 내용은 암호가 아니라 그냥 인코딩 → 가진 사람은 누구나 열어 본다. 서명이 막는 건 '못 바꾸게'이지 '못 보게'가 아니다 → 민감정보(비밀번호 등)를 토큰에 넣으면 안 됨. 또 한 번 내준 토큰은 즉시 무효화가 어렵다(지울 기록이 서버에 없음) → 보통 만료를 짧게 두고 '리프레시 토큰'으로 조용히 갱신.
그래서 뭘 쓰나: 단순한 웹앱 + 즉시 로그아웃·차단이 중요 → 세션이 더 단순·안전 · 여러 서비스·모바일·큰 확장 → 토큰이 유리. 가장 중요한 실용 결론 — 이걸 밑바닥부터 직접 만들지 말 것. MDN조차 "세션 관리를 직접 구현하지 말고 검증된 라이브러리를 쓰라"고 못 박는다. Auth 서비스(Supabase Auth·Clerk·Auth0)가 발급·만료·갱신·저장까지 안전하게 처리. 우리 몫은 "무슨 방식인지 이해하고, 믿을 만한 도구에 맡기는" 판단. (출처: MDN, "Session management")

캐시와 Redis — 로그인이 빠른 비결

캐시(cache) = 자주 쓰는 데이터를 메모리에 잠깐 두어 빠르게 꺼내 쓰기. 매번 창고(디스크·DB)까지 가는 대신 자주 찾는 물건을 계산대 옆 바구니에 올려 두는 셈. 속도가 크게 오르는 대신 원본과 잠깐 어긋날 수 있어 '언제까지 신선한 걸로 볼지(수명, TTL)'를 정해 둔다.

로그인에서 캐시가 등장하는 자리가 바로 위의 세션 저장소다. 세션 확인은 거의 모든 요청마다 일어나 '빠름'이 생명 → 앞서 본 Redis(키-값·메모리 기반)가 딱. 조회가 매우 빠르고 키마다 자동 만료(TTL)를 걸어 '30분 활동 없으면 자동 로그아웃'을 손쉽게 만든다. 그래서 "세션을 여러 서버가 나눠 쓰는 문제"와 "캐시로 속도 얻기"가 같은 도구 Redis 하나로 만난다. (출처: MDN "HTTP caching"; Redis 공식)

🔒 보안 메모: 브라우저에서 로그인 증표를 어디에 두느냐도 중요. localStorage는 악성 스크립트(XSS)에 그대로 읽혀 위험 → 보안 표준(OWASP)은 자바스크립트가 못 읽는 httpOnly 쿠키를 권한다. 비전공자는 세 줄만 — ① 저장·갱신은 Auth 서비스에 맡기고 ② 민감정보는 토큰에 넣지 말고 ③ 저장 위치는 안전한 곳으로. (출처: OWASP HTML5 Security Cheat Sheet)

5·5 · 데이터를 안전하게 — 보안과 백업 00:14

데이터베이스에 진짜 정보가 쌓이기 시작하면 두 가지 위험이 따라온다 — 누가 훔쳐 가는 것(보안)과 실수로 날아가는 것(백업). 코드는 AI가 쓰지만, 이 위험을 아는 건 내 몫이다.

보안 이야기 — "Little Bobby Tables"(SQL 주입)

프로그래머 사이 전설의 만화(xkcd, 2007). 학교에서 엄마에게 전화가 온다 — "아드님 이름을 DB에 넣었더니 학생 기록이 전부 지워졌어요." 아들 이름은 Robert'); DROP TABLE Students;--. 이름 안에 든 DB 명령어처럼 생긴 문자를, 프로그램이 이름이 아니라 명령으로 실행해 Students 표를 통째로 삭제(DROP)한 것.

이게 그 유명한 SQL 주입(SQL Injection) — 25년 넘게 보안 위협 1위권을 지키는 공격이다. 로그인 창에 특정 문자를 넣어 비밀번호 없이 로그인하는 것도 같은 원리. (출처: xkcd "Exploits of a Mom" https://xkcd.com/327/ · Malwarebytes "What is SQL injection" · 언어별 방어법 https://bobby-tables.com/)

방어법 — "레시피와 재료를 섞지 않는다": 핵심은 하나, 명령(레시피)과 사용자 입력(재료)을 안 섞기.
· 위험 — 사용자가 친 글자를 명령 문장에 그대로 이어 붙임 → DB가 "어디까지 명령이고 어디부터 데이터"인지 헷갈림.
· 안전 — 자리표(?)를 비워 둔 명령을 만들고 입력은 따로 건넴 → DB가 입력을 무조건 '그냥 데이터'로만 취급. 이게 파라미터화 쿼리(prepared statement), 보안 전문가들의 표준 방어법. (출처: OWASP "SQL Injection Prevention Cheat Sheet")

AI에게 — "DB에 넣는 값은 문자열로 이어 붙이지 말고 파라미터화 쿼리로 처리해줘. 사용자 입력을 그대로 신뢰하지 마." 한 문장이 학생 기록을 지킨다.

실화 — 기본을 안 지켜 뚫린 사고: 2015년 영국 통신사 TalkTalk이 SQL 주입으로 뚫렸는데 공격자는 십대들이었고, 쓰인 구멍은 3년 반 전에 이미 패치가 나와 있던 것을 방치한 것. 고객 약 15.7만 명 정보 유출(1.5만여 명은 계좌까지), 사고 비용 약 4,200만 파운드, 규제기관 총평 — "가장 기본적인 보안 조치조차 하지 않았다." · 규모로는 2008년 미국 Heartland가 같은 공격으로 카드 약 1.3억 장 유출(당시 사상 최대). (출처: 영국 ICO 발표; BBC "TalkTalk hack costs reach £42m" 2016; Wikipedia "Albert Gonzalez")

개인정보(PII) 다루기 — 모을수록 무거워진다

이름·이메일·비밀번호 같은 개인정보가 쌓이면 자산이자 책임이다. 새면 사용자가 다친다. 비전공자가 지킬 기본 네 가지:

원칙내용
최소 수집목적에 필요한 최소한만. "이메일만 되는데 전화·생년월일까지 받고 있진 않나?" (개인정보 보호법 제3·16조)
비밀번호는 해시절대 원문 저장 금지. 그것도 빠른 해시 말고 bcrypt·Argon2 같은 비밀번호 전용. (OWASP Password Storage)
민감정보는 위탁주민번호·카드번호는 직접 저장 말고 결제대행사(PG)·인증 서비스에 맡김
목적 제한모을 때 밝힌 목적에만 사용. 다른 데 쓰거나 넘기려면 별도 동의

데이터를 잃지 않기 — 백업, 그리고 유명한 참사들

해커가 없어도 데이터는 실수 한 번으로 사라진다. 그래서 백업(사본을 따로 보관)이 필요하다.

5·6 · 무료로 쓸 수 있는 DB 고르기 00:14

DB 전체가 파일 하나. 서버를 안 켜도 앱이 그 파일을 직접 읽고 씀. 작은 앱·프로토타입·모바일이 즐겨 씀(파이썬 등에 기본 내장 = 사실상 설치 없이).

회사들이 실제로 쓰는 오픈소스 DB를 내 컴퓨터에 무료 설치. 클라우드에서 빌리던 것과 같은 걸 내 손안에서.

Docker — 명령 한 줄로 띄우기: 설치가 번거로우면 DB를 '컨테이너'로 띄운다. 내 컴퓨터를 어지럽히지 않고 명령 한 줄로 켜고, 필요 없으면 통째로 지움.
로컬 vs 클라우드, 언제 뭘? 로컬 = 완전 무료·오프라인·내 데이터가 내 손안 → 개발·학습·실험 최고(단, 남에게 열려면 배포 필요, 내 PC 꺼지면 멈춤). 클라우드 = 24시간·안정적 공개 → 실제 서비스. 흔한 흐름 — 내 컴퓨터에서 만들고 → 세상에 낼 때 클라우드로. AI에게 — "개발용은 가벼운 로컬 SQLite로 시작하고, 배포할 때 클라우드 DB로 바꿀 수 있게 해줘."
무료 티어, 시작 전 확인 5: ① 용량 한도(예: 500MB — 이미지 많으면 금방 참) ② 미사용 시 일시정지(Supabase 1주) ③ 백업 방법(무료면 직접 백업 필요한지) ④ 정책 변경(무료가 사라질 수 있음) ⑤ 트래픽 늘면 유료(요금·용량 알림 켜 두기).
🤖 DB 붙이는 프롬프트: 문법 몰라도 무엇을 저장할지 + 조건만 주면 스키마 설계~연결~보안까지 도와준다. ① 저장할 것[상품·주문]의 표·열 스키마 제안 ② 무료 SQL(Supabase/Postgres) 기준 만드는 법 ③ 접속키는 코드 말고 환경변수로 ④ 값 넣을 땐 파라미터화 쿼리, 비밀번호는 해시로. (③④는 지난 시간 환경변수·보안 + 오늘 SQL 주입 방어와 이어짐)

5·7 · 실습 — 에이전트로 '내 Todo 앱' 만들기 00:35

배운 걸 모아 직접 만든다. 내 컴퓨터에서 도는 작은 Todo 앱 — 할 일을 추가·확인·완료·삭제하고, 데이터가 무료 DB에 저장돼 껐다 켜도 남는다. Todo는 저장·조회·수정·삭제(CRUD)의 교과서라 DB 익히기에 딱. 방식은 에이전틱 — 목표(PRD)를 주면 에이전트가 계획·설치·구현, 우리는 검토·판단·수정.

길 A — 내 컴퓨터(로컬)

무료 로컬 DB(예: SQLite)에 저장. 인터넷 없이 내 PC에서 완결.

길 B — 클라우드(Supabase)

같은 앱을 클라우드 DB로. 남에게 링크로 보여줄 때.

1부 — PRD로 시작하고 DB 구조 설계받기

먼저 PRD(만들 것 한 장)부터:

[무엇] 내 컴퓨터에서 도는 Todo 앱
[기본 기능(CRUD)] 추가 / 목록 / 완료 표시 / 삭제
[저장] 내 컴퓨터의 무료 DB(껐다 켜도 남기)
[내가 넣고 싶은 기능] ← 1~2개 직접(마감일·태그·오늘만 보기·검색)
[규칙] 무료 도구만 · 비밀값은 .env로 (로그인은 검증된 라이브러리로)

이 PRD를 에이전트에게 주고 DB 구조부터 설계받는다 — "이 PRD로 만들 Todo 앱의 DB 구조(표·열·관계)를 먼저 구조도로 그려 줘. 커스텀 기능(마감일·태그)도 반영. 로컬 무료 DB(SQLite)를 쓸 거야."

확인 포인트: AI가 표(예: todos, 태그를 넣었다면 tags와 연결표 todo_tags)와 열을 구조도로 보여 준다. 할 일 ↔ 태그는 다대다(N:M)라 연결표로 이어짐 — 5·3에서 배운 '표 사이 관계'대로 잡혔는지 눈으로 확인.

2부 — 길 A: 내 컴퓨터에 만들어 돌리기(로컬)

구조에 합의했으면 에이전트에게 실제 구현을 맡긴다 — "이 PRD와 합의한 DB 구조로 Todo 앱을 만들어 줘. 데이터는 로컬 SQLite 파일 하나(todo.db)에 저장, 브라우저에서 localhost로 열리게. .env·todo.db는 .gitignore에. 먼저 전체 계획을 보여주고 각 단계 끝에 뭘 확인하면 되는지 알려 줘."

우리가 할 일 셋: ① 계획 한 번 읽기(PRD와 맞는지) ② 위험한 명령은 승인(이 프로젝트 폴더 안인지 확인) ③ 잘 되던 순간마다 Git 커밋(되돌릴 지점).
통과 기준: 할 일을 추가하면 목록에 뜨고 새로고침·앱 재시작 후에도 남는다. DB 뷰어('DB Browser for SQLite' 등)로 todos 표에 행이 실제로 쌓였는지 확인.
자주 막히는 증상확인 / 해결
better-sqlite3 빌드 에러(node-gyp)네이티브 모듈이라 빌드 도구 필요. 에러를 그대로 에이전트에 주고 "내 OS 기준 빌드 도구 설치·확인"(맥: Xcode CLT / 윈도우: 빌드 툴·Python). Node는 LTS
포트 already in use(EADDRINUSE)이전 서버가 떠 있음 → 그 창을 끄거나 다른 포트로
추가했는데 새로고침하면 사라짐화면에만 있고 DB엔 안 들어감 → "추가한 할 일을 DB에 저장하고 다시 읽어오게 해 줘"
코드 고쳤는데 반영 안 됨개발 서버를 껐다 켜기(특히 .env 수정 후)
todo.db를 GitHub에 올려버림.gitignore에 추가. 이미 커밋했다면 "git 이력에서도 빼 줘"까지

안 되면 — 에러를 그대로 에이전트에 붙여넣어 고치게(관찰→수정 반복). 계속 못 고치면 마지막으로 되던 지점으로 되돌리고 문제를 잘게 쪼개 다시 시킨다.

3부 — 길 B: 클라우드(Supabase)로도

같은 앱을 클라우드 DB로도 만들어 두면 나중에 링크로 공유 가능(로컬은 내 PC에서만 열림) — "같은 Todo 앱을 로컬 SQLite 대신 Supabase(무료 클라우드 Postgres)에 저장하도록 바꿔 줘. 접속 키는 .env로, RLS(행 수준 보안)를 켜서 허용된 접근만 되게."

확인: Supabase 대시보드 Table Editor에서 todos 표에 행이 쌓이는지 확인. 그리고 RLS 꼭 켜기 — 안 켜면 공개 키로 누구나 데이터에 접근할 수 있다. (흐름: 로컬에서 익히고 → 배포 땐 연결만 클라우드로)

2-4강 · AI 에이전트 도구(8강 · 하네스 엔지니어링)

범용 AI를 '내 서비스 전용 조수'로 바꾸는 법. 핵심 공식은 에이전트 = 모델 + 하네스, 그 하네스를 강하게 만드는 세 무기가 MCP·스킬·규칙. (출처: 모두의연구소 「바이브 코딩으로 웹 만들기 2」 course 2130 · Anthropic "Building effective agents")

6·1 · 어시스턴트에서 에이전트로 — 도구의 정체 00:10

우리가 쓰던 그 도구, 사실은 '에이전트'였다. 한 문장으로: 목표를 주면 필요한 도구를 스스로 골라 쓰고, 결과를 보며 다음 수를 정하기를 끝까지 반복하는 AI.

우리는 "1번 클릭, 2번 클릭"이 아니라 "로그인 기능 붙여 줘"처럼 목표만 준다. 세부 단계는 AI가 짠다. 게다가 말로만 답하지 않고 파일을 고치고·명령을 실행하고·검색하며 실제로 움직인다. 그리고 한 번에 끝내는 대신 해보고 → 결과 확인 → 틀리면 고치기를 목표가 될 때까지 되풀이한다. 코딩뿐 아니라 고객 문의 응대·자료 조사 보고서 작성도 뼈대는 같다.

어시스턴트 vs 에이전트 — 칼로 자르는 게 아니라 '정도의 차이'

어시스턴트(assistant)에이전트(agent)
하는 일다음 줄 추천, 질문에 답여러 파일 수정·명령 실행·반복
단계를 정하는 주체사람AI가 스스로
도구 사용거의 없음(텍스트 위주)파일·터미널·검색 등
비유곁에서 거드는 조수일을 맡아 끝내는 일꾼

'agent'는 라틴어 '행하다(agere)'에서 온 오래된 말("남 대신 행동하는 이"). AI가 지어낸 신조어가 아니다. 러셀·노빅 『AI: 현대적 접근』(1995)은 인공지능을 아예 "스스로 행동하는 에이전트 연구"로 정의했고, 1990년대 비서 프로그램도 '에이전트'라 불렀다. 달라진 건 그 '머리'를 이제 LLM이 맡는다는 것뿐.

🔑 핵심 공식: 에이전트 = 모델(Model) + 하네스(Harness). 모델=언어·코드·이미지를 다루는 지능(Claude·GPT). 하네스=그 지능을 감싸 실제로 일하게 하는 실행환경(읽고→고치고→실행→반복 루프·파일 접근·안전장치). 같은 모델도 하네스에 따라 결과가 크게 달라진다. 이 노드가 강하게 만드는 대상이 바로 하네스(MCP·스킬·규칙).

루프 = 에이전트의 심장 (ReAct)

챗봇과 에이전트의 결정적 차이는 반복. 네 단계를 목표가 이뤄질 때까지 돈다 — 계획(Plan) → 실행(Act) → 관찰(Observe·에러? 됐나?) → 반성(Reflect) → 다시 계획. 프로그래밍의 while문처럼. 이 '생각-행동-관찰'을 연구자들은 ReAct(Reasoning+Acting)라 부른다. 품질관리 PDCA·공군 OODA 루프와 같은 모양 — 오래된 지혜를 기계 속도로 돌리는 것.

🤔 왜 '한 번에 정답'보다 나은가: "로그인이 안 된다" → 에이전트는 바로 코드를 고치지 않고 먼저 오류 로그를 읽어(Action) 401(인증 실패)인지 설정값 누락인지 관찰(Observation)한 뒤에만 수정법을 고른다. 추측을 검증된 증거로 교체하는 것. 그래서 좋은 지시는 "고쳐 줘"가 아니라 "관련 파일·오류 먼저 확인하고, 수정 뒤 테스트 돌려 결과 보고해 줘"처럼 관찰 지점을 요구한다.
⚠️ 2023 봄 AutoGPT 소동의 교훈: GPT-4 직후 "목표 한 줄이면 알아서 끝까지" 프로그램(AutoGPT·BabyAGI)이 GitHub 별 10만↑ 최고속으로 화제. 그러나 언제 멈출지 몰라 무한 반복·요금 폭탄·틀린 목표로 폭주. → 오늘의 에이전트는 완전 자율보다 작은 작업 단위·명확한 권한·중간 확인을 더 중시. 멋진 자율성보다 내가 멈추고 검토할 수 있는 흐름이 실전에서 안전하다.
오늘의 핵심 한 문장: AI 에이전트는 모델 + 하네스. MCP·CLI로 내 데이터·도구에 연결하고, 규칙 파일(AGENTS.md·CLAUDE.md)로 프로젝트 안내를 상시 걸고, 스킬로 반복 작업을 외워 두면 — 범용 AI가 '내 서비스 전용 조수'로 바뀐다. 도구는 바뀌어도 이 개념은 남는다.

6·2 · MCP — AI를 내 데이터에 꽂는 'USB-C' 00:12

하네스를 강하게 만드는 첫 번째 무기. 문제는 AI가 '내 것'(내 DB·파일·캘린더·회사 문서)을 모른다는 것.

복사·붙여넣기로 때울 순 있지만, 붙일 도구가 많아지면 "AI 3개 × 도구 5개 = 연결 15개"를 일일이 만드는 지옥이 된다(=M×N 문제).

🔌 MCP = "AI를 위한 USB-C 포트": MCP(Model Context Protocol)는 이 문제를 푸는 표준 규격(Anthropic 2024년 말 공개 → 업계 표준). USB-C가 기기·충전기를 통일했듯, 도구를 한 번 MCP 규격으로 만들면 MCP 지원 AI 아무 데나 꽂힌다. 15개 연결이 M+N=8개로 준다.

무엇을 할 수 있나: AI가 내 DB를 직접 조회해 답하기 · Figma 디자인을 읽어 그대로 화면 만들기 · 캘린더·Notion·Slack에 접근해 개인 비서처럼 굴기.

어쩌다 '표준'이 됐나 — 경쟁사가 같이 떠받침

Anthropic 2024·11 공개 → OpenAI(2025·3)·구글(2025·4)이 잇따라 채택 → 2025 말 중립 재단(Linux Foundation 산하 Agentic AI Foundation)으로 이관. 경쟁사들이 같은 규격을 함께 떠받치는 건 드문 일. 즉 MCP는 특정 회사 기능이 아니라 업계 공용 약속 — 한 번 익히면 도구가 바뀌어도 계속 쓴다.

공개적으로 유명한 서버는 아직 손에 꼽지만, 가장 값진 건 회사 내부용 MCP(사내 DB·문서 연결). 공개 안 해도 내 데이터에 AI를 붙이는 것만으로 큰 힘. (레지스트리: registry.modelcontextprotocol.io)

MCP가 건네는 3가지 — 이름만 알아 두기

종류뜻예
도구(Tools)AI가 실행하는 기능검색하기, DB 조회하기
자원(Resources)AI가 읽어들이는 데이터파일, 문서, 기록
프롬프트(Prompts)미리 만든 요청 템플릿재사용 요청

이게 좋은 이유: ① 연결을 매번 새로 안 만들어도 됨 ② AI가 실제 최신 데이터를 보니 환각↓ ③ 학습 시점 이후 최신 정보도 사용. 구조: host(AI 담은 앱)→client(연결 관리)→server(Notion·DB 등 기능 제공). M×N→M+N.

⚠️ 딱 하나 조심: USB-C 비유의 진짜 교훈 — 플러그 모양이 같다고 권한까지 같은 건 아니다. MCP는 연결 규칙이지 자동 안전 보장이 아니다. 공개 서버 수천 개 중 검증 안 된 것 다수 → 공식 서버부터. 붙이는 순간 AI에 내 데이터·계정 접근권을 준다. tool은 실제 행동(삭제·배포까지) 가능하니 '읽기'부터 연결하고 쓰기·삭제·배포는 승인 게이트. 최신 데이터를 읽어도 해석이 자동으로 맞는 건 아니므로 중요한 결과는 원본과 대조.

6·3 · 스킬(Skill) — 반복 작업을 AI에게 '외워' 두기 00:12

하네스를 강하게 만드는 두 번째 무기. 문제는 매번 같은 설명을 다시 하는 것(="우리 브랜드 톤으로", "이 형식으로 보고서") → 지겹고 결과도 그때그때 다름. 이게 재프롬프트(re-prompting) 문제.

스킬 = 재사용하는 '플레이북'. 자주 하는 작업을 한 번 적어 두고 계속 꺼내 쓰는 설명서. 기술적으로는 그냥 폴더 하나 — 안에 SKILL.md에 "무엇을, 언제 쓰는지"를 적어 두면 AI가 관련된 일을 만났을 때 알아서 꺼내 쓴다.

💡 켜 둬도 거의 공짜 — 점진 공개(progressive disclosure): AI는 평소 스킬의 이름 + 한 줄 설명(약 30~50 토큰)만 알고 있다가, 정말 필요할 때만 본문을 읽는다. 그래서 수십 개를 켜 둬도 안 느려진다. → description을 잘 쓰는 게 중요(AI가 언제 이 스킬을 꺼낼지 그 한 줄로 판단).

SKILL.md는 실제로 어떻게 생겼나 — 머리말 + 본문

description: 사용자용 안내 문구를 우리 서비스 말투(정중하고 짧게)로 다듬을 때 사용
---

4. 마지막에 도움받을 곳(고객센터 링크)을 붙인다.

남들이 이미 만들어 둔 스킬

가장 유명한 건 Anthropic의 '문서 스킬' 4개 — pptx·xlsx·docx·pdf. claude.ai에서 "이 자료로 발표 슬라이드 만들어 줘" 하면 뒤에서 이 스킬이 돌아 마크다운 흉내가 아니라 서식까지 갖춘 진짜 파일을 뽑는다. Claude 문서 생성이 통째로 이 스킬 위에서 돌고 소스도 공개(교본). 그 외 '스킬 만드는 스킬', 'MCP 빌더', 브랜드 가이드 스킬 등. 공식 저장소 github.com/anthropics/skills + 커뮤니티 수백 개.

⚠️ 조심: 스킬은 결국 'AI에게 주는 지시문 + 실행 코드'라 출처 불명을 함부로 켜면 위험. 내가 만들었거나 공식 저장소 것 위주로, 남의 스킬은 안에 뭐가 들었는지 열어 보고 쓴다.

규칙 파일 (AGENTS.md · CLAUDE.md) — "에이전트를 위한 README"

스킬과 비슷하지만 더 기본. "우리 프로젝트는 이런 규칙이야"를 파일 하나에 적어 둔다. 많은 도구가 대화 시작 때 자동으로 먼저 읽는다(단, Claude Code는 CLAUDE.md만 자동으로 읽음). 적으면 유용한 것: 프로젝트 개요 · 자주 쓰는 명령(설명보다 명령어 먼저) · 규칙·컨벤션 · 하지 말 것. 2025년 여러 회사가 모여 AGENTS.md 공통 표준을 만듦(수만 프로젝트 채택).

6·4 · 하네스 종류 비교 — 어떤 AI 코딩 도구를 고를까 00:16

도구 고르기 전 내 자리부터. 모델 이름만 보지 말고 네 가지 확인: ①파일 읽고 쓰기 ②터미널·브라우저 도구 ③MCP로 내 자료 연결 ④실행 전후 내가 승인·검토. 처음엔 되돌리기 쉬운 작은 수정으로 시험하는 게 최고의 비교법.

하네스 4부류 — '어디서 돌고, 내가 얼마나 지켜보나'

부류특징예
터미널/CLI형명령창에서. 깊고 강력, 조금 낯섦Claude Code, Codex CLI, Antigravity(agy), Aider, OpenCode
편집기(IDE)형VS Code 등 편집기 안. 눈으로 보며 승인/취소Cursor, Devin Desktop(구 Windsurf), Cline
브라우저형웹에서 클릭 몇 번, 배포까지. 입문 최저 문턱Lovable, Bolt, Replit Agent
클라우드/비동기형"이 이슈 고쳐 줘" 통째로 맡기면 백그라운드→초안Devin, Google Jules, OpenAI Codex(클라우드)

앞 셋=내 컴퓨터에서 지켜보며, 마지막=멀리 맡기고 결과만. Devin(클라우드)과 Devin Desktop(구 Windsurf 편집기)은 같은 회사(Cognition). Google Antigravity=Gemini CLI를 이어받은 통합 도구(2026-06 전환, 터미널은 agy).

🤷 "정답 하나는 없다" — 도구마다 성격이 뚜렷해 고르는 사람의 자리·취향을 탄다("코파일럿은 회사가, Claude Code·Cursor는 엔지니어가 직접 고른다"는 우스개). 고르는 기준: 터미널 vs 편집기 / 비용 / 확장(MCP·스킬) 필요? / 빠른 시제품 vs 어려운 문제. "다들 최고라는 것" 말고 내 조건으로.

초보 5가지 판단 기준

① 모델을 고를 수 있나(오픈소스 계열이 자유) ② 비용 구조(월정액 vs "프로그램 무료+쓴 만큼 API 요금" — 후자는 확 오를 수 있음) ③ 오픈소스 여부(접어도 남이 이어받아 살림 — 실제 2026년 한 오픈소스 확장 종료가 이 리스크를 보여 줌) ④ 터미널이 편한가 ⑤ 맡길까 지켜볼까(승인=편집기형 / 통째로=클라우드형). 실무자도 하나만 안 쓴다 — 어려운 건 터미널, 일상 편집은 편집기, 가끔 클라우드를 섞어 씀.

🕵️ 역사 — "세계 최초 AI SW 엔지니어" Devin 논쟁(2024·3): Cognition이 Devin을 "첫 AI 엔지니어"로 공개, 프리랜서 일 받는 시연이 화제. 그러나 홍보만큼 못 끝냈고 잘 되는 장면만 골라 담았다는 반론 + 독립 테스트 성공률 미달. 교훈: ①화려한 데모는 가려 볼 것("된다"≠"믿고 맡긴다") ②그럼에도 자율 코딩 경쟁에 불을 붙인 촉매. 과대광고와 진짜 발전은 늘 섞여 온다.
🔐 안전·권한: 에이전트는 파일 지우고 명령 실행하고 외부 요청까지 함. 좋은 도구는 위험 행동 전 "해도 될까요?"를 묻지만, 반복되면 사람이 무심코 "예"만 누름(사용자가 권한 요청의 약 93%를 그냥 승인). 권한 방식=매번 확인 / 편집만 자동 / 위험한 것만 막고 자동 / 아무것도 안 물음(=폐기 가능한 실험환경만). 안전 습관 3: ①중요 폴더에서 '자동 수락' 끄기 ②위험 작업 전 커밋으로 되돌릴 지점 ③"아무것도 안 물어봄" + "중요한 것 전체 접근"을 동시에 켜지 마라.

로컬 모델 · 열린/닫힌 모델 · 비용

로컬 모델(Ollama 등): 오픈 모델을 내 컴퓨터에서 직접. 좋은 점=사생활(입력이 기기 밖으로 안 나감)·무료·오프라인. 아쉬운 점=최신 클라우드보다 뒤처지고 고사양 GPU 필요. 결론=아직 클라우드가 무난, 단 "민감 자료는 로컬, 어려운 작업은 클라우드" 섞어 쓰기.

열린/닫힌 모델: Anthropic(Claude)=가중치 비공개(반드시 API). OpenAI=절충(최상위 GPT-5.x는 닫고 gpt-oss는 공개). 반전=가장 센 오픈웨이트는 중국(DeepSeek·Qwen·Kimi·GLM). 이 바닥은 빠르게 바뀐다 — 도구 이름·가격·회사는 선택 직전에 공식 공지·저장소 상태로 확인할 것(비교 글은 출발점, 최신성 보증 아님). 도구 이름 외우기보다 개념(에이전트=모델+하네스, MCP=연결, 스킬=반복 자동화)을 잡으면 새 도구도 바로 판단됨.

6·5 · 스킬 너머 — 확장과 요즘 흐름 00:18

Claude 플러그인 — 스킬·도구를 통째로 나누기

에이전트 팀을 통째로 얹기

여러 전문 에이전트+스킬+명령을 한꺼번에 얹어 Claude Code를 '지휘자'로. 큰 일이 오면 조사·구현·검증 담당에 나눠 병렬. oh-my-claudecode(팀 우선 오케스트레이션 레이어, 설치하면 수십 개 전문 에이전트가 딸려 옴). oh-my-openagent(omo)=모델 불가지론(Claude·GPT·Gemini를 한 인터페이스 뒤에 묶어 "잔일엔 값싼, 판단엔 똑똑한 모델"). 첫날 도구는 아니나, '하나에게 다 시키기→여러 전문 AI 지휘하기' 흐름의 끝.

명세 주도 개발(spec-driven) — '먼저 적고 시키기'

스킬·AGENTS.md의 공통점=시키기 전에 '무엇을·어떻게'를 먼저 글로. 이걸 프로젝트 전체로 키운 것. 배경: 설명 없이 "만들어 줘"만 하는 바이브 코딩은 빠르지만 AI가 의도에서 벗어나거나 없는 기능을 지어냄 → 요구사항·설계·할 일을 먼저 문서화하고 AI가 그 문서대로. 앞서 과제로 쓴 PRD가 바로 그 명세의 출발점. 도구: GitHub Spec Kit(별 12만↑, '원칙→명세→계획→할 일→구현'), AWS Kiro, 더 가벼운 OpenSpec. 결론=섞어 쓰기(가벼운 탐색·시제품은 대화로, 오래 유지할 중요한 건 명세로). 진짜 값어치는 '만들기 전에 한 번 더 생각하게' 하는 것.

루프 엔지니어링 · 'Ralph'(개발자 제프리 헌틀리): 같은 지시(스펙)를 계속 다시 주고, 한 번에 한 가지만 하고 커밋을 목표까지 반복. 진행은 대화 기억이 아니라 파일·git에 쌓음(컨텍스트 부패 회피). 2023 AutoGPT의 '무한 반복' 실패를 '멈출 기준(스펙)+파일에 쌓는 진행'으로 다시 푼 것(헌틀리는 이 방식으로 언어 하나를 ~$300로 제작). 이제 Claude Code에 /goal 명령까지. 단 안전장치 없으면 무한 반복·요금 폭발 → 멈출 기준·검토는 사람 몫.

🔀 요즘 흐름 — MCP에서 '스킬 + CLI'로: MCP는 편하나 공짜 아님 — 연결하면 도구 설명서를 미리 통째로 읽어 컨텍스트를 수만 토큰 잡아먹음. 그래서 AI가 원래 아는 CLI를 그냥 쓰게 하는 반작용(예: GitHub는 gh 사용). 한 벤치마크에서 같은 일에 CLI ~1천 토큰 vs MCP ~4만 토큰(800토큰 스킬이 2.8만 토큰 MCP 설명서보다 나음). 단 Claude Code는 요즘 툴 서치(필요할 때만 불러오기)로 이 부담을 이미 덜어 줌. 결론=셋 중 하나가 아니라 스킬·MCP·CLI를 함께 — 혼자 작은 건 'CLI+짧은 스킬'이 가볍고, 여럿이 쓰고 권한·기록 챙기면 MCP.

1단계 — 앱에 명령줄 사용법(CLI) 붙이기

AI가 다루기 가장 편한 창구는 화면이 아니라 명령줄(CLI)(명령을 쳐서 결과를 글로 받음). 앱을 새로 만들 필요 없이 기존 로직을 터미널에서 부를 작은 진입점만 얹고, 데이터는 쓰던 todo.db 그대로("같은 데이터를 여는 문을 하나 더"). AI에게: "기존 로직 재사용해 터미널 todo 명령(add·list·done·summary)을 만들어 줘. todo.db 그대로 쓰고 앱 코드는 갈아엎지 말고 얇은 진입점만." summary가 '오늘 완료'를 세려면 완료 시각이 데이터에 있어야 함(없으면 "완료 시각도 기록하게" 함께 부탁).

$ node todo.js list
[x] 1. 발표 자료 초안 작성하기
[ ] 4. 헬스장 PT 예약 변경 문의하기

사용법을 앱 루트의 규칙 파일에 5~10줄로. Cursor 등은 AGENTS.md를, Claude Code는 CLAUDE.md만 자동으로 읽음(CLAUDE.md 첫 줄에 @AGENTS.md로 불러와도 됨).

# Todo 앱
## 명령
- 목록: node todo.js list   · 추가: node todo.js add "<내용>"
- 완료: node todo.js done <번호>   · 오늘 완료 요약: node todo.js summary
## 하지 말 것
- todo.db(실데이터)를 직접 지우거나 덮어쓰지 말 것.

2단계 — AI에게 그 CLI를 실제로 돌리게 하기

"오늘 할 일 목록 보여 줘" → 규칙 파일을 읽은 AI가 스스로 node todo.js list를 실행해 출력을 가져옴(복붙 없이). 소스 코드가 아니라 실제 할 일 데이터를 보는 게 핵심. 권한을 관찰: 읽기(list·summary)는 승인, 한 번은 일부러 "todo.db 지워 줘" → 승인창 거부하고 '내가 문지기'라는 감. 시키기 전 git commit으로 되돌릴 지점.

3단계 — 반복 작업을 스킬로 외워 두기 (Claude Code 기준)

매번 똑같이 시키는 일("오늘 완료한 것만 커밋 메시지 형식으로")을 .claude/skills/todo-summary/SKILL.md에 저장. 핵심=스킬이 직접 데이터를 뒤지는 게 아니라 앞서 만든 CLI를 대신 불러 준다는 것. 다른 대화에서 "요약 스킬 써서 정리해 줘" 한마디면 AI가 알아서 꺼내 todo summary를 돌림 = 점진 공개가 실제로 동작(스킬 폴더 처음 만들면 도구 재시작).

덤 — 같은 걸 MCP로 붙이면? 파일시스템 MCP 서버를 등록하면 AI가 앱 폴더를 읽음. 단 이 서버는 파일(소스) 읽기엔 좋지만 데이터가 담긴 todo.db는 이진 파일이라 내용까지 세지 못함 — "오늘 완료 몇 건"은 결국 앱 명령(todo summary)을 돌려야 나옴. → 혼자 작은 앱이면 CLI+스킬이 가볍고, 여럿이 쓰고 권한·기록 챙기면 MCP. 도구가 아니라 상황이 고른다.
✅ 오늘의 결과물: ①앱에 CLI(list·summary 등) + 규칙 파일에 사용법 ②AI가 그 CLI를 직접 실행해 내 데이터 읽음(권한창 한 번 거부·위험 전 커밋) ③SKILL.md 1개(한마디로 발동, 스킬이 CLI 대신 호출) ④(덤) MCP로도 붙여 비교. 범용 AI가 내 앱을 아는 조수로 바뀐 지점은 어디였나?(규칙 파일·CLI 실행·스킬) 그 감각이 핵심.

7·2 · 범위 좁히기 — MVP와 '안 만들 것' 00:15

MVP = 완성품 축소판도, 화면만 그럴듯한 시제품도 아니다. 사용자가 의미 있는 한 가지 일을 처음부터 끝까지 해내는 의도적으로 작은 버전.

🛹 스케이트보드 vs 자동차 타이어: 목표가 "이동"이면 스케이트보드는 훌륭한 MVP(느려도 실제로 굴러감). 자동차 타이어 하나는 MVP 아님(아무 데도 못 감). 기능 수가 아니라 사용자 흐름의 완결성이 MVP의 기준.

역사 — 거대 앱도 시작은 초라했다

에어비앤비(거실 에어매트리스 3개+허름한 페이지, 예약·리뷰·결제 없음 → "낯선 집에 돈 내고 묵을까" 검증) · 드롭박스(제품 없이 3분 영상만 → 대기자 하룻밤 5천→7.5만) · 인스타그램(기능 많던 Burbn에서 사진 공유만 남겨 좁힘 → 2년 만에 10억달러). 하나같이 딱 하나의 핵심만, 초라하지만 끝까지 되게.

피벗 — 안 되는 건 빨리 접고 방향 틀기

슬랙(게임 Glitch 회사의 사내 채팅 도구가 진짜) · 트위터(팟캐스트 회사 Odeo → 140자 글) · 인스타(Burbn → 사진). 공통점 = "내가 부수적으로 만든 것"이 진짜 제품. "어? 이 부분이 진짜 쓸 만한데?" 싶으면 방향을 트는 용기를.

✂️ 자를 용기 — '안 만들 것' 목록이 비밀 무기: 테스트 = "이 기능 없이 출시해도 사용자가 핵심 문제를 푸나?" → 그렇다면(또는 애매하면) 자른다(버전2로). MoSCoW(꼭Must/있으면좋음Should/나중Could/지금안함Won't) 중 MVP는 Must만. 특히 Won't 최소 5개를 적어 두면 스코프 크립(기능이 슬금슬금 느는 것)의 유혹을 막는다.
⚠️ 스코프 크립 — 기능 더하다 가라앉은 배들: 하나하나는 다 그럴듯해서("고객이 원해서"·"경쟁사에 있어서"·"하루면 된다길래") 진보처럼 느껴진다. 경쟁사 급조 기능→버그투성이, "다 들어드릴게요"→뭐 하나 제대로 못 하는 도구, 윈도우 비스타·초기 아이튠즈. 성공한 제품은 "무엇을 만들었나"만큼 "무엇을 거절했나"로 정의된다.

한 문장 정의 + 범위 워크북

좋은 정의는 해결책이 아니라 문제에 초점: [누가]가 [어떤 상황/언제]에 [무엇을 못 해서] [어떤 불편]을 겪는다. (예: "동네 카페 단골이 재방문 때 종이 쿠폰을 자주 잃어버려 사장·손님 모두 적립 혜택을 놓친다.") 사용자를 '모두'로 두지 말고 가장 득 볼 한 사람으로. 문장에 "앱으로·자동으로·AI로" 수단이 들면 빼서 문제만.

📝 범위 워크북(오늘 좁혀 적기): 한 문장 설명 · 핵심 흐름 하나(예: 고르기→장바구니→주문) · 꼭(Must) 3개 · 지금은 안 함(Won't) 5개 · 완성 기준(done) 한 줄. 완성 기준·마감을 먼저 정하고 범위를 맞춘다(범위에 마감을 맞추면 끝없이 늘어남).
📚 출처: Designli · CB Insights "Startup Pivot" · Bolder Apps · Topflight · PMI · Atlassian

7·3 · 실전 기록 — 스펙부터 배포·관리자까지 00:15

에이전트와 대화로 신청 앱을 정원 규칙부터 배포·로그인·관리자까지 끌고 간 전체 그림. 네 걸음: 코드보다 스펙 먼저 → Supabase 직접 연결 → 정적 화면 무료 배포 → 관리자 얹기.

1) 스펙부터 — 코드보다 문서 먼저

바로 "만들어 줘" 하면 정원 초과 금지 같은 내 규칙이 슬쩍 빠진다. 그래서 스펙 먼저(스펙 주도). PRD가 '무엇을'이면 OpenSpec은 그걸 제안·설계·명세·작업목록으로 정형화하는 심화판(필수 아님).

npm install -g @fission-ai/openspec@latest
openspec init

opsx 워크플로(실험적): /opsx:explore(요구 정리) → /opsx:propose(change 문서 생성) → /opsx:apply(구현) → /opsx:sync(메인 명세 반영) → /opsx:archive(보관). 일을 change(변경 제안) 단위로 다뤄 '왜 바꿨는지'가 남는다. PRD엔 기술 선택 넣지 않되(무엇=사람), 스택은 AI가 고르고 근거를 설계 문서에.

2) Supabase를 에이전트에 직접 연결

Supabase = 표(DB)+회원가입·로그인이 함께 오는 백엔드. Claude Code면 MCP로 붙여 에이전트가 표·정책·함수를 직접 생성.

claude mcp add --scope project --transport http supabase "https://mcp.supabase.com/mcp?project_ref=<내 ref>"
🔒 여기서 핵심 두 가지가 실제로: ① 정원 초과 방지는 화면이 아니라 DB 안에서 — 브라우저에서 '세고 나서 넣기'는 동시 클릭에 뚫림. '현재 신청 수 < 정원' 조건의 단일 원자적 저장을 DB 함수로 만들어 무조건 거치게(마지막 한 자리에 10명 눌러도 딱 1명). ② RLS(행 수준 보안) — '본인 신청만 조회' 정책으로 로그인해도 남의 것 못 봄.

3) 무료 배포 + 관리자

정적 화면 → GitHub Pages(HTML/CSS/JS 한 벌은 서버 없이 정적 호스팅). 배포물엔 공개 키만(공개 저장소라 비밀 키 넣으면 그대로 노출). 관리자 페이지 = 권한을 DB에서 강제 — '화면에서 버튼 숨기기'는 보안 아님(누구든 요청 직접 전송). 관리자 명단을 DB에 두고 관리용 함수가 실행 때마다 '관리자냐'를 재확인. 화면은 패널만 보여줄 뿐, 진짜 문지기는 DB.

📚 출처: OpenSpec(github.com/Fission-AI/openspec) · Supabase MCP · Supabase RLS 문서

7·4 · 5단계 실습 — 원데이 클래스 신청 00:35

"정원 10명 클래스에 이름으로 신청, 남은 자리 표시, 차면 '마감'. 새로고침해도 남고, 동시에 눌러도 정원 안 넘김." 로그인·결제·후기는 안 만든다. 이 하나를 다섯 단계로.

단계하는 일통과하면
1. PRD만들 것을 한 장으로AI에게 줄 설계도
2. 작업대하네스 고르고 빈 프로젝트화면에 프로젝트가 뜬다
3. 화면눈에 보이게(목업)클래스·남은 자리가 진짜처럼
4. 저장+정원신청 저장·정원 지키기신청이 남고 정원 안 넘음
5. 한 바퀴+배포끝까지 돌리고 내보내기누구나 드는 신청 주소
🗝️ 핵심 태도: 명령을 '무엇을'로 준다("어떤 DB 써라"·"이 방식으로 막아라" 같은 how는 AI 몫). 우리는 의도 + 됐는지 확인에 집중. 한 번에 안 나와도 정상 — 지시 → 결과 확인 → 안 되면 다시 지시를 몇 바퀴.

1단계 PRD — mkdir class-signup → cd class-signup → claude(한 줄씩; &&는 터미널마다 달라 한 줄씩이 안전). /plan에 PRD 붙여넣고 스택·구조는 AI가 정하게. 누가 쓰는가(로그인 필요? 이름만?)도 여기서 — 이 앱은 이름만으로 완결이라 로그인은 다음 단계로(빠뜨림 아니라 순서). 확인: AI 계획에 정원 지키는 방법이 담겼나(빠지면 코드 나오기 전에 짚기=제일 쌈). 통과 시 git init/add/commit.

2단계 작업대 — 하네스 3형(브라우저형=아주 처음 / 편집기형=파일 여러 개 / 터미널형=설치·배포까지 AI가). 이 과정은 Claude Code(터미널형). "빈 프로젝트 세워 줘, 무료로, 실행법·규칙을 CLAUDE.md에". 확인: 빈 화면이라도 프로젝트가 한 번 떠야. 안 뜨면 오류를 그대로 붙여넣기.

3단계 화면(목업) — 저장 없이 예시 값으로 화면부터("제목·일시, 남은 자리 '7/10', 이름칸, 신청 버튼"). 눈에 보이면 방향 판단이 빠름. 한 걸음 더 — 디자인: 기본 화면은 '흔한 보라 그라데이션'의 AI 슬롭. Anthropic frontend-design 스킬(/plugin marketplace add anthropics/claude-code)로 주제 맞는 개성 있는 타이포·색·레이아웃.

🏔️ 4단계 저장+정원 = 오늘의 진짜 산: "신청하면 남은 자리 줄고 새로고침해도 남게, 차면 '마감', 여러 명이 동시에 마지막 한 자리를 눌러도 정원 절대 안 넘김." (how는 AI에.) 확인 넷: 즉시 반영 / 새로고침 유지 / 마감 / 동시 신청 테스트를 AI에게 만들게 해 딱 1건만 성공·정원≤10을 로그로(설명만 믿지 말고 눈으로). 지난 2-3강 트랜잭션·제약이 여기서 실전. 통과 시 즉시 커밋.

5단계 한 바퀴+배포 — 사용자가 되어 끝까지(신청→감소→마감→새로고침 유지) 끊김 없이. "무료 배포처 추천·순서, 재배포·재시작 후에도 데이터 남는 곳으로, 비밀값은 배포 설정에 넣고 코드엔 X". 확인: 다른 기기/시크릿 창에서 실제 신청되나, 재배포 후에도 남나(서버에 DB 얹는 무료 배포처는 재시작 시 사라지기도 → Supabase 같은 외부 DB면 걱정 없음). 배포까지 못 가도 3단계(화면)까지면 오늘 성공.

7·5 · 로그인 — Supabase 회원가입·로그인 00:15

'이름만' 다음의 탈것(자전거). '누구나 이름만 신청'→'로그인한 사람이 신청하고 자기 내역만 보는' 서비스로. 밑바닥부터면 비밀번호 저장·세션·보안이 무거워 → Supabase 같은 BaaS(백엔드 서비스)에 맡긴다(비밀번호 해시·세션·토큰은 Supabase가, 우리는 '무엇을'만). ※ 이 확장은 선택 — MVP가 굴러가면 그대로도 충분.

붙이는 순서

① supabase.com 무료 프로젝트(DB+인증 함께) → Project URL·공개 키(publishable/anon)를 .env에(코드에 박지 말 것). Claude Code면 공식 Supabase 플러그인/MCP로 AI가 표·인증·정책 생성. ② 이메일·비밀번호 회원가입·로그인 화면 연결(Supabase 기본 제공, 로그아웃 버튼도). ③ 신청을 로그인 사용자와 잇고 RLS로 자기 신청만 보고 취소(안 켜면 로그인한 아무나 남의 것 조회 → 반드시 켜기). 정원 초과 방지는 그대로.

⚠️ 여기서 막히는 함정 넷(+α):
· 가짜 이메일 거부 — test@example.com은 email_address_invalid. 실제 받을 이메일 쓰거나 개발 중 '이메일 확인' 끄기.
· 가입하자마자 로그인 실패 — 기본은 이메일 확인 요구(Email not confirmed), 무료 티어 확인 메일은 시간당 제한. 수업 중엔 대시보드 Authentication에서 'Confirm email' 잠시 끄기(배포 전 다시 켜기).
· 실제 발송엔 커스텀 SMTP — 기본 내장 메일은 팀 멤버 주소로 시간당 두어 통(over_email_send_rate_limit). Resend·SendGrid·SES·Postmark 등 SMTP를 대시보드에.
· 확인 메일 링크가 localhost로 — Authentication의 Site URL·Redirect URLs를 배포 주소로.
· RLS 안 켜면 남의 데이터가 다 보임 / 키 두 개 구분 — 공개 키(anon)는 화면에 써도 됨(그래도 .env), 비밀 키(service_role)는 절대 코드·git·채팅에 X(노출 시 RLS 우회로 DB 전체 열림 → 즉시 재발급).
💡 정리 — 무엇을 손에 익혔나: '클래스 신청 앱'이 아니라 다섯 단계 절차(PRD→작업대→화면→저장→한 바퀴+배포). 그 위에 로그인(Supabase)을 얹었을 뿐. 어떤 서비스를 만들든 이 절차는 그대로 통한다.
📚 출처: Supabase Docs "Password-based Auth"·"Row Level Security"

7·6 · 내 도메인으로 — 자유 제작 00:35

이제 정해진 예시가 아니라 내가 만들고 싶은 것. 스택·순서·어디까지 갈지 자유. 앞 절차는 필요할 때 꺼내 보는 참고서. 딱 하나만: 작게, 한 흐름이 끝까지 굴러가게.

🚀 시작해서 제출까지:
① 한 문장으로("[누가] [무엇을] 할 수 있는 [무엇]", 예: '동네 모임 주최자가 참가 신청을 받는 페이지'). 거창하지 않아도.
② AI와 자유롭게 — 의도→확인→막히면 다시 지시. 스택·방법은 AI에, 나는 '무엇을·됐는지'만.
③ 내 규칙 하나는 지키게(있다면 — 정원·선착순·중복금지·하루 한 번이 실제로 지켜지나 눈으로).
④ 막히면 다음 강의 네 규율.
⑤ 작동하는 지점에서 제출 — 완성 아닌 '굴러가는 첫 조각'. 실행 저장소/배포 주소(로컬만이면 실행법+캡처), 한 문장 정의, AI 대화(효과적 프롬프트 하나 표시), 회고 한 줄(무엇을 잘랐나/어디서 막혀 어떻게 풀었나/다음 한 걸음).

어디까지 가든 좋다 — 화면만 떠도, 저장까지 돼도, 배포까지 가도 각자 낸 만큼이 성과. 이게 모듈 최종 제출(Main Quest 01)의 착수분.

7·7 · 막혔을 때 — 네 가지 규율 00:12

"앱 전체 완성해줘" → 파일 20개 바뀌고 실행 안 됨. 더 긴 프롬프트로 밀면 문제 위치만 더 묻힌다. 작게 요청하고, 눈으로 확인하고, 되돌릴 길을 남기는 규율이 속도를 안전한 진척으로 만든다.

🧭 네 가지 규율:
① 한 번에 하나만 — 프롬프트 하나에 다섯 요구=몇 개는 틀려 디버깅 난망. 고칠 때도 "필요한 것만 바꿔줘". 버그 하나→확인→다음.
② 작동하면 저장(커밋) — 바뀔 때마다가 아니라 '작동할 때마다'. 깨지면 마지막 정상 지점으로.
③ 막히면 되돌리고 리셋 — 빙빙 돌면 랜덤 수정 금지(코드 이해불능의 지름길). 5~10번마다 "뭐가 됐고·뭐가 남고·지금 문제는 뭐다" 요약으로 AI 초점 재조정.
④ 이해 못 한 코드는 배포 X — 시제품은 "다 수락"도 OK, 남에게 내보낼 거면 그 코드가 뭘 하는지 알아야. AI 생성 코드의 약 45%가 보안 테스트 실패(Veracode) → "안전한지 봐 줘"는 필수.
🔧 Git 심화 — '저장 버튼'이 아니라 실험을 안전하게 하는 기록 장치: 세 장소 = working tree(펼친 원고) / staging(index)(git add=다음 기록에 포함할 선택) / repository(git commit=확정 체크포인트). 되돌리기 위험도 구분: git restore(작업 트리 미커밋 변경 버림) · git reset --hard(스테이징·작업트리까지 지움=강력·위험) · git revert(과거 안 지우고 반대 커밋 추가=공유 브랜치·푸시분 취소에 안전). reset=내 로컬 책상 정리, revert=공유 역사의 취소 메모. 좋은 체크포인트는 시간('오후 작업') 말고 행동·관찰 결과("새로고침해도 남는다"). AI에게도 "reset --hard·강제 푸시는 승인 전 실행 말라"고 규율로. 비밀키가 diff에 보이면 커밋 말고 .gitignore·환경변수 먼저(Git은 비밀 자동 차단 안 함). 원격에 올라간 키는 삭제로 안전해지지 않음 → 재발급.
🦆 러버덕 디버깅: 막히면 고무 오리(=AI)에게 "뭘 하려는지·무엇까지 됐고·어디서 막혔는지"를 처음부터 또박또박 글로. 말로 옮기는 순간 당연하다 여긴 전제의 빈틈이 드러난다(자기 설명 효과, 『실용주의 프로그래머』 1999). 설명≠관찰 구분("아마 DB 고장"=추측 / "콘솔에 Cannot read undefined"=관찰). 재현 절차+기대값을 적으면 AI에게 검증 가능한 질문이 된다. 도움 청할 땐 코드 전체 대신 목적 한 문장·재현 단계·기대vs실제·오류 전문/관련 조각.
🕳️ 둠 루프(doom loop) 주의: "거의 다 됐는데" 버그가 안 잡히고, AI는 "고쳤다"는데 안 고쳐지고, 프롬프트 넣을수록 멀쩡하던 것까지 깨짐 — 운이 아니라 구조적(대화 길어져 맥락 오염 + AI엔 의존 지도 없어 증상만 땜질). 탈출: 같은 프롬프트 반복 말고 "한 발 물러서 전체 그림 다시" / 마지막 정상 지점으로 리셋 후 새 대화로 시작 / "로그인 고쳐줘" 대신 "폼이 데이터 제대로 보내는지부터". 막힘이 30분 넘으면 더 파기 전에 멈추고 되돌리는 게 대개 더 빠름.
💣 초보 지뢰(증상→처방): 검증 없이 믿기→'초안'으로 대하고 "이 코드 뭘 하는지 설명해줘" · 비밀키 커밋→.env+즉시 .gitignore, 노출 시 삭제 아니라 재발급 · 예외·에러 처리 누락→"빈 입력·네트워크 실패·잘못된 값도 처리해줘" · 불필요한 복잡도→"이거 없이 기본 기능만으로 안 돼?". 혼자 막힐 때: 에러 그대로 복붙 검색(맨 마지막 줄이 잘 맞음) → 최소 재현 예시(MRE)로 좁혀 커뮤니티에 '잘' 묻기(목표·최소코드·정확한 에러·시도한 것).
