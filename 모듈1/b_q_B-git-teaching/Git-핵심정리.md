# Git & GitHub 핵심 정리 + 예상문제 (팀 공유용)

> 출처 ① 허남철 「초보자를 위한 git과 Github 시작하기」(실습) · ② 조해창(퍼실, 2026.07.09) 「Git이 뭔데, 설치하라는 건가요?」(개념·AI시대)
> **한 줄 프레임:** AI는 생성하고, Git은 고정한다. AI는 속도를, Git은 되돌아갈 길을 준다.

---

## 📌 30초 요약
- **Git** = 프로젝트가 **언제·누가·무엇을·왜** 바꿨는지 기록하고 **되돌릴 수 있게** 하는 버전관리 도구. 내 컴퓨터(로컬)에서 인터넷 없이도 동작.
- **GitHub** = Git 저장소를 **온라인에 보관 + 협업**(Issue·PR·리뷰·자동화)하게 해주는 서비스. GitHub는 여러 선택지 중 하나(GitLab·Bitbucket도 있음).
- **AI 시대에 Git이 더 중요해진 이유** = AI가 코드를 빠르게 대량으로 바꾸는 만큼, **무엇이 바뀌었고 문제 시 되돌릴 수 있는 안전장치**가 필수이기 때문.

## 1. 블로그 요약

### ① 실습편 (허남철)
Git 설치 확인부터 계정 등록 → 로컬 저장소 생성(`git init`) → 파일 변경 → `add`/`commit` → `log` 확인 → GitHub 원격 저장소 생성·연결(`remote add`) → 토큰 인증 → `push`까지, **초보자가 손으로 따라 하는 전체 흐름**. 마지막에 **branch(브랜치)** 개념과 활용 사례(기능개발·버그수정·실험)를 소개.

### ② 개념·AI시대편 (조해창, 퍼실 · 오늘자)
"왜 AI 쓰는데 Git을 배워야 하나?"에서 출발. `보고서_진짜최종.docx` 같은 **원시적 버전관리**의 한계 → Git의 탄생(리눅스·Linus·2005 BitKeeper 사건) → Git 내부(스냅샷·해시·blob/tree/commit) → **branch=포인터**, **merge/충돌=정직한 신호** → **Git ≠ GitHub** 구분 → AI Agent가 "Git 필요" vs "GitHub 권한 필요"의 차이 → **바이브코딩을 지속가능하게 만드는 Git 루프**로 마무리.

## 2. 핵심 개념 총정리

### Git vs GitHub
| 구분 | Git | GitHub |
| --- | --- | --- |
| 정체 | 버전관리 **도구** | Git 저장소 **호스팅·협업 서비스** |
| 위치 | 내 컴퓨터(로컬), 오프라인 OK | 온라인(원격) |
| 계정 | 없어도 됨 | 필요 |
| 대체재 | — | GitLab·Bitbucket·자체서버 |
| 핵심 기능 | 커밋·브랜치·이력 비교·동기화 | Issue·PR·Code Review·Actions |

### 버전관리란?
프로젝트가 시간에 따라 어떻게 바뀌었는지 **기록하고, 비교하고, 되돌리는** 일. `보고서_최종 → 진짜최종 → 진짜진짜최종`의 체계화된 형태. 코드는 파일이 수십~수백 개라 더 절실.

### 파일이 저장되는 3개의 공간
| 공간 | 설명 |
| --- | --- |
| Working Directory | 파일을 실제로 수정하는 작업 공간 |
| Staging Area (Index) | `git add`로 **커밋할 파일을 준비**하는 공간 |
| HEAD | 브랜치의 **최신 커밋을 가리키는 포인터** |

흐름: **작업(Working) → `add` → 준비(Staging) → `commit` → 확정(스냅샷)**

### 주요 명령어
| 명령어 | 의미 |
| --- | --- |
| `git init` | 폴더를 git으로 관리 시작 (`.git` 폴더 생성) |
| `git status` | 현재 변경 상태 확인 |
| `git add <file>` | 변경 파일을 스테이징에 올림 |
| `git commit -m "메시지"` | 특정 시점 스냅샷 확정 |
| `git log` | 커밋 이력(해시·작성자·메시지) 확인 |
| `git remote add origin <url>` | 원격 저장소 연결 (origin=별칭) |
| `git push origin main` | 로컬 커밋 → 원격으로 전송 |
| `git pull origin main` | 원격 변경 → 로컬로 가져와 병합 |
| `git clone <url>` | 원격 저장소를 통째로 복제 |
| `git diff` | 이전 상태와 현재의 차이 확인 |

### Branch(브랜치) & Merge(병합)
- **브랜치 = 특정 커밋을 가리키는 가벼운 이름표(포인터).** 프로젝트 통째 복사가 **아님**. "여기서부터 다른 가능성을 실험" 하는 표식.
- 활용: **새 기능**(예: 위시리스트) / **버그 수정**(예: 로그인-버그-수정) / **실험**(예: 새-디자인). 메인에 영향 없이 작업 후 마음에 들면 **merge**, 아니면 삭제.
- **merge** = 갈래 합치기. 서로 다른 부분이면 자동 병합, **같은 부분을 다르게** 고쳤으면 **충돌(conflict)**.
- **충돌은 Git 고장이 아니라 "여기는 사람이 판단하라"는 정직한 신호.**

### 로컬 / 원격 / remote
- **로컬 저장소** = 내 노트북 안. **원격 저장소** = 내 컴퓨터 밖(GitHub·GitLab·회사서버).
- **remote** = 내 로컬이 알고 있는 "다른 저장소의 주소". `push`는 로컬→원격, `pull`은 원격→로컬.

### AI 시대에 Git이 필요한 5가지 이유
1. **상태 파악** — `status`/`diff`/`log`로 무엇이 바뀌었는지 안정적으로 읽음
2. **안전장치** — 커밋 = 게임 세이브포인트. 큰 작업 전 커밋 = 실패 비용↓
3. **협업의 언어** — PR·리뷰·revert가 Git 기록 위에서 작동
4. **로컬 우선** — 네트워크 없이도 로그·브랜치·커밋·비교 가능
5. **책임 경계** — 어느 파일·어느 줄을 왜 바꿨는지 기록(좋은 커밋 메시지)

### 지속가능한 바이브코딩 루프
작업 전 커밋 → AI에 작은 변경 요청 → `git diff`로 확인 → 테스트 → OK면 커밋 / 아니면 되돌리기 → 다른 브랜치로 다른 접근.

## 3. 출제 유력 문제 + 해설  (⭐ = 유력)

**⭐ Q1. Git과 GitHub의 차이는?**
- **답:** Git=내 컴퓨터에서도 돌아가는 버전관리 도구 / GitHub=Git 저장소를 온라인에 보관·협업하게 해주는 서비스 중 하나.
- **해설:** Git은 GitHub 없이도, 인터넷·계정 없이도 쓸 수 있음. GitHub는 GitLab·Bitbucket처럼 여러 호스팅 서비스 중 가장 유명한 하나일 뿐. "GitHub=Git"이 아님이 핵심.

**⭐ Q2. 버전관리란 무엇이고 왜 필요한가?**
- **답:** 프로젝트가 시간에 따라 어떻게 바뀌었는지 기록하고 비교·되돌리는 것.
- **해설:** `보고서_진짜최종.docx`가 원시적 버전관리. 코드는 파일이 수십~수백 개라 "어느 변경 때문에 망가졌는지" 추적이 필수 → Git이 이 문제를 해결.

**⭐ Q3. add와 commit의 차이 / Staging Area란?**
- **답:** add=변경 파일을 스테이징에 올리는 준비 단계 / commit=그 순간을 스냅샷으로 확정.
- **해설:** Working Directory(작업) → `add` → Staging Area(준비) → `commit`(확정). 바로 커밋되는 게 아니라 **준비→확정** 2단계인 것이 포인트. HEAD는 최신 커밋을 가리키는 포인터.

**⭐ Q4. branch란? 프로젝트를 통째로 복사하는 것인가?**
- **답:** 아니다. 브랜치는 특정 커밋을 가리키는 **가벼운 이름표(포인터)**.
- **해설:** 통째 복사가 아니라 "여기서부터 다른 가능성을 실험하겠다"는 표식 하나를 만드는 것. main·feature/login 같은 이름표가 각각 다른 커밋을 가리킴.

**⭐ Q5. merge와 충돌(conflict)? 충돌은 Git이 고장난 것인가?**
- **답:** merge=서로 다른 갈래를 합치기 / 충돌=같은 부분을 다르게 고쳤을 때 발생. 고장이 아님.
- **해설:** 충돌은 "두 변화가 동시에 맞을 수 없으니 **사람이 판단하라**"는 Git의 정직한 신호. 무서워할 게 아니라 당연한 과정.

**⭐ Q6. AI 시대에 Git이 왜 더 중요한가? (하나 이상)**
- **답:** ①상태 파악 ②안전장치(커밋=세이브포인트) ③협업의 언어 ④로컬 우선 ⑤인간-AI 책임 경계.
- **해설:** AI가 코드를 빠르고 대량으로 바꾸므로, 무엇이 바뀌었는지 확인하고 문제 시 되돌릴 장치가 더 필요해짐. "AI는 생성, Git은 고정."

**Q7. push와 pull의 차이? origin은 무엇인가?**
- **답:** push=로컬 커밋을 원격으로 전송 / pull=원격 변경을 로컬로 가져와 병합. origin=원격 저장소의 별칭(alias).
- **해설:** origin은 긴 URL을 대신하는 짧은 이름. 다른 이름도 가능하지만 관례상 origin.

**Q8. git init을 하면 무슨 일이 일어나나?**
- **답:** 그 폴더를 git으로 관리 시작하고, 관리 정보를 담는 `.git` 숨김 폴더를 생성.
- **해설:** `ls -a`로 `.git` 생성을 확인. 이 폴더가 있어야 그 디렉토리가 Git 저장소가 됨.

**Q9. 로컬 저장소와 원격 저장소, remote의 뜻은?**
- **답:** 로컬=내 컴퓨터 안 저장소 / 원격=내 컴퓨터 밖(GitHub 등) / remote=로컬이 아는 다른 저장소의 주소.

**Q10. AI 도구가 "Git 필요"라 할 때와 "GitHub 권한 필요"라 할 때 차이는?**
- **답:** Git 필요=내 코드 변경 이력 추적·되돌리기(로컬·diff) / GitHub 권한=온라인 저장소·Issue·PR·리뷰·Actions까지 읽고쓰기.
- **해설:** 이 둘을 구분하는 게 이 글의 핵심 메시지 중 하나. 로컬 작업이면 Git, 협업 플랫폼 연동이면 GitHub.

**Q11. 좋은 커밋 메시지가 왜 중요하고, 좋은 예는?**
- **답:** 커밋 메시지는 "판단의 단위". "버그 수정"보다 "로그인 실패 시 토큰 갱신 루프를 막기 위해 만료 조건 추가"가 좋음.
- **해설:** 나중에 AI에게 다시 물을 때 강력한 맥락이 됨. AI가 짠 코드일수록 검토 가능한 기록이 더 필요.

**Q12. git log에 나오는 SHA-1 해시란?**
- **답:** 각 커밋을 고유하게 식별하는 40자리 16진수 문자열.
- **해설:** 내용이 조금만 바뀌어도 해시가 바뀜 → 데이터 변조·손상 감지, 특정 시점 복귀의 기준.

**Q13. Git은 누가 왜 만들었나?**
- **답:** 리눅스 커널 개발이 BitKeeper 무료 사용권을 잃자(2005) **Linus Torvalds**가 직접 제작.
- **해설:** 목표=속도·단순 설계·비선형(브랜치) 개발·완전 분산·대형 프로젝트 처리. "수많은 사람이 거대한 코드를 동시에 바꾸는 상황"을 견디려 태어남 = AI 시대와 닮음.

**Q14. 좋은 바이브코딩 흐름을 Git으로 설명하면?**
- **답:** 작업 전 커밋 → AI에 작은 변경 요청 → `git diff` 확인 → 테스트 → OK면 커밋 / 아니면 되돌리기 → 다른 브랜치 시도.
- **해설:** 이렇게 하면 AI가 "위험한 마법사"가 아니라 "빠른 조수"가 됨.

**Q15. (심화) Git 내부 객체 3가지는?**
- **답:** blob(파일 내용)·tree(디렉토리 구조)·commit(tree+부모 커밋+작성자+메시지).
- **해설:** Git은 파일을 **이름이 아니라 내용**으로 기억하고, 각 객체를 해시로 식별.
