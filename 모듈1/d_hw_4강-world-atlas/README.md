## 백엔드 (Supabase · RLS 보안)
- **저장·집계 기능** — 방문자 수 · 나라별 조회수(인기순위) · 댓글 · 가봤어요/가고싶어요 · 즐겨찾기
- **인가** — 익명 로그인 → `auth.uid()` 로 본인 것만 수정
- **프라이버시** — 방문 로그 원문은 SELECT 차단, 집계는 `SECURITY DEFINER` 함수로만 노출
- **비밀값** — 공개(publishable) 키만 프론트에, service_role 비밀키 미사용 · 데이터는 RLS가 보호

## 화면
- **한/영 토글(i18n)** — UI·40개국 데이터·명소 해설(ko/en 위키백과 자동 전환)
- **다크모드 토글** (선택 저장) · 위키백과 랜드마크 히어로 연동
- 목록 검색·지역 필터·즐겨찾기·인기순 정렬 · 상세페이지 **인구·면적 상대규모 시각화 + 국기 의미 해설**

## 📋 평가 기준 대조 (노드4 루브릭)

퍼실님이 항목마다 바로 확인하실 수 있게 정리했습니다.

### 1) 실제 동작하는 백엔드 · 프론트 ↔ 백엔드 연동
| 확인 | 방법 |
|---|---|
| 저장이 진짜 되나 | 상세페이지에서 ✈️가봤어요·💬댓글 → **새로고침·다른 기기에서도 그대로** |
| 집계가 되나 | 홈 상단 "👁 누적 방문·오늘", 카드 👁조회수, 🔥인기순 TOP5 |
| 연동 흐름 | 화면 클릭 → `supabase.from(...).insert/select` → DB 반영 → 재렌더 |

### 2) 보안 자가점검 — AI 코드 3대 약점
| 약점 | 대응 | 근거 위치 |
|---|---|---|
| ⓐ **입력 검증** | 프론트 제한 **+ DB CHECK 이중**(닉네임 20자·댓글 300자·`kind` 화이트리스트) + 출력 이스케이프(XSS 방지) | `SUPABASE_설정.sql` `check(...)`, `data.js` `esc()` |
| ⓑ **비밀값 관리** | publishable(공개)키만 프론트에, **service_role 비밀키 미사용** — 데이터는 RLS가 보호 | `data.js` `SB_KEY` |
| ⓒ **인증·인가** | 익명 로그인 → `auth.uid() = user_id` 로 **본인 것만** insert/delete | `SUPABASE_설정.sql` 정책 |
| ＋ 프라이버시 | 방문 로그 원문 직접 SELECT 차단, 집계는 `SECURITY DEFINER` 함수로만 노출 | `get_visit_stats` / `get_view_counts` |

### 🔴 직접 공격해서 막히는지 검증한 결과 (실측)
> "눈으로 될 것 같다"가 아니라, 실제로 공격을 넣어 **백엔드가 거부**하는 것을 확인했습니다.

| 공격 시도 | 결과 |
|---|---|
| 댓글 **301자** 삽입 | ❌ 거부 — DB CHECK 위반 (Postgres `23514`) |
| `kind='hack'` (화이트리스트 밖) 삽입 | ❌ 거부 — DB CHECK 위반 |
| **타인 user_id로 위조**한 댓글 삽입 | ❌ 거부 — **RLS 정책 위반** |
| `page_views` 원본 로그 **직접 조회** | ❌ 0행 — SELECT 정책 없음(집계만 공개) |

→ 프론트를 우회해도 **백엔드(DB)가 최종 방어**함을 실측으로 확인.

### 3) 2분 시연 시나리오
1. 홈 상단 **"👁 누적 방문"** 숫자 확인 → 백엔드 연결됨
2. 아무 나라 진입 → **✈️가봤어요 + 💬댓글** 작성
3. **새로고침** → 그대로 남아있음 (= 서버에 저장된 것)
4. 홈 복귀 → 그 나라 카드에 👁조회수·반응 반영, **🔥인기순** 상단 이동

### 아키텍처 한 줄
`브라우저(GitHub Pages · 정적)` → `@supabase-js` → `Supabase Postgres(RLS 정책 + 집계 RPC)`
프론트가 어디에 배포돼도 **DB 보안 규칙은 하나로 모든 접근에 동일 적용**됩니다.

### ℹ️ "키가 코드에 보이는데 괜찮나요?"
`data.js`의 `SB_KEY`는 **publishable(공개용) 키**로, 브라우저에 노출되도록 설계된 키입니다. 숨겨야 하는 건 `service_role` 키인데 이건 **사용하지 않았습니다.** 실제 데이터 방어는 키가 아니라 **RLS 정책**이 합니다 — `.env`로 가릴 필요가 없는 게 정상(최신 Supabase 방식).
