COURSE Ⓐ · ANTHROPIC · AI FLUENCY FOR STUDENTS

앤트로픽 · AI Fluency for Students (4D 프레임워크)

AI를 효과적(effective)·효율적(efficient)·윤리적(ethical)·안전(safe)하게 협업하는 능력. 모델·도구·프롬프트를 넘어서 오래 남는 것 = AI 유창성(Fluency). 총 5개 레슨.

※ 이 섹션은 바탕화면에 보관된 실제 강의 트랜스크립트 5종을 직접 대조해 정리했습니다. 저작: Rick Dakan, Joseph Feller & Anthropic (2025) · CC BY-NC-SA 4.0. 각 D는 3개 하위 요소로 이루어집니다.

L1·L2 · 4D 한눈에 — 지도부터 편다

"AI를 쓰는 것"과 "AI에 유창한 것"은 다르다. 대부분의 AI 가이드는 새 모델·제품이 나오면 낡지만, AI 유창성은 모델·도구·프롬프트를 넘어 지속된다.

핵심 질문들: 무엇을 AI에 자동화하고 무엇은 안 할까? 어떤 도구를 고를까? 정확성은 어떻게 검증할까? AI를 썼다고 교사·고용주에게 알릴까, 알린다면 어떻게?

단순 자동화(automation) vs 증강(augmentation)
깊이 생각 없이 "AI가 날 대신해 일하게" 하는 게 자동화. AI 유창성은 내가 이미 하는 좋은 일에 AI를 더하는(augment) 것. 특히 학습에서는 AI가 날 대신하는 게 아니라 내가 더 잘하도록 돕게 한다.

4D — 서로 얽힌 네 능력

① Delegation (위임)

무엇을 할지, 누가 할지 — 사람 몫·AI 몫·함께할 몫을 나눈다.

② Description (설명)

AI에게 어떻게 말하나. 명령이 아니라 협업자와의 대화.

③ Discernment (분별)

좋은/나쁜 결과를 알아보는 판단력.

④ Diligence (성실·책임)

책임 있게·투명하게·윤리적으로.

이 강의는 개요: 각 능력의 더 깊은 통찰·실습은 앤트로픽의 「AI Fluency: Framework & Foundations」 코스에 있음. 핵심은 "책임 있는 human in the loop"가 되는 것.

① Delegation — 위임

"무엇을 하려는가, 그리고 누가 해야 하나." 사람이 할 일 / AI가 할 일 / 함께할 일을 나누는 것 — 사려 깊은 협업의 토대.

Problem Awareness
(문제 인식)

AI를 켜기 전에 "내가 진짜 뭘 하려는가"를 명확히. 시험공부? → 뭘 해야 하나(단어 복습? 사례 준비?), 성공은 어떤 모습인가. 이 이해가 작업 내내 나의 북극성(north star).

Platform Awareness
(플랫폼 인식)

AI는 다 같지 않다. 어떤 건 복잡한 추론에, 어떤 건 학습 지원용으로 특화. 문제 인식이 "할 일"이면 플랫폼 인식은 "최적의 도구·파트너 찾기". (걷기·자전거·버스·운전 중 고르듯)

Task Delegation
(작업 분담)

실제로 나와 AI에게 일을 나눈다. 사람=창의·판단·현실 맥락, AI=속도·일관성·방대한 정보 처리. 양쪽 강점을 살릴 때 최고 결과.

② Description — 설명

AI에게 실제로 말하는 법. 단순 프롬프팅이 아니라 완벽한 프롬프트 암기가 아닌, 협업자와의 대화에 가깝다.

Product Description
(결과물)

최종 결과로 뭘 원하나 — 상세할수록 좋다. "이메일 써줘"가 아니라 길이·핵심 메시지·수신자·문체·형식까지.

Process Description
(과정)

AI가 어떻게 접근하길 원하나. 남에게 가르치듯. "단계적으로 생각해", "여러 관점 고려해", "가장 중요한 것부터".

Performance Description
(역할·태도)

상호작용 중 AI가 어떻게 행동하길 — "내 아이디어에 반박하는 비판적 편집자", "지지적 브레인스토밍" 등 역할 지정.

③ Discernment — 분별

Description이 "원하는 걸 설명"이라면, Discernment는 결과가 좋은지 나쁜지 알아보는 판단력. 오류를 막고, 새로운 창의적 길도 발견한다.

Product Discernment

결과물의 품질 — 정보가 정확한가? 제안이 실제 도움되나? + 내가 생각 못한 걸 보여줬나? 오류 점검 + 새 관점 열기 둘 다.

Process Discernment

AI가 어떻게 도달했나 — 관련 요소를 다 고려했나? 합리적 가정? 논리적 단계? 추론을 이해하면 좋은 출력은 신뢰하고 문제는 잡는다.

Performance Discernment

상호작용 중 AI가 어떻게 행동했나 — 필요한 방식으로 도왔나? 요청한 역할을 지켰나? 적절히 반박했나? 다음 상호작용 조정에 활용.

분별 ≠ 오류 사냥만. AI가 놀랍게 하거나 더 잘 생각하게 도울 때를 알아보는 것도 분별. 때론 내가 못한 접근·놓친 연결·밝은 관점을 준다.
Description ↔ Discernment 피드백 루프
설명 → 결과 판별 → 조정·재설명. 좋은 대화처럼 함께 이해를 쌓아간다. 이 반복 과정이 진짜 AI 유창성이 자라는 곳. "에세이 아이디어 좀 도와줘"가 구조·논증·근거에 대한 정교한 사고로 진화 → AI가 지시 따르는 도구에서 함께 생각하는 파트너로 바뀐다.

④ Diligence — 성실·책임

AI 사용에 책임지는 것. 모든 단계에서 중요. 지루한 규정준수가 아니라 "내가 떳떳하게 뒤에 설 수 있는" 방식으로 쓰는 것.

Creation Diligence
(생성 책임)

어떤 AI를 쓰고 어떻게 상호작용할지 신중히 — 프라이버시·보안·맥락 적절성 고려.

Transparency Diligence
(투명성 책임)

AI를 어떻게 썼는지 정직하게. 교수·팀원·미래 고용주는 알 자격이 있다. 제대로 하면 숨길 게 없고 오히려 협업 능력을 보여주는 것.

Deployment Diligence
(배포 책임)

결과물의 주인은 나. 정확성 검증·적절성 확인·책임. "AI가 이메일 초안을 써도, 보내는 건 나. AI가 아이디어를 내도, 고르는 건 나."

내 상황 연결: Deployment Diligence = 내 CLAUDE.md의 "완료=산출물로 판정"·비가역 승인 게이트. Transparency = 블로그에 AI 사용 밝히기. Discernment = 블로그 "검증된 사실" 노트.

L3 · 응용 1 — AI를 학습 파트너로

핵심 구분: AI가 날 대신해 일하게 vs AI로 내가 더 잘 배우게. AI가 대신 생각해주면 실전에서 쓸 비판적 사고 근육을 못 기른다(시험·면접·발표에서 드러남).

4D를 학습에 적용

  • Delegation: 목표가 "논증적 글쓰기 향상"이면 AI에게 논증을 대신 쓰게 하지 말 것. 대신 토론 상대(debate partner)로 써서 반론을 던지게 하고 내 입장을 세운다. 운전대는 내가.
  • Description: 대부분 AI는 곧장 답부터 주려 한다(학습엔 도움 안 됨). "튜터/교수처럼 행동해", "풀지 말고 질문만 해", "표로 계속 갱신해줘"처럼 역할·형식을 지정해 학습에 맞게 조종.
  • Discernment: 결과 품질뿐 아니라 "내가 실제로 배우고 있나, 그냥 묻어가나"를 정직하게. 기초가 없으면 AI 정확성 판단도 어렵다 → 비-AI 출처로 중요한 지식은 검증.
  • Diligence: 학교가 허용하는 AI·정책 준수(Creation), AI 사용 기록·공개(Transparency), 제출물의 모든 부분을 설명·응용할 수 있어야 함(Deployment). "왜 이렇게 했어?"에 "AI가 정했어서 몰라요"면 진짜 배운 게 아니다.
학습용 프롬프트 예: "대학 1학년인데 세포 수준 광합성을 이해 중이야. 설명하지 말고, 지난주 배운 것과 연결되는 질문을 던져줘." / "이 로그 문제 풀어주지 말고 시작점만 짚어주고, 같이 풀 쉬운 연습문제 몇 개 줘."
패턴: AI를 대체 선수가 아니라 코치로. 최애 기법 = AI를 "내 학생"으로 만들어 가르치기.
🔍 자가 점검 (gut check): ① 배운 걸 남에게 설명할 수 있나? ② AI 없이 비슷한 문제를 풀 수 있나? ③ 그냥 되는 게 아니라 왜 되는지 아나?
막혀서 답답한 그 느낌 = 뇌가 새 연결을 만드는 중. 쉬운 길이 아니라 진짜 성공으로 이끄는 길.

L4 · 응용 2 — AI를 커리어 설계에

이력서·자소서를 AI로 찍어내면 채용담당자가 바로 알아채는 제네릭 콘텐츠가 된다. AI를 결정·콘텐츠를 대신 만드는 서비스가 아니라 사고 파트너로.

커리어 탐색 = 두 부분: ① 정보 수집(AI가 강함 — 뭐가 있고, 어떤 스킬을 원하고, 산업이 어떻게 변하는지) + ② 개인적 성찰(오직 나만 앎 — 뭐가 나를 살아있게 하고, 어떤 트레이드오프를 받아들이고, 내 가치와 맞는지).

AI를 커리어에 쓰는 법

용도예시 질문
산업 리서치·트렌드"환경과학+데이터분석을 결합한 지속가능 테크의 신흥 직무는?"
스킬 갭 파악"UX 디자이너가 되려는데, 부트캠프가 가르치는 것 vs 고용주가 실제 보는 스킬 차이는?"
연봉·시장 인사이트"도시별 신입 컨설턴트의 실제 연봉 상승 곡선은?"
경로·타임라인"교직에서 기업 교육 직무로 전환한 다양한 경로를 보여줘"
역할 브레인스토밍"저널리즘과 비슷한 스킬을 쓰면서 고용 안정성이 더 나은 직업은?"
자기 가치 끌어내기AI가 나에게 질문하게 해서 내 커리어 목표·가치를 명료화 (스스로도 안 해본 것)

스킬 개발 · 취업

  • 스킬 개발 = 학습 파트너와 같은 원리. 구체적으로. "데이터 분석 배우게 도와줘" ❌ → "마케팅 매니저인데 설문 데이터 분석이 필요해. 엑셀 피벗테이블 기초부터 계획 세우고 연습까지 같이 하자" ✅. AI를 코치로.
  • 취업: AI는 브레인스토밍·코칭·연습·초안까지. 하지만 최종적으로 "나"를 표현하는 책임은 나. 전이 가능한 스킬 찾기, 면접 연습(특정 팀 면접관 역할극), 자소서 각도 찾기에 활용.
철칙: 내 전문성·커리어 목표가 AI 협업을 이끌어야 한다 — AI는 뭐가 나를 충족시키는지, 어떤 트레이드오프를 감수할지 모른다. AI 유창성 자체가 이제 값진 커리어 스킬이니 이력서에 넣어라.

L5 · 마무리 — Human in the Loop & 나만의 약속

"Human in the loop" = 모든 AI 상호작용에서 결정하고 배를 조종하는 건 사람. 뭘 물을지, 말이 되는지 평가할지, 쓸 만한지 정하는 게 나. 수동적으로 받아들이지 않고 내 판단·창의·옳고 그름을 가져온다.

✍️ 핵심 실습 — 나만의 AI 협업 원칙(personal commitment) 쓰기
내 가치를 깊이 생각해 직접 써본다. 길든 짧든:
  • 어떤 human in the loop이 되고 싶은가 (언제 AI를 쓰고, 언제 안 쓸지)
  • 학습에 AI를 어떻게 쓸지
  • AI 사용을 언제·어떻게 공개할지
  • AI가 있어도 내 스킬을 어떻게 날카롭게 유지할지
보기 좋은 말이 아니라 진짜로. 이 약속이 AI 사용의 북극성이 된다.

계속 성찰하기

최고의 AI 사용자가 남들이 건너뛰는 것: 자기 실천을 끊임없이 성찰. "이 방식이 날 성장시키나? 미래에 필요한 스킬을 기르나, 그냥 과제를 끝내나? AI 역할을 정직하게 밝히나? 내 가치와 맞나?"

결론: 새 AI는 계속 나온다. 하지만 사려 깊게 위임하고, 명확히 설명하고, 비판적으로 분별하고, 성실히 책임지면 무엇이 와도 준비된 것. 내 인간다움(내 재능·경험·관점·관계)은 대체 불가. AI는 그걸 더 잘 표현하게 도울 뿐. AI 유창성은 목적지가 아니라 여정.

4D 한눈에 — 오늘은 지도만 펼칩니다

낯선 도시에선 골목부터 걷지 말고 지도를 먼저 편다. 4D가 그 지도.

AI Fluency(AI 유창성)란 프롬프트 몇 개를 아는 게 아니라, "지금 나는 어느 구역에 있고 다음에 어디로 가야 하는지"를 스스로 판단하는 상태입니다. 그 지도가 네 가지 능력, 4D예요.

① Delegation (위임)

뭘 AI에 맡기고 뭘 내가 쥘지 정한다. 모든 협업의 출발점.

② Description (설명)

AI가 알아듣게 정확히 전달한다. 프롬프트는 이것의 일부.

③ Discernment (분별)

나온 결과를 곧이곧대로 믿지 않고 비판적으로 판별한다.

④ Diligence (성실·책임)

책임지고 투명하게. 최종 책임은 나. (능력보다 태도)

오늘 꼭 챙길 두 가지
1) ②Description과 ③Discernment는 한 몸 — 설명한다 → 결과 나온다 → 판별한다 → "이걸 안 알려줬네" → 다시 설명한다. 이 설명↔판별 루프를 몇 바퀴 잘 도느냐가 실력을 가른다.
2) 4D의 주어는 전부 "나" — AI가 위임·분별하는 게 아니라 내가 한다. 그래서 마지막이 Human in the Loop로 끝난다.

① Delegation — 위임 1강

"뭘 AI에 맡기고, 뭘 내가 할까"를 의식적으로 나누는 능력. 다 던지는 것도, 다 끌어안는 것도 실패.

Delegation은 세 가지 인식으로 이루어집니다.

Problem Awareness
(문제 인식)

내가 진짜 풀려는 게 뭔가? 목표·성공 기준·제약을 스스로 또렷이 하는 것. 문제를 모르면 위임도 못 한다.

Platform Awareness
(플랫폼 인식)

이 일에 AI가 잘 맞나, 어떤 AI·도구가 강점인가? AI의 한계와 능력을 알아야 맡길지 말지 정한다.

Task Delegation
(작업 분담)

일을 사람 몫·AI 몫·같이 할 몫으로 실제로 쪼갠다. 어디까지 맡기고 어디서 내가 개입할지 결정.

내 상황 연결: 하네스에서 "이건 매니저(나)에게, 이건 웹팀에게" 나누는 그 근육이 바로 Delegation.

② Description — 설명 2강

AI는 눈치가 없다. 내 머릿속 맥락을 모른다. 원하는 걸 말로 정확히 그려주는 능력. "프롬프트"는 이 D의 일부일 뿐.

Description도 세 가지를 말해줘야 합니다.

Product Description
(결과물 설명)

무엇을 원하나 — 형식·길이·톤·대상 독자·포함/제외할 것. "무엇을 만들지"의 그림.

Process Description
(과정 설명)

어떻게 해주길 원하나 — 단계로 나눠서, 먼저 계획을 보여줘, 질문부터 해 등 작업 방식 지정.

Performance Description
(역할·성능 설명)

어떤 역할·태도로 임할지 — "튜터처럼", "비판적으로", "전문가 관점으로" 등 AI의 페르소나 설정.

핵심: 좋은 설명 = 결과물(What) + 과정(How) + 역할(Who)을 함께 준다. 한 번에 완벽할 필요는 없다 — 다음 D(분별)와 왕복하며 다듬는다.

③ Discernment — 분별 3강

AI는 그럴듯하게 틀린다. 문장은 매끄러운데 사실이 틀리고, 코드는 돌아가는데 헛다리를 짚는다. "이거 진짜 맞아?"를 판별하는 능력.

Product Discernment
(결과물 분별)

나온 답·코드·글이 정확한가, 내 목표에 맞나, 빠진 건 없나를 비판적으로 검토.

Process Discernment
(과정 분별)

AI가 밟은 추론·단계가 타당한가? 근거 없이 건너뛴 곳은 없나 살핀다.

Performance Discernment
(성능 분별)

이 AI·이 방식이 이 일에 적절했나? 더 나은 도구·접근은 없었나 되돌아본다.

Description ↔ Discernment 피드백 루프
설명(②) → 생성 → 분별(③) → "아 이걸 안 알려줬네" → 다시 설명(②). 이 루프를 잘 도는 사람이 AI를 진짜 잘 쓰는 사람이다. (= 설명→판단→다듬기 반복 = 진짜 유창성)
내 상황 연결: 블로그의 "검증된 사실"·정확성 노트가 정확히 이 Discernment 근육.

④ Diligence — 성실·책임 4강

능력이라기보다 태도. 그래서 4D의 마지막. "AI가 이메일 초안을 써도, 보내는 건 나."

생성 책임
(Creation Diligence)

어떤 AI를 쓸지 신중히 — 프라이버시·보안·적절성을 따진다.

투명성 책임
(Transparency Diligence)

AI를 썼다는 걸 정직하게 밝힌다.

배포 책임
(Deployment Diligence)

최종 결과는 내 책임 — 내보내기 전에 내가 확인하고 책임진다.

응용 2가지 — 학습 · 커리어

📚 학습 파트너로

AI한테 답을 받지 말고 튜터·코치로 쓴다. "설명하지 말고 질문해줘", "풀지 말고 방향만", "네가 내 학생 해봐(가르치면 내가 배움)".

자가점검: 남한테 설명 가능? AI 없이 비슷한 문제 풀 수 있어? 왜 되는지 알아?

🧭 커리어 설계에

AI = 정보·리서치·코칭(산업 트렌드·스킬갭·연봉·모의면접), 나 = 가치·판단·"나를 표현".

제네릭 AI 이력서는 티 나서 실패.

마무리 — Human in the Loop 5강

매 순간 결정하고 방향 잡는 건 사람. 4D의 결론.

핵심 실습: "나만의 AI 협업 원칙(personal commitment)"을 직접 써본다 — 언제 AI 쓰고 안 쓸지, 학습에 어떻게, 언제 공개할지, 스킬을 어떻게 유지할지.

왜 나에게 중요하냐 (매니저 코멘트)

Claude 다수 세션·Codex로 Le ju3n 굴리고, 블로그도 AI로 만드는 주영님 그 자체 얘기. 4D가 그걸 잘하는 법이다.

  • Delegation = 하네스에서 "뭘 AI에 맡길지"(매니저·팀 분담)
  • Discernment = 블로그 "검증된 사실"(정확성 노트)·웹팀 피드백 판별
  • Diligence = "완료=산출물로 판정"·비가역 승인 게이트 (이미 내 CLAUDE.md에 있음)

인프런 · AI 에이전트 개발 오리엔테이션

강사 승귤, 총 9강. What·Why·How로 개념을 잡고 Google ADK로 실제 에이전트를 만들어본다. 에이전트는 최종 목표가 아니라 AGI로 가는 과정의 한 단계.

1. 강의소개 01:48

  • ChatGPT(2022) 이후 AI가 일상 소프트웨어가 됐고, 2025년 "AI 에이전트"가 화두가 되며 다들 개발에 뛰어드는 상황.
  • 대상: "에이전트라는 말은 들었는데 뭔지·왜 이렇게 뜨는지·어떻게 만드는지" 궁금한 사람.
  • 도착점: 구글 ADK로 간단한 에이전트를 만들어 테스트하는 입문 튜토리얼.
  • 목차 = What / Why / How 세 키워드.

2. AI 에이전트란 04:03 What

학계에 확정된 정의는 없다 → 선도 3사의 정의로 접근.

회사정의 요지
OpenAI"사용자를 대신해 독립적으로 작업을 수행하는 시스템." (샘 올트먼: 에이전트 = 가상의 동료)
AnthropicLLM이 도구 사용·절차를 동적으로 바꾸는 것. + "모든 경우에 에이전트가 필요한 건 아니다. 워크플로우만으로 충분한 경우도 많다."
Google"목표를 추구·태스크를 완료하는 소프트웨어. 추론·계획·기억 가능, 자율성으로 의사결정·학습·조정." (가장 기술적)
강사의 정리: 정의만 보면 헷갈리지만, 내가 할 일을 AI에게 대신 시키고 결과만 확인하고 싶은 게 핵심. ChatGPT는 답만 주고 실행은 내가 함 → 그 다음 단계가 에이전트(자율·자동 실행).

3. 왜 AI에이전트일까 05:30 Why

OpenAI 에이전트 모드, MS Copilot Studio, Google Agent Spaces, Anthropic MCP 등 제품이 쏟아지는 중. 이유는 OpenAI의 AGI 5단계로 설명된다.

단계이름설명
1대화형 AIChatGPT
2추론형 AIthinking 모델
3자율형 AI← 에이전트. 며칠 걸리는 일도 수행
4혁신형 AI스스로 발명·개선
5조직형 AI사람 개입 없이 회사 전체 운영
결론: 에이전트는 최종 목표가 아니라 AGI로 가는 과정의 3단계. 현재 쓸만한 예 — 코딩 에이전트(Cursor, Claude Code), 고객서비스(Salesforce Agentforce), ChatGPT Agent 등.

4. AI 에이전트를 만들려면? 04:10 How

필요한 재료 4가지

① 언어모델 (두뇌)

사람의 말/글을 이해하는 LLM. 에이전트의 뇌.

② 추론·계획

프롬프트 설계 또는 모델의 thinking으로 계획 수립.

③ 도구 (tools)

브라우저·터미널 등. 외부 API 연결(MCP로 만들기도).

④ 메모리

멀티턴 대화용. 이전 정보를 저장·활용.

동작 루프

요청 → LLM 추론 → 도구 선택·호출 → 결과를 메모리에 저장 → 반복 → 완료 판단 시 결과물 생성

워크플로우 vs 에이전트

워크플로우에이전트
방식규칙대로 실행 (정적)LLM이 동적으로 의사결정
적합패턴화된 업무복잡·열린 문제
특징성능 안정·통제 쉬움유연하지만 변동성
"이걸 다 직접 만들어야 하나?" → 걱정 X. ADK가 다 만들어 놨으니 ADK를 쓸 것.

5. ADK 개발 환경 준비하기 04:13

ADK = Agent Development Kit (Google, Python·Java).

ADK가 제공하는 것

  • 에이전트 타입: LLM 에이전트(기본), 워크플로우 에이전트(Sequential·Loop·Parallel)
  • 빌트인 도구: Google Search, 코드 실행, Vertex AI·BigQuery(GCP 필요)
  • 세션(= 채팅방, 저장값=State) / 메모리(여러 세션 검색, 단기·장기)

개발환경 3단계

API 키 발급 — Google AI Studio(aistudio.google.com) → Get API Key. 기본 무료.
Google ADK 설치 — pip install google-adk
환경변수 설정 — .env에 2개(API 키 + Vertex 사용여부 등).

6. 헬로 ADK 03:02

가장 단순한 인사 에이전트(약 10줄)로 골격 익히기.

디렉토리 규칙: hello-agent/ 폴더 안에 .env + 파일명은 반드시 agent.py
from google.adk.agents import Agent

def greet_user():
    return "안녕하세요!"          # 사용자에게 인사하는 도구(tool)

root_agent = Agent(
    name="hello_agent",
    model="gemini-2.5-flash-preview-05-20",
    description="유저와 인사하는 에이전트입니다.",
    instruction="사용자와 반갑게 인사해 주세요.",   # 시스템 프롬프트 역할
    tools=[greet_user],                              # 도구는 리스트로 여러 개 가능
)

※ LLM 에이전트 = "그냥 에이전트"와 같은 말. 위 코드는 강의 흐름에 맞춘 재구성입니다.
